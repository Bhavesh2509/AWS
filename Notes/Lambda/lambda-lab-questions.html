PART 1 — TROUBLESHOOTING LAB SCENARIOS

Each lab gives you a broken or misconfigured setup. Your job is to identify the root cause and fix it. Think through IAM, timeouts, environment, VPC, triggers, and runtime behaviour.

🔐 GROUP A — Permissions & IAM

LAB 01 HARD
Your Lambda function is supposed to read an object from an S3 bucket and write a record to a DynamoDB table. The function executes without error, but no record ever appears in DynamoDB. CloudWatch logs show: AccessDeniedException: User: arn:aws:sts::123456789:assumed-role/lambda-role/fn is not authorized to perform: dynamodb:PutItem
🔧 What is wrong and how do you fix it?
Hint: Check the execution role attached to the function. The role has AmazonS3ReadOnlyAccess but is missing DynamoDB permissions.
LAB 02 MED
You want Lambda to send a message to an SQS queue. You attached an inline policy to the execution role allowing sqs:SendMessage on arn:aws:sqs:us-east-1:123456789:my-queue. The function throws: Access to the resource arn:aws:sqs:ap-south-1:123456789:my-queue is denied.
🔧 What is the root cause?
Hint: The function is deployed in ap-south-1 but the IAM policy ARN hardcodes us-east-1. The region in the ARN doesn't match the actual queue's region.
LAB 03 MED
Lambda needs to read a secret from AWS Secrets Manager. The execution role has secretsmanager:GetSecretValue permission. The function still gets ResourceNotFoundException.
🔧 What could be wrong? (Two possible causes.)
Hint: (1) The secret name/ARN in code is wrong or has a typo. (2) The secret is in a different region than the function. Secrets Manager is regional — you must call the endpoint in the correct region.
LAB 04 HARD
A Lambda function uses a resource-based policy to allow API Gateway to invoke it. After redeploying the API to a new stage called prod, the API returns 403 errors. The old dev stage still works.
🔧 Why does prod fail?
Hint: The resource-based policy on the Lambda function explicitly allows invocation only from arn:aws:execute-api:...:*/dev/GET/resource. The new prod stage ARN is different. You must add a new permission statement for the prod stage.
LAB 05 EASY
You're invoking a Lambda function cross-account. The function's resource-based policy grants lambda:InvokeFunction to the calling account. But you still get AccessDeniedException.
🔧 What is the missing piece?
Hint: Cross-account invocation requires BOTH: (1) the Lambda resource-based policy allows the calling account, AND (2) the IAM role/user in the calling account has lambda:InvokeFunction in its identity-based policy. Both sides must grant access.

⏱️ GROUP B — Timeout, Memory & Performance

LAB 06 MED
A Lambda function processes images uploaded to S3. It works fine for small images but fails for large files (>50MB) with the error: Task timed out after 3.00 seconds.
🔧 How do you fix this? What two settings should you adjust?
Hint: (1) Increase the function timeout (max: 15 minutes). (2) Increase memory allocation — Lambda CPU scales proportionally with memory, so more memory = more CPU = faster processing. Also consider using /tmp storage (up to 10 GB) for staging the file.
LAB 07 HARD
A Lambda function calls an external HTTP API. In testing it works fine. In production it hangs and times out every time. No errors in the third-party API. CloudWatch shows the function runs for exactly its configured timeout (30s) and then stops.
🔧 What is the likely cause?
Hint: The function is inside a VPC. VPC-attached Lambda functions have no internet access by default. To reach external APIs, you need a NAT Gateway in a public subnet, or use a VPC endpoint if calling an AWS service. Without this, the HTTP call just hangs until timeout.
LAB 08 MED
Your Lambda function processes SQS messages. It works correctly but you notice some messages are being processed 2–3 times. There are no errors in CloudWatch.
🔧 Why is this happening and how do you fix it?
Hint: SQS standard queues have at-least-once delivery. If the function's execution time is close to or exceeds the SQS queue's visibility timeout, SQS re-enqueues the message thinking it wasn't processed. Fix: Set the queue's visibility timeout to at least 6× the function's timeout.
LAB 09 EASY
A Lambda function needs to store a 600 MB temporary file during processing. The function throws: No space left on device.
🔧 What is the fix?
Hint: /tmp default size is 512 MB. You can increase it up to 10,240 MB (10 GB) in the function configuration. Alternatively, stream the file from/to S3 instead of downloading it entirely.

🌐 GROUP C — VPC & Networking

LAB 10 HARD
Lambda is deployed inside a VPC and needs to call the AWS SSM Parameter Store API. All outbound traffic is blocked at the security group level except to specific VPC endpoints. The call fails with a timeout.
🔧 What is needed?
Hint: You must create an Interface VPC Endpoint for SSM (com.amazonaws.region.ssm) in your VPC, and ensure the Lambda's security group allows outbound HTTPS (port 443) to the endpoint's security group. Without this, there's no route from a VPC Lambda to SSM.
LAB 11 MED
Lambda inside a VPC cannot connect to an RDS MySQL database in the same VPC. The Lambda code gets Connection timed out.
🔧 What are the two most common causes?
Hint: (1) RDS security group doesn't allow inbound traffic on port 3306 from the Lambda's security group. (2) Lambda and RDS are in different subnets that have no route to each other, OR Lambda isn't in the same VPC as RDS. Fix: Add an inbound rule to RDS SG allowing port 3306 from Lambda SG.
LAB 12 MED
You moved a Lambda function from "no VPC" to inside a VPC to access a private database. Now it can reach the DB, but calls to S3 are timing out.
🔧 Why did S3 stop working and how do you fix it?
Hint: Previously, Lambda reached S3 over the public internet. Inside a VPC, it has no internet access. Fix: Create a Gateway VPC Endpoint for S3 (free) and add it to the route table of Lambda's subnet. This routes S3 traffic inside AWS's network without needing a NAT Gateway.

⚡ GROUP D — Triggers & Event Sources

LAB 13 HARD
Lambda is configured to trigger from an SQS queue. Messages are piling up in the queue but Lambda is never invoked. The function works fine when invoked manually.
🔧 What are three things to check?
Hint: (1) The Event Source Mapping (ESM) is disabled — check if the trigger is enabled in the Lambda console. (2) The execution role is missing sqs:ReceiveMessage, sqs:DeleteMessage, sqs:GetQueueAttributes. (3) The Lambda function has too many errors — SQS ESM backs off on repeated failures.
LAB 14 MED
A Lambda function is triggered by S3 ObjectCreated events. You upload a file but the function is invoked twice.
🔧 What is the cause?
Hint: Two S3 event notifications have been configured for the same bucket + event type — possibly one at the bucket level and another via EventBridge. Check the S3 bucket's event notification configuration and EventBridge rules. Remove the duplicate trigger.
LAB 15 HARD
Lambda triggered by DynamoDB Streams is processing old events from hours ago even though the stream was just enabled. The function processes thousands of records it shouldn't touch.
🔧 What setting controls this and how do you fix it?
Hint: When creating the Event Source Mapping for DynamoDB Streams, the Starting Position was set to TRIM_HORIZON (reads from the oldest available record). It should have been set to LATEST to only process new records going forward.
LAB 16 MED
You have a Lambda function triggered by EventBridge (CloudWatch Events) on a cron schedule every 5 minutes. It stopped firing after you deployed a new version and updated the alias. Logs show no invocations at all.
🔧 What is likely wrong?
Hint: The EventBridge rule's target ARN points to a specific function version (e.g., function:myFn:3) that no longer exists or the resource-based policy on the new version/alias doesn't allow EventBridge to invoke it. Update the rule to target the alias ARN and add the correct permission.

🧊 GROUP E — Cold Starts, Runtime & Environment

LAB 17 MED
A Java Lambda function has inconsistent response times — sometimes 200ms, sometimes 4–5 seconds. Users randomly experience slow responses.
🔧 What is happening and what are two ways to fix it?
Hint: Cold starts. Java has a heavy JVM initialization. Fix: (1) Enable SnapStart — snapshots the initialized environment, restores it on cold starts (sub-second). (2) Use Provisioned Concurrency to keep N instances always warm. Both eliminate cold start latency.
LAB 18 EASY
A Lambda function reads a config file from /tmp/config.json that it wrote in a previous invocation. Sometimes it works, sometimes it throws FileNotFoundError.
🔧 Why is this unreliable?
Hint: /tmp is shared within an execution environment but NOT guaranteed to persist between invocations. Lambda can spin up new execution environments at any time (cold starts, scaling). You cannot rely on /tmp across invocations. Store persistent config in S3, SSM Parameter Store, or environment variables.
LAB 19 MED
Your Lambda function uses a Python library not available in the default runtime. You packaged it in a ZIP file but get: Unable to import module 'lambda_function': No module named 'pandas'.
🔧 What are the two correct ways to fix this?
Hint: (1) Include the pandas library in the ZIP deployment package alongside your code (install with pip install pandas -t ./). (2) Create a Lambda Layer containing pandas and attach it to the function. Layers are shared across functions and reduce package size.
LAB 20 HARD
Lambda function runs fine locally. In AWS it fails with ImportError: libgomp.so.1: cannot open shared object file.
🔧 What is the cause?
Hint: The Python library (e.g., scikit-learn, LightGBM) was compiled on your local OS (macOS/Windows). Lambda runs on Amazon Linux 2. Native shared libraries (.so files) are OS-specific. Fix: Build your deployment package on Amazon Linux 2 (use a Docker container: public.ecr.aws/lambda/python:3.12), or use a container image deployment.

🔁 GROUP F — Error Handling, Retries & Destinations

LAB 21 MED
Lambda is asynchronously invoked. It sometimes fails. You want failed events to go to an SQS Dead Letter Queue (DLQ) for inspection, but they never appear there.
🔧 What is wrong?
Hint: Lambda retries asynchronous invocations twice (3 attempts total) before sending to DLQ. Also, the execution role must have sqs:SendMessage permission on the DLQ ARN. If the role lacks this permission, Lambda silently drops the event after retries. Check both the retry count and the role permission.
LAB 22 HARD
Lambda processes Kinesis stream records. One bad record keeps causing an error, blocking all subsequent records in the shard. Processing is completely stalled.
🔧 How do you unblock processing without losing data?
Hint: Enable bisect-on-error (splits the batch in half to isolate the bad record) and set a maximum retry attempts limit on the ESM. Also configure a failure destination (SQS or SNS) so failed records are captured. Set ReportBatchItemFailures to allow partial batch success — good records still proceed.
LAB 23 MED
You configured a Lambda Destination for async invocations. On success, it routes to EventBridge. On failure, it routes to SQS. Failures go to SQS correctly, but successes never appear in EventBridge.
🔧 What is the likely cause?
Hint: The execution role lacks events:PutEvents permission on the EventBridge bus. Lambda Destinations require the execution role to have permission to deliver to the destination resource — this is separate from the destination resource's own policy.

🔢 GROUP G — Concurrency & Throttling

LAB 24 HARD
You have 5 Lambda functions in your account. One high-traffic function is consuming almost all of the account's concurrency. Other functions are being throttled and returning 429 errors.
🔧 How do you prevent one function from starving others?
Hint: Set a Reserved Concurrency limit on the high-traffic function (e.g., cap it at 500 concurrent executions). This guarantees remaining concurrency for other functions. Note: Reserved concurrency also serves as a throttle ceiling for that function.
LAB 25 MED
A critical API-backed Lambda function must never have cold starts. You set Provisioned Concurrency to 10. During a traffic spike, 15 concurrent requests come in. The extra 5 requests are slow.
🔧 Why are 5 requests slow and how can you address this?
Hint: Provisioned Concurrency pre-warms exactly N instances. Requests beyond that N go to on-demand (cold) instances — they experience cold starts. Fix: (1) Increase Provisioned Concurrency. (2) Use Application Auto Scaling to automatically adjust Provisioned Concurrency based on a schedule or utilization metric.

PART 2 — BASICS FROM DOCUMENTATION (Q&A)

⚙️ Core Limits & Configuration

Q1 Limits
What is the maximum execution timeout for a Lambda function?
900 seconds (15 minutes). Default is 3 seconds. You must explicitly increase it. For workloads beyond 15 min, use Step Functions or ECS.
Q2 Limits
What is the maximum and default memory allocation for a Lambda function?
Min: 128 MB. Max: 10,240 MB (10 GB). Default: 128 MB. CPU and network bandwidth scale proportionally with memory. There is no separate CPU setting — more memory = more CPU.
Q3 Limits
What is the maximum size of a deployment package (ZIP) and a container image?
ZIP (unzipped): 250 MB. ZIP (compressed, uploaded directly): 50 MB. Via S3: 250 MB unzipped. Container image: 10 GB.
Q4 Limits
What is the maximum /tmp storage size?
512 MB default, configurable up to 10,240 MB (10 GB). Shared within an execution environment. Not guaranteed to persist between invocations.
Q5 Limits
What is the default account-level concurrency limit for Lambda?
1,000 concurrent executions per region (default). This is a soft limit — can be increased via AWS Support. 100 of these are reserved for functions without explicit reserved concurrency.
Q6 Limits
What is the maximum number of Lambda Layers that can be attached to a function?
5 layers maximum per function. The total unzipped size of the function and all layers combined cannot exceed 250 MB.
Q7 Limits
What is the maximum payload size for synchronous and asynchronous invocations?
Request payload: 6 MB (synchronous), 256 KB (asynchronous). Response payload: 6 MB (synchronous). Streaming responses can be larger.

🔄 Invocation Types & Behaviour

Q8 Invocation
What are the three invocation types in Lambda and when does each retry on failure?
1. Synchronous (RequestResponse): No automatic retries by Lambda. Caller gets the error and must handle retries.
2. Asynchronous (Event): Lambda retries 2 times (3 total attempts). Failed events can go to DLQ or Destination.
3. Poll-based (Stream/Queue — ESM): Retries until the record expires in the stream or max retry attempts is hit.
Q9 Invocation
What is the difference between Reserved Concurrency and Provisioned Concurrency?
Reserved Concurrency: Caps the maximum concurrent executions for a function. Guarantees that concurrency is available (not stolen by other functions). Acts as a throttle. Does NOT eliminate cold starts.

Provisioned Concurrency: Pre-initializes N execution environments. Eliminates cold starts for those N instances. You pay for it even when idle. Does NOT cap max concurrency.
Q10 Invocation
What happens when you set Reserved Concurrency to 0?
The function is effectively throttled completely — it cannot be invoked at all (returns 429 TooManyRequestsException). Useful to temporarily disable a function without deleting it.
Q11 Invocation
What is a Lambda Function URL and how does it differ from API Gateway?
A Function URL is a dedicated HTTPS endpoint built into Lambda — no API Gateway needed. Supports NONE (public) or AWS_IAM auth. No throttling, usage plans, request validation, or WAF compared to API Gateway. Best for simple webhooks or single-function APIs.

🏗️ Architecture & Execution Model

Q12 Execution
What is a cold start and what are the phases that happen during one?
Cold start = Lambda initializing a new execution environment. Phases:
1. Download code → 2. Start MicroVM (Firecracker) → 3. Start runtime (e.g., JVM, Node) → 4. Run Init code (outside handler) → 5. Run handler
Phases 1–4 are the "cold start overhead." Only phase 5 happens on warm invocations.
Q13 Execution
What is the handler function signature? Give the Python example.
def handler_name(event, context):

event: Dict containing the trigger data (e.g., S3 event, API Gateway request).
context: Object with runtime info — function name, memory limit, remaining time (context.get_remaining_time_in_millis()), request ID, log stream name.
Q14 Execution
Code placed outside the handler function runs only once per execution environment. True or False? What is this useful for?
True. Code outside the handler (init code) runs during the init phase. Use this to: establish DB connections, load ML models, read config files, initialize SDK clients. These are reused across warm invocations — significantly reduces per-invocation latency.
Q15 Execution
What is the execution environment lifecycle? When does Lambda freeze and thaw an environment?
Init → Invoke → Freeze → (Invoke again = thaw) → Shutdown
After handling a request, Lambda freezes the environment (pauses all processes). On the next request, it thaws (resumes). If idle too long (~15 min), it is shut down. Background threads/timers are frozen and do not run between invocations.

🔐 Security

Q16 Security
What are the two types of policies used in Lambda security?
1. Execution Role (Identity-based policy): What the function IS ALLOWED TO DO — e.g., call S3, DynamoDB, SNS. Attached to the function as an IAM role.

2. Resource-based Policy (Function Policy): Who is ALLOWED TO INVOKE the function — e.g., API Gateway, S3, EventBridge, other accounts. Attached to the function itself.
Q17 Security
How do you securely pass secrets (DB passwords, API keys) to a Lambda function?
Best practices (in order):
1. AWS Secrets Manager — encrypted, rotatable, audit logged. Fetch at runtime.
2. SSM Parameter Store (SecureString) — encrypted with KMS, free tier available.
3. Environment variables with KMS encryption — simpler, but no rotation.
Never: hardcode in source code, store plaintext in environment variables for sensitive values.

📦 Versions, Aliases & Deployment

Q18 Deployment
What is a Lambda version and what is an alias? How do they work together?
Version: An immutable snapshot of your function code + configuration. Published with PublishVersion. Referenced as function:1, function:2, etc. $LATEST is the mutable draft.

Alias: A named pointer to a version (e.g., prod → version 5). Can split traffic between two versions for canary/blue-green deployments (e.g., 90% v5, 10% v6). Aliases have their own ARN used in triggers.
Q19 Deployment
What is SnapStart and which runtimes support it?
SnapStart takes a snapshot of the initialized execution environment (after init phase). On cold start, Lambda restores the snapshot instead of initializing from scratch — dramatically reducing cold start time.

Supported runtimes: Java (up to 10× faster), Python, .NET. Not available for all runtimes. Requires publishing a version to activate.
Q20 Deployment
What is a Lambda Layer? What are its limits?
A Layer is a ZIP archive of libraries, dependencies, or custom runtimes that can be shared across multiple functions. Max 5 layers per function. Total size (function + layers unzipped) must be under 250 MB. Layers are versioned and immutable. Can be shared across accounts.

📊 Monitoring & Observability

Q21 Monitoring
What are the key CloudWatch metrics automatically published by Lambda?
Invocations — total times function was invoked
Errors — invocations that resulted in errors
Duration — execution time (min/max/avg/p99)
Throttles — invocations rejected due to concurrency limits
ConcurrentExecutions — number running simultaneously
UnreservedConcurrentExecutions — account-level pool usage
DeadLetterErrors — failures sending to DLQ
Q22 Monitoring
How does Lambda pricing work? What is a GB-second?
Billed on two dimensions:
1. Number of requests — $0.20 per 1M requests (after free tier of 1M/month)
2. Duration (GB-seconds) — memory (GB) × execution time (seconds). E.g., 512 MB × 2 seconds = 1 GB-second.
Free tier: 400,000 GB-seconds/month (permanent). ARM/Graviton is ~20% cheaper per GB-second.

🗂️ Event Source Mappings

Q23 ESM
Which event sources use Event Source Mappings (poll-based) vs. push-based triggers?
Poll-based (ESM — Lambda polls the source): SQS, Kinesis, DynamoDB Streams, MSK, self-managed Kafka, MQ

Push-based (source invokes Lambda directly): S3, SNS, API Gateway, ALB, EventBridge, Cognito, IoT, CloudFront (Lambda@Edge)
Q24 ESM
What is ReportBatchItemFailures and why is it important for SQS?
By default, if any message in an SQS batch fails, Lambda retries the entire batch (even successfully processed messages).

ReportBatchItemFailures lets your function return a list of failed message IDs. Lambda only retries those specific messages — successfully processed ones are deleted. Prevents duplicate processing and speeds up error recovery.
Q25 ESM
What is the difference between SQS Standard Queue and FIFO Queue behavior with Lambda?
Standard: At-least-once delivery. Lambda scales up to 1,000 concurrent executions. Messages can be out of order. Up to 10 messages per batch.

FIFO: Exactly-once processing. Messages processed in order per message group. Lambda scales up to max 300 concurrent executions (one per active message group ID). Up to 10 messages per batch.
