① VPC — Basic Concepts
Q1What is an Amazon VPC and what are its core components? Basic▼
Answer

A VPC is a logically isolated virtual network in AWS where you launch resources. Core components:

  • Subnets — Segments of the VPC CIDR; each resides in one AZ. Public subnets have an IGW route; private subnets do not.
  • Internet Gateway (IGW) — Allows VPC resources to communicate with the public internet.
  • Route Tables — Rules that determine where traffic is sent. Each subnet has one route table.
  • Security Groups — Stateful, instance-level firewalls (allow rules only).
  • Network ACLs — Stateless, subnet-level firewalls (allow and deny rules).
  • NAT Gateway — Lets private subnet instances initiate outbound internet connections.
  • VPC Endpoints — Private connectivity to AWS services without internet.

Q3What are VPC Flow Logs and what do they capture? Basic▼
Answer

VPC Flow Logs capture metadata (NOT packet content) about IP traffic to/from network interfaces. Each record includes: source IP, destination IP, ports, protocol, bytes, packets, timestamps, and action (ACCEPT/REJECT).

Can be enabled at VPC, subnet, or ENI level. Stored in CloudWatch Logs or S3.

Use cases: troubleshooting connectivity, identifying rejected traffic, detecting security anomalies, capacity planning.
Q4What is the difference between a public subnet and a private subnet? Basic▼
Answer

Public Subnet: Route table has 0.0.0.0/0 → igw-xxx. Resources can have public IPs and are reachable from the internet.

Private Subnet: No IGW route. Resources have only private IPs. Outbound internet access goes through a NAT Gateway in a public subnet.

Classic 3-tier: Public (ALB) → Private (App) → Private (DB). Databases should never be in a public subnet.
Q5What is VPC Peering and what are its limitations? Basic▼
Answer

VPC Peering is a direct private connection between two VPCs (same or cross account/region). Traffic stays on the AWS private network.

Limitations:

  • No transitive routing — A↔B and B↔C does not give A↔C. Requires a direct A↔C peering.
  • No overlapping CIDRs — VPCs must have non-overlapping IP ranges.
  • Scales poorly — N VPCs need N(N-1)/2 connections (10 VPCs = 45 connections).
  • No edge-to-edge routing (cannot use the peered VPC's VPN or DX gateway).

Q7What is AWS Transit Gateway and when should you use it over VPC Peering? Basic▼
Answer

Transit Gateway is a hub-and-spoke cloud router. Each VPC or on-premises network connects once to TGW; TGW handles routing between all attached networks.

Use TGW over Peering when: you have many VPCs, need transitive routing, need centralized inspection, need traffic segmentation, or have hybrid (VPN/DX) connectivity.

Rule of thumb: ≤5 VPCs → consider peering. More than 5 or hybrid → use Transit Gateway.

Q10What is a target group in ELB and what targets can it contain? Basic▼
Answer

A Target Group is a logical collection of targets that a load balancer routes traffic to. The LB health-checks each target and only sends to healthy ones.

  • Instance — EC2 instances by instance ID.
  • IP — Private IP addresses, including on-premises via DX/VPN.
  • Lambda — ALB only. Invoke Lambda for serverless backends.
  • ALB — NLB only. Forward to an ALB as a target.
Q11What is connection draining / deregistration delay? Basic▼
Answer

When a target is deregistered, the LB stops sending new requests but lets in-flight requests complete during the deregistration delay (default 300 seconds, range 0–3600).

Low value (30s) for short HTTP connections; higher for WebSockets or file uploads. Set to 0 for immediate removal.

Q13Explain all Route 53 routing policies and when to use each. Basic▼
Answer
  • Simple — Single resource or multiple IPs (client picks randomly). Single-server setups.
  • Weighted — Traffic split by % weight. A/B testing, canary releases.
  • Latency-based — Route to lowest-latency region. Global apps.
  • Failover — Active-passive. Primary serves; secondary activates on health check failure.
  • Geolocation — Route by user's country/continent. Content localization, data residency.
  • Geoproximity — Route by proximity with configurable bias. Requires Traffic Flow.
  • Multi-Value Answer — Up to 8 healthy records. Client-side load balancing.
  • IP-based — Route by source IP CIDR. On-premises to AWS routing.
Q14What is Route 53 Resolver and why is it important for hybrid architectures? Basic▼
Answer

Route 53 Resolver is the DNS resolver built into every VPC at the VPC base CIDR +2 address. For hybrid connectivity:

  • Inbound Endpoints — On-premises DNS forwards queries for AWS-hosted names to Resolver in the VPC.
  • Outbound Endpoints — VPC resources forward queries for on-premises names to on-premises DNS.

Forwarding Rules define which domains go where, enabling seamless DNS resolution across cloud and on-premises.

Q17What is AWS PrivateLink and how does it differ from VPC Peering? Basic▼
Answer

PrivateLink — Exposes a specific service via an Interface Endpoint (private IP). Unidirectional. Works with overlapping CIDRs. Best for SaaS-style service exposure.

⑥ Scenario-Based Questions (Original)
S1Your EC2 instance in a private subnet cannot reach the internet for updates. How do you troubleshoot? Scenario▼
Checklist
  • 1. Route table — Does it have 0.0.0.0/0 → nat-gateway-id?
  • 2. NAT Gateway status — Is it "Available" and in a PUBLIC subnet?
  • 3. NAT Gateway's subnet route table — Must have 0.0.0.0/0 → igw-xxx.
  • 4. NAT Gateway EIP — Must have an associated Elastic IP.
  • 5. SG outbound rules — Default allows all outbound; ensure it hasn't been locked down.
  • 6. NACL — Stateless! Must allow outbound 80/443 AND inbound ephemeral ports (1024–65535) for return traffic.
  • 7. VPC DNS — enableDnsSupport and enableDnsHostnames must be enabled.
Top causes: route table missing NAT gateway route · NAT Gateway in wrong subnet · NACL blocking ephemeral ports.
S2Your company has 50 VPCs across multiple AWS accounts that all need to communicate. What architecture do you design? Scenario▼
Answer

AWS Transit Gateway + AWS Resource Access Manager (RAM)

  • Create a central TGW in a Network/Shared Services account.
  • Share the TGW across all accounts using RAM — no peering mesh needed.
  • Each VPC attaches to the shared TGW (one attachment each).
  • Configure TGW route tables for connectivity (full mesh or segmented prod vs dev).
  • Add centralized inspection VPC with Network Firewall for east-west traffic.
50 VPCs via peering = 1,225 connections. With TGW = 50 attachments.
S3Your application is in us-east-1. Users in Asia and Europe have high latency. How do you fix it without re-architecting? Scenario▼
Answer

Option A — Global Accelerator: Users connect to the nearest AWS PoP via Anycast IPs, then route over AWS backbone to us-east-1. No code changes.

Option B — CloudFront: Cache static assets globally at 600+ PoPs. Dynamic calls still go to origin but benefit from AWS backbone routing.

HTTP/API app → Global Accelerator. Static asset-heavy → CloudFront for better cost efficiency.
S4Two VPCs both use 10.0.0.0/16 (overlapping). They need to share a specific service. How? Scenario▼
Answer

Peering and TGW require non-overlapping CIDRs. Use AWS PrivateLink:

  • Provider VPC: deploy service behind NLB, create a VPC Endpoint Service.
  • Consumer VPC: create an Interface Endpoint — AWS creates an ENI with a private IP from the consumer's own CIDR space. No conflict.
PrivateLink is the only native AWS solution when CIDRs overlap.
S5Your Lambda functions in a VPC cannot access the internet or S3. What is wrong? Scenario▼
Answer

When Lambda is in a VPC it loses default internet access and uses your VPC network path.

  • No internet — Lambda's private subnet has no route to a NAT Gateway. Fix: add NAT GW in public subnet with route 0.0.0.0/0 → nat-gw.
  • No S3 — Add a Gateway VPC Endpoint for S3 (free).
  • Other AWS services — Add Interface VPC Endpoints (SSM, Secrets Manager, etc.).
  • DNS failing — Ensure enableDnsSupport = true and enableDnsHostnames = true.
⑦ Architecture Design & Hard Questions
D1Design a 3-tier web application architecture in AWS with high availability and security. Hard▼
Architecture

VPC (per region, 2+ AZs): Public subnets (ALB, NAT GW) → Private App subnets (EC2/ECS in ASG) → Private DB subnets (RDS Multi-AZ).

Traffic flow: Route 53 → CloudFront (WAF + Shield) → ALB → App → RDS.

Security: SG chain: ALB allows 443 from internet → App allows 8080 from ALB SG → RDS allows 5432 from App SG. NACLs restrict DB subnets to App subnets only. VPC Flow Logs on. Secrets Manager for DB credentials.

D2What happens when you enable "Private DNS" on an Interface VPC Endpoint? Hard▼
Answer

Route 53 Resolver creates a private hosted zone that overrides the public DNS for that AWS service with the endpoint ENI's private IP. E.g., ssm.us-east-1.amazonaws.com now resolves to a private IP — traffic never leaves the VPC.

Requires VPC attributes: enableDnsSupport = true and enableDnsHostnames = true.

With on-premises DNS forwarding via Inbound Resolver Endpoint, on-premises hosts also resolve to the private endpoint IP — private access to AWS services over DX/VPN.
D3Design cost-optimized hybrid connectivity between on-premises and AWS for a company migrating workloads. Hard▼
Answer

Phase 1 (Migration): Site-to-Site VPN — fast, cheap, encrypted over internet. Use BGP for dynamic routing.

Phase 2 (Steady state): Order Direct Connect (1G/10G via partner). Keep VPN as failover. Terminate both at Transit Gateway (Transit VIF + DX Gateway).

Cost tips: Use Hosted Connection if bandwidth is modest. S3 Gateway Endpoint to bypass NAT charges. DX Public VIF for AWS public services to save on data transfer.

⑧ New Scenarios — VPC Core & Connectivity
V1You launch an EC2 instance in a public subnet with a public IP, but you still cannot SSH into it from your laptop. What do you check? Scenario▼
Checklist
  • 1. SG inbound rule — Is TCP port 22 allowed from your IP? Default SG denies all inbound.
  • 2. Route table — Does the subnet have 0.0.0.0/0 → igw-xxx? Without this it is not truly public.
  • 3. IGW attached — Is an Internet Gateway attached to the VPC?
  • 4. NACL stateless check — NACL must allow inbound port 22 AND outbound ephemeral ports (1024–65535) for SSH return traffic.
  • 5. Correct key pair — Right .pem file and correct username (ec2-user, ubuntu, etc.)?
  • 6. Instance running — Passed both system status checks and instance status checks?
  • 7. Correct IP — Stop/start releases the auto-assigned public IP unless using an Elastic IP.
  • 8. OS firewall — Could iptables or firewalld inside the OS be blocking port 22?
Top exam cause: SG missing inbound port 22 rule OR route table has no IGW route.
V2EC2 in VPC-A needs to talk to RDS in VPC-B (same region, same account). What are your options? Scenario▼
Answer

Option 1 — VPC Peering (simplest):

  • Create peering connection between VPC-A and VPC-B. Accept the request.
  • Add route in VPC-A: VPC-B CIDR → pcx-xxx and in VPC-B: VPC-A CIDR → pcx-xxx.
  • Update RDS SG: allow inbound DB port from VPC-A CIDR or EC2's SG ID.
  • Requirement: CIDRs must not overlap.

Option 2 — Transit Gateway: Overkill for 2 VPCs but valid if already on TGW.

Option 3 — PrivateLink: Put RDS behind NLB and expose via PrivateLink — needed if CIDRs overlap.

For 2 VPCs with non-overlapping CIDRs in the same account: VPC Peering is the cheapest and simplest choice.
V3VPC-A peers with VPC-B, and VPC-B peers with VPC-C. Users report VPC-A cannot reach VPC-C. Why, and what is the fix? Scenario▼
Answer

Root cause: VPC Peering is non-transitive. Even with A↔B and B↔C peerings, traffic cannot hop A → B → C. B does not act as a router between A and C.

Fix 1 — Direct peering: Create a new VPC Peering connection directly between VPC-A and VPC-C. Add routes in both route tables.

Fix 2 — Transit Gateway: Attach A, B, C to a TGW. TGW provides transitive routing natively without needing full-mesh peering.

Classic exam scenario: peering = non-transitive, TGW = transitive hub. Remember this distinction.
V4Your EC2 instances in a private subnet need to call the SSM API but have no internet access. How do you enable this without a NAT Gateway? Scenario▼
Answer

Create Interface VPC Endpoints (PrivateLink) for the three SSM-required services:

  • com.amazonaws.region.ssm
  • com.amazonaws.region.ec2messages
  • com.amazonaws.region.ssmmessages
  • Create each endpoint in the VPC, selecting subnets in each AZ where instances run.
  • Attach a SG to endpoint ENIs allowing inbound HTTPS (443) from the EC2 instances' SG.
  • Enable Private DNS on each endpoint so the public SSM hostname resolves to the private ENI IP.
  • Ensure VPC has enableDnsSupport = true and enableDnsHostnames = true.
Interface Endpoints are cheaper than NAT Gateway for high-volume AWS API calls and keep all traffic within AWS's network.
V5You accidentally deleted the default VPC in a region. What are the consequences and how do you recover? Scenario▼
Answer

Consequences: Resources that relied on the default VPC (e.g., EC2 launched without specifying a VPC, some older RDS configurations) will fail to launch. Existing resources in the default VPC stop working.

Recovery:

  • VPC Console → Your VPCs → Actions → Create Default VPC. AWS recreates it with the original CIDR (172.31.0.0/16), subnets per AZ, IGW, and default route table/SG/NACL.
  • CLI: aws ec2 create-default-vpc --region us-east-1
Best practice: never use the default VPC for production. Create custom VPCs with proper CIDR planning and security controls from the start.
V6Two VPCs need to share a common DNS namespace (internal.corp.local). How do you set this up? Scenario▼
Answer

Use a Route 53 Private Hosted Zone associated with multiple VPCs.

  • Create a private hosted zone for internal.corp.local.
  • Associate it with both VPC-A and VPC-B.
  • Resources in both VPCs resolve app.internal.corp.local via Route 53 Resolver. No peering or routing changes required for DNS.
  • For cross-account VPC associations: use CLI (create-vpc-association-authorization + associate-vpc-with-hosted-zone) — the console doesn't support cross-account associations.
A single private hosted zone supports up to 500 VPC associations. DNS resolution is handled entirely within Route 53 Resolver — no extra routing needed.
⑨ New Scenarios — Security Groups & NACLs
N1A web server in a public subnet can reach the internet outbound, but it cannot receive HTTP requests from the internet. What is likely misconfigured? Scenario▼
Answer

Most likely: Security Group is missing an inbound rule for port 80 and/or 443.

  • SGs deny all inbound by default. Even with outbound working, without inbound port 80/443 from 0.0.0.0/0, no external HTTP traffic reaches the instance.
  • Also check NACL inbound: Custom NACLs default to deny-all — must allow inbound 80/443.
  • Also check NACL outbound: Stateless — must allow outbound ephemeral ports (1024–65535) for return packets to clients.
SGs are stateful (return traffic automatic). NACLs are stateless (must allow both directions). This distinction is a top exam trap.
N2You want to block a specific malicious IP (1.2.3.4) from accessing resources in your subnet. Security Groups cannot block — how do you do it? Scenario▼
Answer

Use a Network ACL — the only VPC-native mechanism that supports explicit DENY rules.

  • Add an inbound DENY rule for 1.2.3.4/32 at a low rule number (e.g., rule 50).
  • Ensure the DENY rule has a lower number than any ALLOW rule you want it to override — NACLs evaluate ascending, first match wins.
  • This blocks the IP at the subnet boundary before it ever reaches instance SGs.
SGs can only ALLOW. If you need to explicitly block an IP or CIDR at the VPC level, a NACL is your only native option. For application-layer blocking, use WAF.
N3You add a NACL inbound DENY rule for port 80 at rule number 50, but HTTP traffic still gets through. Why? Scenario▼
Answer

Root cause: There is an ALLOW rule with a lower rule number (e.g., rule 40 or rule 10) that matches port 80 traffic before rule 50 is evaluated.

NACL rules are evaluated in ascending numerical order and stop at the first match. If rule 40 says "Allow all from 0.0.0.0/0", the traffic is allowed before ever reaching the DENY at rule 50.

Fix: Change the DENY rule to a number lower than any conflicting allow rule (e.g., change from rule 50 to rule 10).

Always add DENY rules with lower numbers than ALLOW rules you want them to override. This is the #1 NACL gotcha.
N4Your DB instance is in a private subnet. The app SG allows outbound to the DB on port 5432, but the app still cannot connect to the DB. What are you missing? Scenario▼
Answer

The DB SG inbound rule is missing. SGs are applied at both endpoints — both sides must allow the traffic:

  • App SG outbound: Allow TCP 5432 to DB SG or DB CIDR. ✅ Already done.
  • DB SG inbound: Allow TCP 5432 from App SG ID (or App CIDR). ❌ Missing.

Best practice: reference the App SG ID directly in the DB SG inbound rule. Any new instance added to the App SG automatically gets DB access without further changes.

Classic mistake: configuring only one side of the SG chain. Both the sender's outbound AND the receiver's inbound must permit the traffic.
N5Two EC2 instances are in the same subnet. You add a custom NACL that allows inbound SSH (port 22) from 0.0.0.0/0. But SSH still fails. Why? Scenario▼
Answer

NACLs are stateless — you must also add an outbound rule for the return traffic.

SSH uses TCP. When a client connects to port 22, the server responds on an ephemeral port on the client side (1024–65535). The NACL must allow:

  • Inbound: TCP port 22 from 0.0.0.0/0. ✅ Added.
  • Outbound: TCP ports 1024–65535 to 0.0.0.0/0 (for the SSH response packets back to the client). ❌ Missing.

Additionally, the Security Group on the instance must allow inbound port 22 — if using a custom SG (not the default), this may also be missing.

Whenever you add a custom NACL ALLOW rule for a protocol, remember to add the matching outbound ephemeral ports rule for return traffic. Forgetting this is the most common NACL mistake.
N6You want to allow EC2 instances in Subnet-A to access RDS in Subnet-B, but no other resources in Subnet-A should reach anything in Subnet-B. How do you implement this with SGs? Scenario▼
Answer

Use Security Group referencing for precise control:

  • Create a dedicated SG for the EC2 instances that need DB access: sg-app.
  • Create the RDS SG: sg-rds. Add an inbound rule: Allow TCP 5432 (or 3306) from source = sg-app.
  • Only instances attached to sg-app can reach RDS on port 5432. No other instances in Subnet-A have sg-app attached, so they are blocked.
  • This is more precise than CIDR-based rules because it controls access at the instance level, not subnet level.
SG referencing (using another SG as the source) is the AWS best practice for database access control. It works even when IPs change and scales automatically when new instances are added to the referenced SG.
⑩ New Scenarios — Routing & Traffic Flow
R1An instance in a private subnet needs internet access. You deploy a NAT Gateway in the same private subnet. It still doesn't work. Why? Scenario▼
Answer

Root cause: NAT Gateway must be deployed in a PUBLIC subnet, not the private subnet.

The NAT Gateway itself needs internet access to forward traffic. It must be in a subnet with a route to an Internet Gateway (0.0.0.0/0 → igw-xxx) and must have an Elastic IP.

Correct architecture:

  • NAT Gateway lives in a public subnet (has IGW route + EIP).
  • Private subnet's route table has 0.0.0.0/0 → nat-gateway-id.
  • Traffic: Private EC2 → NAT GW (public subnet) → IGW → Internet.
Deploy one NAT Gateway per AZ. Using a NAT GW in AZ-a for instances in AZ-b incurs cross-AZ data transfer charges.
R2Your VPC has both a NAT Gateway route and an S3 Gateway Endpoint in the same route table. Which path does S3 traffic take? Scenario▼
Answer

The S3 Gateway Endpoint route wins — AWS injects specific S3 prefix list routes (e.g., pl-63a5400a) that are more specific than the 0.0.0.0/0 NAT Gateway route.

IP routing always follows the most specific route (longest prefix match). The S3 prefix list covers only S3 IPs and is more specific than the default route, so S3 traffic bypasses NAT and goes directly through the endpoint.

This is exactly why Gateway Endpoints save money: S3 traffic bypasses the NAT Gateway (which charges per GB) and routes to S3 for free through the endpoint.
R3You have a shared services VPC (DNS, Active Directory, monitoring) and 20 spoke VPCs that need access to it. What is the most scalable design? Scenario▼
Answer

Hub-and-spoke with Transit Gateway:

  • Attach the Shared Services VPC and all 20 spoke VPCs to a TGW.
  • Create a separate TGW route table for spoke VPCs that only has a route to the Shared Services VPC — NOT to other spokes. This prevents unintended spoke-to-spoke traffic.
  • The Shared Services VPC route table has routes back to all spoke CIDRs via TGW.

Alternative — PrivateLink: Expose shared services (internal API, monitoring endpoint) via NLB + Endpoint Service. Each spoke creates an Interface Endpoint. Works even with overlapping CIDRs. Better for least-privilege if you only need specific service access rather than full network access.

TGW = full bidirectional network access. PrivateLink = unidirectional, service-specific access. Choose based on whether full connectivity or just service access is needed.
R4Traffic from your VPC to on-premises works over Direct Connect, but after a change, on-premises can reach some subnets but not others. What might cause this? Scenario▼
Answer
  • BGP route not advertised: New subnets may not be advertised via BGP over the DX Private VIF. Check which CIDRs are propagated in the VGW/TGW route tables.
  • Route table not updated: A static route for the new subnet may be missing on the on-premises router.
  • Security Group: New subnets have different SGs that don't allow inbound from on-premises CIDR ranges.
  • NACL change: A custom NACL was applied to the new subnets with deny rules.
  • On-prem routing: A more specific route on the on-premises side points elsewhere for those subnet CIDRs.
Start with BGP route table verification on both sides. Use VPC Reachability Analyzer to confirm the AWS-side logical path is valid from DX attachment to target instance.
R5Your VPC has two route tables. Subnet-A is associated with RT-1 and Subnet-B is associated with RT-2. RT-1 has no route to Subnet-B's CIDR. Can an instance in Subnet-A talk to an instance in Subnet-B? Scenario▼
Answer

Yes — intra-VPC communication within the same VPC does NOT require explicit route table entries.

Every VPC has a local route automatically added to every route table: VPC CIDR → local. This route cannot be deleted and it covers all subnets within the VPC.

So even if RT-1 has no explicit route to Subnet-B's CIDR, the local route handles it — as long as both subnets share the same VPC CIDR (or the VPC's secondary CIDRs).

Traffic is still subject to SG and NACL rules.

You only need explicit routes for traffic going OUTSIDE the VPC (internet via IGW, on-premises via VGW, other VPCs via peering/TGW). Intra-VPC routing is always handled by the implicit local route.
⑪ New Scenarios — Subnets, IPs & CIDR
C1You create a subnet with CIDR 10.0.1.0/24. How many usable IP addresses do you actually get for EC2 instances? Scenario▼
Answer

/24 = 256 total addresses. AWS reserves 5 IPs per subnet:

  • 10.0.1.0 — Network address
  • 10.0.1.1 — VPC router
  • 10.0.1.2 — DNS resolver (VPC base +2)
  • 10.0.1.3 — Reserved by AWS for future use
  • 10.0.1.255 — Broadcast address

Usable IPs = 256 − 5 = 251

Always subtract 5 from total address count for AWS subnet calculations. This is a standard exam question.
C2You are running out of IP addresses in your VPC CIDR (10.0.0.0/24). You cannot recreate the VPC. What are your options? Scenario▼
Answer

AWS supports adding secondary CIDR blocks to an existing VPC without recreating it.

  • VPC Console → Edit CIDRs → Add IPv4 CIDR block (e.g., 10.1.0.0/24).
  • Create new subnets from the secondary CIDR and update route tables, SGs, and NACLs as needed.
  • Limit: Cannot overlap with existing VPC CIDRs or any peered VPC's CIDRs. Max 5 CIDRs per VPC (soft limit).

Other options: Enable IPv6 (AWS assigns a /56 block — massive address space). Reclaim unused IPs by right-sizing or removing idle resources.

Planning lesson: always start VPCs with a /16 — even if you only use a fraction initially. Running out of IPs later is painful and disruptive to fix.
C3An EC2 instance has a private IP (10.0.1.50) and you need it to keep the same public IP across stop/start cycles. How? Scenario▼
Answer

Use an Elastic IP Address (EIP).

  • Allocate an EIP from the console and associate it with the instance's ENI.
  • The EIP is a static public IPv4 address that persists across stop/start cycles (unlike auto-assigned public IPs, which are released on stop).

Cost: Free when associated with a running instance. Charged (~$0.005/hour) when allocated but NOT associated with a running instance.

Default limit: 5 EIPs per region per account. Use EIPs sparingly — prefer DNS names (ALB/Route 53) over static IPs wherever possible for flexibility.
C4Your application calls a SaaS provider that whitelists by IP. After replacing your NAT Gateway with a new one, calls start failing. Why, and how do you prevent this? Scenario▼
Answer

Root cause: Each NAT Gateway gets its own Elastic IP. The new NAT Gateway has a different EIP — so outbound traffic now appears from a new IP that is not on the SaaS provider's whitelist.

Immediate fix: Provide the new EIP to the SaaS provider for whitelisting.

Preventative fix: When creating the new NAT Gateway, explicitly reuse the same EIP from the old one:

  • Disassociate the EIP from the old NAT Gateway (or delete old NAT GW — EIP is released back to your account).
  • When creating the new NAT Gateway, select that existing EIP instead of allocating a new one.
Treat your NAT Gateway EIPs like production assets — document them and plan for EIP reuse when replacing NAT Gateways to avoid SaaS/firewall disruptions.

C6You need two EC2 instances to communicate via a fixed private IP even when one is replaced. How do you design this? Scenario▼
Answer

Use a secondary Elastic Network Interface (ENI) with a pre-assigned private IP:

  • Create a secondary ENI in the target subnet and assign it a specific private IP (e.g., 10.0.1.100).
  • Attach this ENI to the primary instance. The consuming instance always connects to 10.0.1.100.
  • When the primary instance is replaced, detach the ENI and re-attach it to the new instance. The IP stays the same — no config change needed by the consumer.

Alternative: Use an internal NLB or a Route 53 private hosted zone record — update the DNS record on replacement instead of moving the ENI.

The ENI hot-swap pattern is a classic HA technique for stateful services (databases, license servers, network appliances) that can't use load balancers. ENIs can be moved between instances in the same AZ in seconds.
