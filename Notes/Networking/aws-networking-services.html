① Network Foundations — Core VPC infrastructure, subnets, routing, gateways
Amazon VPC Virtual Private Cloud Your isolated virtual network in the cloud ▼

What it is

Amazon VPC lets you provision a logically isolated section of the AWS Cloud where you launch AWS resources in a virtual network you define. It closely mirrors an on-premises data-center network — but runs on AWS's scalable infrastructure. Every AWS account gets a default VPC per region automatically.

Core components

  • Subnets — Segments of a VPC's IP space. Public subnets route to an Internet Gateway; private subnets do not. Each subnet maps to exactly one Availability Zone.
  • Route Tables — Contain rules (routes) that determine where network traffic is directed. Each subnet must be associated with one route table.
  • Internet Gateway (IGW) — Allows communication between your VPC and the public internet. Horizontally scaled, redundant, highly available.
  • NAT Gateway — Managed NAT service. Allows private-subnet resources to initiate outbound internet traffic without being directly reachable from the internet. Deployed per AZ, billed per hour + data processed.
  • Elastic Network Interface (ENI) — A logical NIC. Carries a private IP (and optionally public IP). Can be attached/detached from EC2 instances.
  • Security Groups — Stateful, instance-level virtual firewalls. Allow rules only (no explicit deny). Return traffic is automatically allowed.
  • Network ACLs (NACLs) — Stateless, subnet-level firewalls. Support both allow and deny rules. Rules evaluated in number order; explicit DENY works here.
  • VPC Peering — Direct private routing between two VPCs (same or different accounts/regions). Non-transitive — traffic cannot flow through a peered VPC to a third.
  • VPC Endpoints (Gateway) — Route table entries that direct traffic to S3 or DynamoDB privately, without IGW or NAT. Free of charge.
  • VPC Endpoints (Interface / PrivateLink) — ENIs with private IPs that connect your VPC to supported AWS services over the private network. Charged hourly + data.
  • Egress-Only Internet Gateway — Stateful gateway for IPv6 traffic; allows outbound-only IPv6 from VPC to internet (equivalent of NAT for IPv6).
  • VPC Flow Logs — Capture metadata about IP traffic (source/dest IP, port, protocol, bytes, action). Stored in CloudWatch Logs or S3. Not real-time packet capture.
  • Traffic Mirroring — Copies actual packet data from ENIs to a target (another ENI or NLB) for deep-packet inspection.
  • Reachability Analyzer — Automated logical network path analysis between two VPC endpoints. Identifies configuration blockers without sending real traffic.
  • Network Access Analyzer — Identify unintended network access to resources across VPCs. Checks against user-defined access requirements.

CIDR & IP addressing

A VPC requires a CIDR block (e.g., 10.0.0.0/16). RFC 1918 private ranges recommended. You can add secondary CIDR blocks. Subnets are subsets of the VPC CIDR; AWS reserves 5 IPs per subnet (network, VPC router, DNS, future, broadcast).

AWS Transit Gateway (TGW) Hub-and-Spoke Routing Central cloud router connecting VPCs and on-premises ▼

What it is

Transit Gateway acts as a highly scalable cloud router — a central hub that connects VPCs, VPN connections, and Direct Connect gateways. Instead of managing point-to-point VPC peering (which grows as N(N-1)/2), each network connects once to TGW and can reach all others through the hub.

Key concepts

  • Attachments — What you connect to TGW: VPC, VPN, Direct Connect Gateway, peering connection to another TGW, or Connect (for SD-WAN).
  • Route Tables — TGW has its own route tables (separate from VPC route tables). You can have multiple route tables for traffic segmentation.
  • Associations & Propagations — An attachment is associated with one route table. Routes can be propagated automatically or added statically.
  • Inter-Region Peering — Connect TGWs across regions to build a global backbone. Traffic transits the AWS private network.
  • Multicast — TGW supports multicast traffic distribution across VPCs (useful for financial market data feeds, video distribution).
  • TGW Connect — Native integration of SD-WAN appliances using Generic Routing Encapsulation (GRE) tunnels over Direct Connect or VPN.

AWS PrivateLink Private Service Connectivity Expose services privately across VPCs without peering complexity ▼

What it is

PrivateLink provides private connectivity between VPCs, AWS services, and on-premises applications entirely within the AWS network — no internet, no NAT, no VPN required. A service provider exposes their service via a Network Load Balancer; consumers access it via a VPC Interface Endpoint (an ENI with a private IP in their VPC).

How it works

  • Service provider side — Deploys application behind an NLB, creates an Endpoint Service.
  • Consumer side — Creates an Interface Endpoint in their VPC. AWS provisions ENIs with private IPs in specified subnets.
  • Traffic flows: Consumer ENI → PrivateLink → Provider NLB → Application. Never touches the internet.
  • Supports cross-account and cross-region (via inter-region VPC peering + PrivateLink) access.

Use cases

  • SaaS providers exposing services to customers without complex peering
  • Private access to AWS services (SSM, S3, EC2 API, Secrets Manager, etc.) without IGW
  • Overlapping CIDR environments — PrivateLink doesn't require non-overlapping IPs unlike VPC peering
  • Compliance-sensitive workloads that must never traverse the internet

AWS Cloud WAN Managed Global WAN Build and manage a global WAN from a single console ▼

What it is

Cloud WAN is a fully managed service for building, managing, and monitoring a unified global WAN. It abstracts the complexity of connecting VPCs, on-premises, SD-WAN, and branch offices across multiple regions behind a declarative policy model — all visible from one dashboard.

Key concepts

  • Global Network — The top-level container that represents your WAN.
  • Core Network — The AWS-managed backbone. Defined by a network policy (JSON). Cloud WAN automatically provisions the underlying infrastructure.
  • Segments — Logical routing domains within the core network (e.g., Production, Development). Segments prevent cross-traffic by default, providing built-in segmentation.
  • Attachments — VPCs, VPNs, Direct Connect, and SD-WAN Connect attachments.
  • Network Policy — A JSON document describing segments, segment actions, and attachment policies. Changes are versioned and apply globally.

② Hybrid & Global Connectivity — On-premises to AWS
AWS Direct Connect (DX) Dedicated Private Connection Private fiber link from your data center to AWS ▼

What it is

Direct Connect provides a private, dedicated network connection between your on-premises data center, office, or colocation environment and AWS — bypassing the public internet entirely. It uses an Ethernet cable (single-mode fiber) from your router to an AWS Direct Connect location.

Connection types

  • Dedicated Connections — Physical port dedicated to a single customer. Available at 1 Gbps, 10 Gbps, 100 Gbps. Ordered directly through AWS.
  • Hosted Connections — Sub-1G connections (50 Mbps to 500 Mbps) provisioned by an AWS Direct Connect Partner. Also up to 10 Gbps hosted connections.

Virtual Interfaces (VIFs)

  • Private VIF — Access resources in your VPC via private IP addresses. Terminates at a Virtual Private Gateway (VGW) or Direct Connect Gateway.
  • Public VIF — Access all AWS public services (S3, DynamoDB, etc.) using public IP addresses, over the private link.
  • Transit VIF — Connect to a Direct Connect Gateway attached to Transit Gateways.

Direct Connect Gateway

A global resource that lets you connect your DX location to VPCs in any AWS region (except China) via a single interface. Required when connecting DX to Transit Gateway.

Resilience patterns

  • Maximum Resiliency — Two separate DX connections at two separate DX locations. Recommended for critical production workloads.
  • High Resiliency — Two connections at one location, or one connection at two locations.
  • DX + VPN backup — DX as primary, IPSec VPN as failover (cheaper than dual DX).

AWS Site-to-Site VPN IPSec VPN Encrypted tunnel between on-premises and your VPC ▼

What it is

Site-to-Site VPN creates an encrypted IPSec tunnel between your on-premises network (Customer Gateway) and an AWS VPC or Transit Gateway (Virtual Private Gateway). Runs over the public internet. Quick to provision — minutes to hours.

Components

  • Virtual Private Gateway (VGW) — The AWS-side VPN concentrator. Attached to a VPC. Supports static and dynamic (BGP) routing.
  • Customer Gateway (CGW) — Logical representation of your on-premises VPN device in AWS. Requires a public IP and VPN software/hardware.
  • VPN Connection — Always creates two tunnels for redundancy (across different AWS endpoints). Each tunnel supports up to 1.25 Gbps.

Routing options

  • Static routing — You manually specify the CIDR ranges to advertise to AWS.
  • Dynamic routing (BGP) — Routes learned automatically. Required for multiple VPN connections and failover.

Accelerated Site-to-Site VPN

Uses AWS Global Accelerator to route VPN traffic to the nearest AWS edge location, reducing latency and improving consistency. Useful for geographically distant offices.

VPN CloudHub

Connect multiple Customer Gateways to a single VGW to enable hub-and-spoke communication between on-premises sites via AWS. Sites communicate via AWS backbone, not directly.

Tags

IPSec AES-128/256 Fast setup Two tunnels per connection ~$0.05/hr per connection
AWS Client VPN Remote Access VPN Remote employees securely access AWS and on-premises resources ▼

What it is

Client VPN is a fully managed, elastic OpenVPN-based remote-access VPN service. Employees use a VPN client on their device to connect to AWS and on-premises resources. AWS handles provisioning, capacity, and scaling automatically.

Key features

  • Authentication: Certificate-based (mutual TLS), Active Directory (via AWS Managed AD or on-prem AD), SAML 2.0 (SSO).
  • Split tunnel — Only VPC-bound traffic goes through the VPN; internet traffic exits locally. Saves bandwidth and cost.
  • Full tunnel — All traffic routes through the VPN endpoint.
  • Client-to-client communication can be enabled or disabled.
  • Integrates with AWS CloudWatch for connection logging and monitoring.
  • Connection authorization rules restrict which users access which networks.

③ Edge & Content Delivery — Global performance at the edge
Amazon CloudFront CDN Global content delivery network — cache at the edge ▼

What it is

CloudFront is AWS's CDN service. It caches copies of content (static files, video, APIs, dynamic content) at 600+ Points of Presence (PoPs) globally, serving end users from the nearest edge location to minimize latency.

Key concepts

  • Distribution — A CloudFront configuration. Has an auto-generated domain (*.cloudfront.net) or custom domain via Route 53/CNAME.
  • Origin — Where CloudFront fetches content if not cached: S3 bucket, ALB, EC2, API Gateway, or any HTTP server.
  • Cache Behaviors — Rules mapping URL patterns to origins and cache settings. E.g., /api/* → no cache, /static/* → cache 1 day.
  • TTL (Time to Live) — How long CloudFront caches an object. Set via Cache-Control headers from origin or CloudFront policy.
  • Signed URLs / Cookies — Restrict content to authenticated users. Useful for premium media, private downloads.
  • Origin Access Control (OAC) — Restricts direct S3 bucket access — only CloudFront can access it (replaces legacy OAI).
  • Lambda@Edge — Run Node.js/Python Lambda functions at CloudFront edge locations. Triggers: Viewer Request, Origin Request, Origin Response, Viewer Response. Useful for A/B testing, auth headers, URL rewrites.
  • CloudFront Functions — Lightweight JavaScript functions at edge PoPs (lower latency than Lambda@Edge, cheaper, limited runtime). Best for simple header manipulation and URL redirects.
  • Field-Level Encryption — Encrypt specific POST fields at the edge before forwarding to origin.
  • Price Classes — Restrict distribution to geographic regions to reduce cost (e.g., only North America + Europe).
  • Real-Time Logs — Stream request logs to Kinesis Data Streams for near-real-time analytics.

Security integration

  • AWS WAF — Attach a WAF Web ACL to a distribution to filter malicious requests at the edge.
  • AWS Shield — Standard is included free; Shield Advanced provides enhanced DDoS protection.
  • SSL/TLS — Free SSL certificates via ACM. Supports TLS 1.0 through 1.3. SNI-based HTTPS (free) or dedicated IP SSL ($600/month).

Common use cases

  • Static website acceleration (S3 + CloudFront)
  • Video on demand and live streaming (MediaPackage → CloudFront)
  • API acceleration for dynamic content
  • Security: WAF + Shield + signed URLs

Tags

600+ PoPs globally Pay-per-use: data transfer + requests HTTPS required for Lambda@Edge
Amazon Route 53 DNS + Health Checking Highly available DNS, domain registration, and traffic routing ▼

What it is

Route 53 is AWS's authoritative, highly available, and scalable DNS service. It also provides domain registration and health-checking. The "53" refers to the DNS port. Route 53 resolves domain names to IP addresses and can intelligently route traffic based on geography, latency, weights, and health.

Record types

  • A — Maps hostname to IPv4 address.
  • AAAA — Maps hostname to IPv6 address.
  • CNAME — Maps hostname to another hostname. Cannot be used at the zone apex (naked domain).
  • Alias — AWS-specific extension. Maps a hostname to an AWS resource (ELB, CloudFront, S3, etc.). Works at zone apex. Free DNS queries.
  • MX — Mail exchange records.
  • TXT — Text records (used for SPF, domain verification).
  • NS — Name server records for the zone.
  • SOA — Start of authority. Contains administrative info about the zone.

Routing policies

  • Simple — Single resource. Can return multiple IPs (client picks randomly).
  • Weighted — Split traffic by percentage. Useful for A/B testing and canary deployments. Weight 0 = no traffic.
  • Latency-based — Routes users to the AWS region with the lowest measured latency. Uses historical latency data.
  • Failover — Active-passive failover. Health checks determine primary/secondary. Automatic switch on health check failure.
  • Geolocation — Routes based on user's geographic location (continent, country, US state). Different content for different regions.
  • Geoproximity — Routes based on geographic proximity with optional bias. Requires Traffic Flow.
  • Multi-Value Answer — Returns multiple healthy records. Client-side load balancing. Up to 8 healthy records per response.
  • IP-based — Route based on the source IP (CIDR blocks). Useful for on-premises to AWS routing.

Health Checks

  • Monitor endpoints (HTTP, HTTPS, TCP) from multiple AWS regions.
  • Calculated health checks — combine results of multiple health checks (AND/OR logic).
  • CloudWatch Alarm health check — trigger failover based on any CloudWatch metric.
  • Private health checks — use CloudWatch metrics to monitor private VPC resources.

Route 53 Resolver

  • Resolves DNS queries for VPC resources.
  • Resolver Endpoints — Inbound (on-premises queries DNS in VPC) and Outbound (VPC queries on-premises DNS).
  • Resolver DNS Firewall — Block outbound DNS queries to known malicious domains. Uses managed domain lists or custom lists.

Tags

100% SLA $0.50/hosted zone/month Domain registration supported
AWS Global Accelerator Network-Layer Acceleration Static Anycast IPs + AWS backbone routing for global apps ▼

What it is

Global Accelerator improves availability and performance of global applications by routing traffic through the AWS global network (instead of the variable public internet) from the nearest edge location. It provides two static Anycast IP addresses as fixed entry points.

How it works

  • Users connect to a static Anycast IP. Traffic enters the AWS network at the nearest edge location (PoP).
  • AWS routes traffic internally over its private backbone (lower latency, less jitter than internet).
  • Traffic exits to regional endpoints (ALB, NLB, EC2, Elastic IP) in configured AWS regions.
  • Intelligent routing adjusts based on health, geography, and routing policies.

Key features

  • Static Anycast IPs — Two fixed IPs regardless of how many regions you add. Useful when clients whitelist IPs.
  • Endpoint Groups — Per-region groups with traffic dial (0–100%) and health check configuration.
  • Traffic Dial — Gradually shift traffic to a new region (blue/green deployments, disaster recovery testing).
  • Endpoint Weights — Distribute traffic within an endpoint group across multiple endpoints.
  • Health Checks — Automatically routes around unhealthy endpoints in under 30 seconds.
  • Custom Routing Accelerator — Route traffic to specific ports on specific EC2 instances (for gaming, IoT, VoIP).

④ Application Networking — Load balancing, service mesh, API management
Elastic Load Balancing (ELB) Load Balancer Suite Distribute traffic across targets; 4 types for different needs ▼

What it is

ELB is AWS's managed load balancing service. It automatically distributes incoming traffic across multiple targets (EC2, containers, Lambda, IPs) in one or more Availability Zones. Four types exist for different use cases.

Application Load Balancer (ALB) — Layer 7

  • HTTP/HTTPS and gRPC traffic.
  • Content-based routing: path-based (/api, /images), host-based, query string, HTTP headers, source IP.
  • Native support for WebSocket and HTTP/2.
  • Integrates with AWS WAF for request filtering.
  • Lambda functions as targets.
  • Authentication via Cognito or OIDC providers (offload auth to ALB).
  • Access Logs to S3. Target response time metrics in CloudWatch.
  • Best for: Microservices, containerized apps, REST APIs, web applications.

Network Load Balancer (NLB) — Layer 4

  • TCP, UDP, TLS traffic at ultra-high performance.
  • Handles millions of requests per second with single-digit millisecond latency.
  • Preserves client source IP (unlike ALB).
  • Static IP per AZ or Elastic IP support (useful for IP whitelisting).
  • TLS termination at NLB (offload from targets).
  • PrivateLink endpoint service — NLB fronts a PrivateLink service.
  • Best for: High throughput, low latency, gaming, financial apps, IoT, non-HTTP protocols.

Gateway Load Balancer (GWLB) — Layer 3

  • Deploys, scales, and manages third-party network virtual appliances (firewalls, IDS/IPS, deep packet inspection).
  • Operates at Layer 3, uses GENEVE protocol (port 6081) to encapsulate traffic.
  • All traffic flows to appliances before reaching destination — transparent bumping in the wire.
  • Best for: Centralized network security inspection, next-gen firewalls (Palo Alto, Fortinet, etc.).

Classic Load Balancer (CLB) — Legacy

Predecessor to ALB/NLB. Layer 4 and limited Layer 7. Use only for existing EC2-Classic workloads. Do not use for new applications.

Common ELB features (all types)

  • Health checks — auto-remove unhealthy targets from rotation.
  • Cross-zone load balancing — distribute evenly across all AZs (free for ALB, optional for NLB/GWLB).
  • Connection draining / Deregistration delay — in-flight requests complete before target is removed.
  • Sticky sessions — route same client to same target (ALB: application cookie or duration-based; NLB: 5-tuple).
  • SSL/TLS termination — offload decryption from targets using ACM certificates.

Tags

ALB: L7 HTTP NLB: L4 TCP/UDP GWLB: L3 appliances Managed, auto-scaling
Amazon API Gateway API Management Create, publish, manage, and secure APIs at scale ▼

What it is

API Gateway is a fully managed service to create, publish, maintain, monitor, and secure REST, HTTP, and WebSocket APIs at any scale. It acts as the "front door" to backend services: Lambda, EC2, ECS, any HTTP endpoint.

API types

  • REST API — Feature-rich. Supports request/response transformation, API keys, usage plans, caching, custom authorizers, request validation, and canary deployments.
  • HTTP API — Lightweight, lower latency, lower cost (~70% cheaper than REST). Supports OIDC/JWT auth, Lambda, HTTP proxy. No usage plans or request transformation.
  • WebSocket API — Bidirectional, persistent connections. Route messages based on content. Ideal for real-time chat, live dashboards, notifications.

Key features

  • Authorizers — Lambda Authorizer (custom auth logic) or Cognito User Pools (JWT).
  • Caching — REST API supports response caching (0.5 GB to 237 GB). TTL configurable per method.
  • Throttling — Account-level (10,000 req/s, 5,000 burst) and per-stage/method limits. Returns 429 on breach.
  • Usage Plans & API Keys — Rate limit specific customers/partners.
  • Stages & Deployments — Deploy to named stages (dev, prod). Canary deployments allow % traffic to new version.
  • Private API — Access only from within a VPC via interface endpoint. Blocks all public internet access.
  • Mutual TLS (mTLS) — Require client certificates for API authentication.
  • Request/Response Mapping — Transform payloads between clients and backends using VTL (Velocity Template Language).
  • WAF Integration — Attach WAF Web ACL to a REST API stage.

Tags

REST / HTTP / WebSocket Pay per API call 1 million calls/month free tier

What it is

VPC Lattice is a fully managed application networking layer that simplifies service-to-service communication across VPCs and accounts. It handles service discovery, connectivity, and access control without requiring manual VPC peering, TGW, or PrivateLink setup per service.

Key concepts

  • Service — Represents a VPC resource (Lambda, ECS, EC2, ALB, K8s pods). Exposed to consumers through the service network.
  • Service Directory — Central registry of all Lattice services within an account.

  • Listeners & Rules — HTTP/HTTPS/gRPC routing rules on services (path-based, header-based routing).
  • Target Groups — Backends: EC2 instances, Lambda, ALB, IP addresses.

AWS App Mesh Service Mesh (Envoy) Sidecar-based service mesh for microservices observability and traffic control ▼

What it is

App Mesh is a service mesh based on the Envoy proxy. It provides consistent observability and traffic controls for microservices by injecting an Envoy sidecar proxy alongside each service. Captures metrics, logs, and traces; controls traffic routing, retries, and circuit breaking — without changing application code.

Key components

  • Mesh — The logical boundary for your microservice application.
  • Virtual Service — An abstraction for a real service. Clients call the virtual service; App Mesh resolves it to actual backends.
  • Virtual Router — Routes traffic based on HTTP headers, path, method, or weight. Enables weighted routing for canary deployments.
  • Virtual Node — Represents a deployment (ECS task, EKS pod, EC2 instance). Has listeners and backends defined.
  • Envoy Proxy — Sidecar that intercepts and manages all traffic in/out of each service.

Capabilities

  • Traffic shifting for canary/blue-green deployments
  • Circuit breaking, retries, and timeouts
  • mTLS between services (zero-trust)
  • Observability: Prometheus metrics, Jaeger/X-Ray traces, CloudWatch
  • Works with ECS, EKS, EC2, and on-premises Kubernetes

AWS WAF Web Application Firewall Filter HTTP/HTTPS requests — block SQLi, XSS, bots ▼

What it is

AWS WAF filters and monitors HTTP/HTTPS requests to your web applications, blocking common web exploits that could affect availability, compromise security, or consume excessive resources.

Key concepts

  • Web ACL — A collection of rules. Associated with CloudFront, ALB, API Gateway, AppSync, or Cognito User Pool.
  • Rules — Conditions that inspect request components (IP, URI, query string, headers, body, HTTP method). Action: Allow, Block, Count, or CAPTCHA.
  • Rule Groups — Reusable collections of rules. Can be your own, AWS Managed, or from AWS Marketplace.
  • AWS Managed Rules — Pre-built rule groups: Core Rule Set (OWASP Top 10), Known Bad Inputs, IP Reputation, Bot Control, Anonymous IP List, SQL Database, PHP Application, etc.
  • Rate-based rules — Automatically block IPs exceeding a request rate threshold in a 5-minute window.
  • Bot Control — Identify and manage bots (crawlers, scrapers, scanners). Distinguish good bots (Googlebot) from bad bots.
  • Fraud Control / Account Takeover Protection (ATP) — Detect credential stuffing attacks on login endpoints.
  • Labels — Rules can add labels to requests for downstream rule evaluation.
  • Logging — Full request logs to Kinesis Firehose, CloudWatch, or S3.

WAF Capacity Units (WCUs)

Rules consume WCUs. Default Web ACL capacity is 1,500 WCUs. Complex regex rules cost more WCUs than simple string match rules.

AWS Shield DDoS Protection Managed DDoS protection — Standard free, Advanced paid ▼

What it is

AWS Shield provides protection against Distributed Denial of Service (DDoS) attacks targeting AWS applications.

Shield Standard (Free)

  • Attack diagnostics — Near-real-time visibility into attacks with CloudWatch metrics.
  • Automatic application-layer DDoS mitigation (with WAF rules).
  • Proactive engagement — DRT proactively contacts you during events.
  • Global threat dashboard for attack intelligence.
  • Protects: CloudFront, Route 53, ALB, ELB, EC2 Elastic IPs, Global Accelerator.

AWS Network Firewall Stateful Network Firewall Managed VPC-level stateful firewall with Suricata rules ▼

What it is

Network Firewall is a managed, stateful network firewall deployed within your VPC. It uses the open-source Suricata rule engine and supports signature-based threat detection, protocol anomaly detection, and domain-list blocking.

Key features

  • Stateful inspection — Tracks connection state. Allows return traffic automatically.
  • Suricata-compatible rules — Use existing Suricata IPS/IDS rule sets or write custom rules.
  • Stateless rules — High-throughput packet filtering (similar to NACLs) evaluated before stateful rules.
  • Domain list filtering — Block outbound HTTP/HTTPS traffic to malicious or unauthorized domains (SNI and HTTP Host header inspection).

AWS Verified Access Zero Trust Application Access VPN-less secure access to corporate applications using Zero Trust ▼

What it is

Verified Access provides secure access to corporate applications without a VPN, based on AWS Zero Trust principles. It evaluates each request in real time against access policies — checking identity, device posture, and other signals — before granting access to specific applications.

How it works

  • User authenticates through an Identity Provider (OIDC: Okta, Azure AD, Cognito, etc.).
  • Optionally, device posture is evaluated via AWS Verified Access Trust Provider (AWS IoT Device Management, Jamf, CrowdStrike).
  • Access policies (Cedar language) specify which users/devices can access which applications.
  • Sits in front of internal applications (ALB, EC2, ECS) as a reverse proxy.
  • No VPN client needed for end users.
