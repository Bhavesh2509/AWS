Cluster & Node Issues
Q1Worker nodes are in NotReady state after a routine update. Control plane is healthy. How do you diagnose?Hard▼
  • Run kubectl get nodes then kubectl describe node <node> — check Conditions and Events
  • SSH or SSM into node: sudo systemctl status kubelet
  • Kubelet logs: sudo journalctl -u kubelet -n 100
  • Common causes: kubelet version mismatch after AMI update, broken VPC CNI, missing IAM permissions, disk/memory pressure
  • CNI issue: check aws-node DaemonSet pods in kube-system
Q2Managed node group stuck in Degraded status. New nodes won't join the cluster.Hard▼
  • Node IAM role: needs AmazonEKSWorkerNodePolicy, AmazonEKS_CNI_Policy, AmazonEC2ContainerRegistryReadOnly
  • Security groups: node SG must allow inbound from control plane SG on ports 443 and 10250
  • Check aws-auth ConfigMap — node role ARN must be present
  • Launch Template: verify --apiserver-endpoint and --b64-cluster-ca bootstrap args
  • Subnet: ensure enough IPs and correct tags (kubernetes.io/cluster/<name>: shared)
Q3Nodes are continuously evicted and replaced. App keeps getting disrupted. How do you identify the cause?Medium▼
  • kubectl describe node — look for MemoryPressure, DiskPressure, PIDPressure conditions
  • kubectl top nodes for live usage
  • If eviction is from Cluster Autoscaler, check CA logs in kube-system
  • Check PodDisruptionBudgets — are they too restrictive?
  • Spot instances: check for EC2 spot interruption notices via instance metadata
  • Fix: set appropriate requests and limits to avoid noisy-neighbor evictions
Pod & Workload Issues
Q4Pod stuck in Pending indefinitely. No errors in pod logs. What do you check?Easy▼
  • kubectl describe pod <name> — read the Events section at the bottom
  • Insufficient resources: no node can satisfy the CPU/memory request
  • Node selector / affinity doesn't match any node label
  • Taints on nodes not tolerated by the pod
  • PVC in Pending state (storage not provisioned)
  • Cluster Autoscaler hit max node count — check CA logs
Q5Pod is in CrashLoopBackOff. Works locally. How do you approach?Medium▼
  • Get last crash logs: kubectl logs <pod> --previous
  • Check env vars, ConfigMaps, Secrets — missing DB endpoint or wrong credentials
  • Check liveness probe — threshold too aggressive causing premature kill
  • Check if OOMKilled: kubectl describe pod → Last State section
  • Architecture mismatch: ARM vs x86 container image on wrong node type
  • Debug with: kubectl run -it debug --image=<image> -- /bin/sh
Q6Deployment has 3 replicas but 2 pods show ImagePullBackOff. One pod is running fine.Easy▼
  • Image deleted from ECR after one pod was already running with a cached image
  • ECR URI typo — wrong tag or repository name
  • Node IAM role missing AmazonEC2ContainerRegistryReadOnly
  • Cross-account ECR: node role missing cross-account pull permissions
  • Private registry: imagePullSecrets not set in pod spec
  • Always check: kubectl describe pod <pod> Events for the exact pull error
Q7Pod gets OOMKilled in production but not in staging. Both have the same resource limits.Medium▼
  • Traffic volume: production serves more requests causing higher memory usage
  • Memory leak that only manifests under sustained load (staging tests are short)
  • JVM heap not tuned for container memory limits
  • Noisy neighbors: other pods on the same node consuming more memory in production
  • Check node allocatable: kubectl describe node → Allocatable section
  • Fix: increase memory limit, tune JVM flags, or fix the application leak
Q8Init container is running for 10+ minutes and the app pod hasn't started. How do you debug?Easy▼
  • Check init container logs: kubectl logs <pod> -c <init-container-name>
  • Common cause: init container is waiting on a dependency (DB, another service) that isn't ready
  • Check if the target service exists and has endpoints: kubectl get endpoints <service>
  • DNS resolution from within init container: exec in and run nslookup <service>
  • Infinite loop in init script — check the script logic
  • Resource starvation: init container CPU throttled — check limits
Networking & DNS
Q9Pods can't communicate across namespaces. Same-namespace pod-to-pod works fine.Medium▼
  • Network Policy blocking cross-namespace traffic: kubectl get networkpolicies -A
  • Check DNS: exec into pod, run nslookup <service>.<namespace>.svc.cluster.local
  • CoreDNS pods running: kubectl get pods -n kube-system -l k8s-app=kube-dns
  • VPC CNI: check aws-node pod logs for IP assignment errors
  • Service selector mismatch: kubectl get endpoints <service> -n <namespace> — if empty, selector is wrong
Q10DNS resolution is intermittently failing. Some pods resolve, some don't. CoreDNS pods look healthy.Hard▼
  • CoreDNS under-resourced — check CPU throttling on CoreDNS pods
  • ndots:5 default: every lookup does 5 DNS queries before resolving — can cause timeout spikes
  • VPC Route 53 Resolver throttling (limit: 1024 packets/sec per ENI)
  • Fix: deploy NodeLocal DNSCache to reduce CoreDNS load
  • CoreDNS logs: kubectl logs -n kube-system -l k8s-app=kube-dns
  • Debug pod: kubectl run dnstest --image=busybox --rm -it -- nslookup kubernetes
Q11ALB Controller created a load balancer but targets are "Unhealthy". Pods are running fine.Hard▼
  • Health check path: default is / — set annotation alb.ingress.kubernetes.io/healthcheck-path to your actual health endpoint
  • Security group: ALB SG must be allowed to reach node/pod SG on the target port
  • Target type ip: pod must have a routable VPC IP (VPC CNI requirement)
  • ALB Controller logs: kubectl logs -n kube-system -l app.kubernetes.io/name=aws-load-balancer-controller
  • Ingress events: kubectl describe ingress <name>
  • Health check interval and threshold — app may be slow to respond on startup
Q12Pods can't connect to RDS. Same connection string works from an EC2 instance in the same VPC.Medium▼
  • RDS security group: allow inbound on port 5432/3306 from the node SG (or pod CIDR)
  • Network Policy: check for an egress deny rule blocking pods from reaching the RDS endpoint
  • Test connectivity: kubectl exec -it <pod> -- nc -zv <rds-endpoint> 5432
  • Secret: verify DB credentials are mounted correctly in the pod
  • IRSA: if using IAM auth for RDS, check the pod's service account has the right role
IAM & Authentication
Q13Developer gets "Unauthorized" on kubectl. Your admin access works fine.Medium▼
  • Check aws-auth ConfigMap: kubectl get cm aws-auth -n kube-system -o yaml
  • Developer's IAM user/role ARN must be listed under mapUsers or mapRoles
  • If using SSO/assume-role: the exact role ARN must match including session name
  • Developer check: aws sts get-caller-identity — confirms which identity is being used
  • kubeconfig: kubectl config current-context — confirm they're hitting the right cluster
  • If cluster uses Access Entries mode: check aws eks list-access-entries instead
Q14Pod using IRSA gets "Access Denied" on S3. IAM role policy looks correct.Hard▼
  • Service account annotation: must have eks.amazonaws.com/role-arn: arn:aws:iam::<account>:role/<role>
  • IAM role trust policy: OIDC condition must match the exact namespace and service account name
  • OIDC provider: must be registered in IAM → Identity Providers for this cluster
  • Pod spec: spec.serviceAccountName must reference the annotated SA
  • Inside pod: AWS_WEB_IDENTITY_TOKEN_FILE and AWS_ROLE_ARN env vars should be auto-injected
  • Old SDK: some older SDK versions don't support web identity token — update the SDK
Q15After migrating from aws-auth to EKS Access Entries, some users can no longer access the cluster.Hard▼
  • Check auth mode: aws eks describe-cluster --name <cluster> --query cluster.accessConfig
  • List access entries: aws eks list-access-entries --cluster-name <cluster>
  • Common mistake: access entry created but no policy associated — must also run associate-access-policy
  • In API_AND_CONFIG_MAP mode both systems are active — check both
  • Use CloudTrail to see which IAM principal is calling and what's rejected
Storage Issues
Q16PVC stays in Pending. StorageClass is configured for EBS.Medium▼
  • EBS CSI driver installed: kubectl get pods -n kube-system -l app=ebs-csi-controller
  • CSI driver IRSA: the driver's service account needs EC2/EBS IAM permissions
  • StorageClass: kubectl describe sc <name> — provisioner must be ebs.csi.aws.com
  • AZ mismatch: EBS is AZ-specific — volume must be in same AZ as the pod's node
  • EBS attachment limit: some instance types max out at 28 volumes — check kubectl describe node
  • Check CSI controller logs for provisioning errors
Q17StatefulSet pod rescheduled to a new node. Stuck at ContainerCreating — can't mount EBS volume.Hard▼
  • EBS volumes are AZ-locked — if new node is in a different AZ, the volume cannot attach
  • Old node may still show the volume as attached in AWS even if terminated — force detach via EC2 console
  • Check: kubectl describe pod and kubectl describe volumeattachment
  • Long-term fix: use topology constraints to pin StatefulSet pods to one AZ, or switch to EFS for multi-AZ
Scaling & Performance
Q18HPA not scaling up even though CPU is above threshold. metrics-server is installed.Medium▼
  • kubectl describe hpa <name> — check Conditions and Events
  • Resource requests not set: HPA calculates % of requests — without requests, it can't compute utilization
  • Verify metrics: kubectl top pods
  • Cooldown: HPA has a scale-up stabilization window (default 0s but scale-down is 5 min)
  • MaxReplicas already reached — increase maxReplicas in HPA spec
  • Custom metrics: check if Prometheus adapter or KEDA is working if using non-CPU metrics
Q19Cluster Autoscaler not adding nodes even though pods are Pending due to insufficient resources.Hard▼
  • CA logs: kubectl logs -n kube-system -l app=cluster-autoscaler
  • IAM: CA needs autoscaling:* and ec2:Describe* permissions
  • ASG tags missing: k8s.io/cluster-autoscaler/enabled: "true" and k8s.io/cluster-autoscaler/<cluster-name>: "owned"
  • ASG MaxSize reached — increase the max on the Auto Scaling Group
  • Pod requests are too large for any instance type in the node group
  • PodDisruptionBudgets blocking scale-down can cause CA oscillation
Q20App is throttled on CPU constantly but CPU usage in CloudWatch looks fine. What's happening?Medium▼
  • CloudWatch shows node-level CPU — container-level throttling is different
  • CPU limits cause CFS (Completely Fair Scheduler) throttling even when node has free CPU
  • Check throttling: kubectl top pod plus container_cpu_cfs_throttled_seconds_total metric in Prometheus
  • Fix: increase CPU limit or remove it (set request but no limit) — known best practice for latency-sensitive apps
  • Java: JVM may not see container CPU count correctly — use -XX:+UseContainerSupport
Upgrades & Add-ons
Q21After upgrading control plane from 1.28 to 1.29, some workloads break. Node groups still on 1.28.Hard▼
  • Deprecated API removals: check if manifests use APIs removed in 1.29 — use kubent tool before upgrades
  • Add-on compatibility: vpc-cni, coredns, kube-proxy must also be upgraded
  • Check add-on versions: aws eks list-addons --cluster-name <cluster>
  • 1.28 nodes + 1.29 control plane: valid (2-version skew allowed) but add-ons must match control plane
  • Use kubectl convert to migrate deprecated API versions in manifests
Q22VPC CNI add-on update failed. Now some pods can't get IP addresses. How do you recover?Hard▼
  • Check aws-node: kubectl get pods -n kube-system -l k8s-app=aws-node
  • Check logs: kubectl logs -n kube-system <aws-node-pod>
  • Roll back: aws eks update-addon --cluster-name <cluster> --addon-name vpc-cni --addon-version <previous>
  • ENI limit: check if the node hit the max ENI/IP limit for its instance type
  • Temporary: delete and recreate aws-node pods to restart IPAMD
Security & Policy
Q23After enabling Pod Security Admission (PSA), deployments are rejected or pods are evicted.Medium▼
  • PSA enforces at namespace level via labels — check: kubectl get ns <name> -o yaml
  • Start with warn mode: pod-security.kubernetes.io/warn: restricted to see violations without blocking
  • Fix pods: set runAsNonRoot: true, allowPrivilegeEscalation: false, drop ALL capabilities
  • System namespaces like kube-system need privileged mode for DaemonSets
  • Audit mode: pod-security.kubernetes.io/audit: restricted — logs violations to API audit log
Q24A pod is querying EC2 metadata (169.254.169.254) to get IAM credentials instead of using IRSA. How do you block this?Hard▼
  • Set HttpPutResponseHopLimit: 1 (IMDSv2) in EC2 Launch Template — pods are an extra hop and can't reach IMDS
  • Network Policy: block pod egress to 169.254.169.254/32 from application namespaces
  • With hop limit=1, the node itself can still access IMDS but pods cannot
  • Test: kubectl exec -it <pod> -- curl http://169.254.169.254/latest/meta-data/ should time out
  • This forces all AWS API access to go through IRSA properly
Ingress & Observability
Q25Nginx Ingress returns 502 Bad Gateway for some routes but not others. All pods are healthy.Medium▼
  • Ingress controller logs: kubectl logs -n ingress-nginx -l app.kubernetes.io/component=controller
  • 502 = backend connection failed — check service port matches container port
  • Protocol mismatch: backend may need HTTPS — set annotation nginx.ingress.kubernetes.io/backend-protocol: HTTPS
  • Upstream timeout: app responds slowly — increase nginx.ingress.kubernetes.io/proxy-read-timeout
  • Empty endpoints: kubectl get endpoints <service> — if empty, service selector is wrong
  • Path rewriting: check rewrite-target annotation matches what the backend expects
Q26CloudWatch Container Insights shows no metrics. Add-on installed, pods appear running.Medium▼
  • Check DaemonSet: kubectl get pods -n amazon-cloudwatch
  • Node IAM role needs: CloudWatchAgentServerPolicy
  • Agent logs: kubectl logs -n amazon-cloudwatch <cwagent-pod>
  • ConfigMap: kubectl get cm cwagentconfig -n amazon-cloudwatch -o yaml — verify region and cluster name
  • Fluent Bit for logs is a separate DaemonSet — check it independently
Q27EKS control plane audit logs enabled but not showing up in CloudWatch.Medium▼
  • Verify enabled log types: aws eks describe-cluster --name <cluster> --query cluster.logging
  • Log group: /aws/eks/<cluster-name>/cluster — check it exists in CloudWatch
  • EKS service role needs CloudWatch Logs write permissions
  • New logs take a few minutes to appear after enabling — wait and refresh
  • Logs Insights query: fields @timestamp, @message | filter @logStream like /kube-apiserver-audit/
Q28Pods generating huge log volumes. CloudWatch Logs costs are skyrocketing. What do you do?Easy▼
  • Set CloudWatch Log Group retention policy (default is never expire)
  • Use Fluent Bit filter plugins to drop noisy/debug lines before shipping
  • Set log level to INFO in production — not DEBUG
  • Log sampling: send 1-in-N debug logs, all error logs
  • Long-term storage: ship to S3 + use Athena for queries — much cheaper than CloudWatch
  • Structured logging (JSON) enables efficient filtering and deduplication

Architecture
Q1What does AWS manage vs what you manage in EKS? Define the boundary.Easy▼
  • AWS manages: control plane (etcd, API server, controller manager, scheduler), HA across AZs, control plane patches, etcd backups
  • You manage: worker nodes (EC2), node OS patching, add-ons, workloads, IAM, networking, storage
  • Fargate: AWS also manages the node infrastructure — you only manage pods
  • Managed Node Groups: reduces operational burden but you still own AMI version and update timing
Q2What are the three compute options in EKS? When do you use each?Easy▼
  • Managed Node Groups: AWS creates/manages EC2 ASG, handles draining on updates — best for most workloads
  • Self-managed Node Groups: full control over AMI, bootstrapping, launch config — for custom configurations
  • Fargate: serverless pods, no node management — ideal for burst/batch, no DaemonSets or host networking allowed
Q3What Kubernetes version skew policy applies to EKS? What happens if nodes are 2 minor versions behind?Medium▼
  • Kubernetes supports nodes up to 2 minor versions behind the control plane (e.g., 1.27 nodes + 1.29 control plane)
  • EKS enforces this — you cannot have nodes 3+ versions behind
  • Upgrade order: control plane first → node groups → add-ons
  • EKS standard support: ~14 months per minor version
  • Extended support available at extra cost for versions past standard EOL
Networking
Q4How does the AWS VPC CNI assign IPs to pods? What are ENIs and secondary IPs here?Medium▼
  • Each EC2 node has multiple ENIs; each ENI has multiple secondary private IPs
  • IPAMD (aws-node) pre-allocates secondary IPs and assigns one per pod
  • Pods get real VPC IPs — no overlay network, direct routing
  • Max pods = (max ENIs × (IPs per ENI − 1)) + 2 — varies by instance type
  • Prefix delegation: allocates /28 CIDR per ENI — scales to more pods per node
  • Custom networking: pods can use different subnet CIDR than nodes
Q5What subnet tags are required for EKS node groups and load balancers to work correctly?Medium▼
  • All EKS subnets: kubernetes.io/cluster/<cluster-name>: shared or owned
  • Public subnets (internet-facing LBs): kubernetes.io/role/elb: 1
  • Private subnets (internal LBs): kubernetes.io/role/internal-elb: 1
  • Missing tags = ALB/NLB Controller can't discover subnets = LB provisioning fails
Q6What is the difference between ClusterIP, NodePort, and LoadBalancer service in EKS?Easy▼
  • ClusterIP: virtual IP accessible only within the cluster — internal service discovery
  • NodePort: static port on every node (30000-32767) — external via <node-ip>:<nodeport>
  • LoadBalancer: provisions an AWS ELB automatically — external traffic entry point
  • In EKS: AWS LB Controller provisions ALB (for Ingress) or NLB (for Service type LoadBalancer)
Q7What is the AWS Load Balancer Controller? ALB vs NLB — when do you use each?Medium▼
  • AWS LB Controller: Kubernetes controller that provisions ALBs for Ingress and NLBs for Service type LoadBalancer
  • ALB: Layer 7, path/host-based routing, HTTPS termination — best for HTTP microservices
  • NLB: Layer 4, ultra-low latency, preserves source IP — best for TCP/UDP, gaming, real-time
  • Target modes: instance (via NodePort) or ip (directly to pod IP — faster, needs VPC CNI)
  • Controller requires IRSA with EC2/ELB IAM permissions
IAM & Security
Q8Explain how IRSA works end-to-end. What does the OIDC provider do?Hard▼
  • EKS has an OIDC issuer URL — register it as Identity Provider in IAM
  • Kubernetes mutating webhook injects AWS_WEB_IDENTITY_TOKEN_FILE and AWS_ROLE_ARN into annotated pods
  • Pod's projected service account token contains OIDC claims (namespace, SA name)
  • AWS SDK calls sts:AssumeRoleWithWebIdentity with this token
  • IAM validates token against OIDC provider, checks trust policy conditions
  • If valid, STS returns temporary credentials scoped to the IAM role
Q9What is EKS Pod Identity? How does it differ from IRSA?Hard▼
  • EKS Pod Identity (2023): simplified alternative — no need to set up OIDC provider manually
  • Requires eks-pod-identity-agent add-on (DaemonSet) installed on the cluster
  • Association: aws eks create-pod-identity-association links a service account to an IAM role
  • Trust policy uses pods.eks.amazonaws.com as service principal — simpler than OIDC conditions
  • Advantage: same role can be reused across clusters without changing trust policy
Q10What IAM policies does a worker node need to join an EKS cluster?Easy▼
  • AmazonEKSWorkerNodePolicy — communicate with EKS control plane
  • AmazonEKS_CNI_Policy — allow VPC CNI to manage ENIs and IPs
  • AmazonEC2ContainerRegistryReadOnly — pull images from ECR
  • Optional: AmazonSSMManagedInstanceCore — SSM Session Manager (replaces SSH)
  • Optional: CloudWatchAgentServerPolicy — Container Insights metrics
Q11Difference between aws-auth ConfigMap and EKS Access Entries?Medium▼
  • aws-auth ConfigMap: legacy; IAM principals mapped to Kubernetes groups; editable via kubectl; a mistake can lock everyone out
  • EKS Access Entries (2023+): managed via EKS API; safer — cluster creator always retains access; fully auditable via CloudTrail
  • Auth modes: CONFIG_MAP, API, API_AND_CONFIG_MAP
  • Pre-defined policies: AmazonEKSClusterAdminPolicy, AmazonEKSViewPolicy, etc.
  • New clusters (1.29+) default to API_AND_CONFIG_MAP
Add-ons & Scaling
Q12Name the four default EKS managed add-ons and what each does.Easy▼
  • vpc-cni: assigns VPC IPs to pods
  • coredns: cluster DNS — resolves service names to ClusterIP
  • kube-proxy: manages iptables/IPVS rules for Service routing on each node
  • aws-ebs-csi-driver: provisions EBS volumes for PVCs (required from EKS 1.23+)
  • Managed add-ons: AWS handles patching and version compatibility
  • Conflict resolution: if you manually edit a managed add-on, set policy to OVERWRITE to prevent revert
Q13What is Karpenter? How does it differ from Cluster Autoscaler?Medium▼
  • Cluster Autoscaler: works with pre-defined ASG node groups; slower (ASG cooldowns); must define instance types in advance
  • Karpenter: provisions EC2 directly via EC2 Fleet API; picks optimal instance type per workload; no pre-defined node groups
  • Karpenter is much faster (seconds vs minutes) and more cost-efficient (spot + on-demand mix)
  • Karpenter consolidates (bin-packs) underutilized nodes automatically
  • Uses NodePool and EC2NodeClass CRDs
Q14HPA vs VPA vs KEDA — what's each for and when do you use them?Medium▼
  • HPA: scales replica count based on CPU/memory/custom metrics — best for stateless variable-load workloads
  • VPA: adjusts CPU/memory requests on pods (needs pod restart) — good for right-sizing, don't combine with HPA on same metric
  • KEDA: scales based on external events (SQS depth, Kafka lag, cron, Prometheus) — can scale to zero — ideal for event-driven and batch workloads
  • Common combo: KEDA for event-driven scale + Karpenter for node provisioning
Storage
Q15When do you choose EFS over EBS for EKS storage?Medium▼
  • EBS: block storage, single-AZ, ReadWriteOnce, high performance — for databases, stateful apps in one AZ. CSI: aws-ebs-csi-driver
  • EFS: managed NFS, multi-AZ, ReadWriteMany, shared across pods — for shared content, ML models, CMS. CSI: aws-efs-csi-driver
  • EFS has higher latency — not suitable for databases
  • Fargate only supports EFS (not EBS) for persistent storage
Q16What is a StorageClass? What key parameters matter for EBS in EKS?Easy▼
  • StorageClass defines how storage is dynamically provisioned — provisioner, type, parameters
  • EBS provisioner: ebs.csi.aws.com
  • Key params: type: gp3, encrypted: "true", optional kmsKeyId
  • volumeBindingMode: WaitForFirstConsumer — delays volume creation until pod is scheduled (ensures correct AZ)
  • reclaimPolicy: Retain vs Delete — Retain keeps EBS volume after PVC deletion
Observability & Operations
Q17What control plane log types can you enable in EKS? How do you query them?Medium▼
  • 5 types: api, audit, authenticator, controllerManager, scheduler
  • Log group: /aws/eks/<cluster-name>/cluster in CloudWatch
  • Query with CloudWatch Logs Insights — filter by log stream prefix per type
  • Useful: audit logs to see who accessed secrets — filter objectRef.resource = "secrets" and verb = "get"
  • Enabling all types is verbose — significant CloudWatch Logs cost increase
Q18What is Amazon Managed Prometheus (AMP) and how does it integrate with EKS?Medium▼
  • AMP: serverless Prometheus-compatible store — no self-managed Prometheus servers needed
  • Integration: deploy ADOT collector or Prometheus in EKS with remote-write pointing to AMP workspace endpoint
  • IRSA: ADOT/Prometheus service account needs aps:RemoteWrite IAM permission
  • Amazon Managed Grafana (AMG): connects to AMP as data source; IAM Identity Center for auth
  • ADOT = AWS Distro for OpenTelemetry — AWS-supported collector for metrics + traces
Upgrades
Q19What is the recommended upgrade order for an EKS cluster?Medium▼
  • 1. Review EKS changelog and check deprecated API usage with kubent
  • 2. Upgrade control plane: aws eks update-cluster-version
  • 3. Update managed add-ons (vpc-cni, coredns, kube-proxy) to compatible versions
  • 4. Update node groups — managed via console/CLI; self-managed by replacing nodes with new AMI
  • 5. Verify: kubectl get nodes, smoke test all workloads
  • Only one minor version at a time — no skipping (1.27 → 1.28 → 1.29)
Q20What are the EKS Optimized AMI variants? Difference between AL2, AL2023, and Bottlerocket?Medium▼
  • AL2: legacy; EOL 2025 — migrate away now
  • AL2023: current default; dnf package manager; SELinux enforcing; more minimal and secure
  • Bottlerocket: purpose-built minimal OS by AWS; immutable root filesystem; API-based config; no SSH by default (use SSM or admin container); best security posture
  • Get latest AMI ID via SSM: /aws/service/eks/optimized-ami/<version>/amazon-linux-2023/recommended/image_id
Fargate & Cost
Q21What are Fargate profiles? What are the key limitations vs EC2 nodes?Medium▼
  • Fargate profile: defines which pods run on Fargate using namespace + label selectors
  • Each pod runs in its own micro-VM — strong isolation, no noisy neighbors
  • Limitations: no DaemonSets, no privileged containers, no hostPath, no GPU, no EBS (EFS only)
  • No node SSH/SSM — only kubectl exec access
  • Slower cold start (new VM per pod) and more expensive than right-sized EC2 for sustained load
Q22How does EKS pricing work? What are you paying for beyond EC2?Easy▼
  • EKS cluster fee: $0.10/hr per cluster (control plane)
  • EC2 worker nodes: normal EC2 pricing — biggest cost; use Spot + Savings Plans
  • Fargate: ~$0.04/vCPU-hr + ~$0.004/GB-hr
  • Additional: NAT Gateway, Load Balancers, EBS/EFS, CloudWatch Logs ingestion, data transfer
  • Extended support (old k8s versions past EOL): extra per-cluster fee
Advanced Concepts
Q23What is the Secrets Store CSI Driver? How does it work with AWS Secrets Manager?Medium▼
  • Mounts secrets from Secrets Manager / SSM Parameter Store directly as volumes into pods
  • Secrets never stored in etcd — more secure than native Kubernetes secrets
  • Also install the AWS provider: secrets-store-csi-driver-provider-aws
  • Uses IRSA: pod SA role needs secretsmanager:GetSecretValue
  • SecretProviderClass CRD defines which secrets to mount
  • Auto-rotation: mounted file updates when secret rotates in Secrets Manager
Q24What is EKS private vs public endpoint? What's the recommended production setup?Medium▼
  • Public endpoint: API server reachable from internet — convenient, attack surface risk
  • Private endpoint: API server only via VPC — needs VPN/Direct Connect for remote kubectl
  • Both enabled (recommended): restrict public access via publicAccessCidrs to company IPs; nodes talk to control plane privately (stays in VPC)
  • Private-only: nodes use a private Route 53 hosted zone to resolve the API server endpoint
Q25What is GuardDuty EKS Protection? What threats does it detect?Medium▼
  • Analyzes EKS audit logs and runtime container behavior to detect threats
  • Audit Log Monitoring: anonymous access, privilege escalation, unusual exec commands, crypto mining deployments
  • Runtime Monitoring: DaemonSet agent monitors container syscalls — container escapes, reverse shells, malicious process spawning
  • Sample findings: PrivilegeEscalation:Kubernetes/PrivilegedContainer, CryptoCurrency:Runtime/BitcoinTool
  • Integrate with Security Hub or EventBridge for automated response
Q26How do you implement multi-tenancy in EKS? What isolation mechanisms exist?Hard▼
  • Namespace + RBAC: logical separation, team access scoped to their namespace — weakest isolation
  • Network Policies: restrict cross-namespace / cross-team traffic
  • Resource Quotas + LimitRanges: prevent one tenant from starving others
  • Pod Security Admission: enforce security standards per namespace
  • Node isolation: taints + tolerations + node selectors to pin tenant workloads to dedicated nodes
  • Fargate: each pod in own micro-VM — strongest isolation for SaaS multi-tenancy
  • Strict compliance: separate clusters per tenant
Q27What are topology spread constraints? Why are they critical in multi-AZ EKS?Medium▼
  • Tells the scheduler how to distribute pods across topology domains (AZs, nodes)
  • Key fields: topologyKey (e.g., topology.kubernetes.io/zone), maxSkew, whenUnsatisfiable
  • Without it: scheduler may place all pods in one AZ — AZ failure takes down everything
  • Better than pod anti-affinity: more flexible, works with any number of replicas
  • Combine with PodDisruptionBudgets to ensure enough pods survive node maintenance
Q28What is GitOps? How do Flux and ArgoCD differ for EKS deployments?Medium▼
  • GitOps: Git is the single source of truth; operator continuously reconciles cluster state to match Git
  • Flux: CNCF graduated; lightweight; Kubernetes-native CRDs; AWS recommended in EKS Blueprints
  • ArgoCD: rich UI; application-centric; supports multi-cluster from single instance; popular in enterprise
  • Both support Helm, Kustomize, plain manifests
  • EKS Blueprints (Terraform/CDK): bootstraps clusters with Flux or ArgoCD pre-installed
Q29What is EKS Anywhere? What is EKS Hybrid Nodes?Medium▼
  • EKS Anywhere: run EKS on-premises on your own hardware (bare metal or VMware); you manage the control plane
  • EKS Hybrid Nodes (2024): connect on-premises or edge servers as nodes to an AWS-managed EKS control plane
  • Use cases: data residency, low-latency edge workloads, air-gapped compliance environments
  • EKS Distro: the open-source Kubernetes distribution AWS uses — can be deployed anywhere
