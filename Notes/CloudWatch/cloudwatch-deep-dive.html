CloudWatch — Complete Services Deep Dive

Amazon CloudWatch is AWS's unified monitoring and observability platform. It collects metrics, logs, events, and traces from AWS resources, on-premises servers, and hybrid environments — giving you operational visibility across your entire stack in one place.

This guide covers every major CloudWatch service and sub-feature with full detail.

CloudWatch Metrics

The foundational data layer — time-series measurements from every AWS service

Metrics are the core building blocks of CloudWatch. A metric is a time-ordered set of data points representing a measurable value (e.g., CPU utilization, request count, latency). Over 70 AWS services automatically push metrics to CloudWatch — you get this for free without any configuration.

Key Concepts

Namespace
Logical container for metrics. AWS services use namespaces like AWS/EC2, AWS/Lambda. You create your own for custom metrics.
Dimensions
Name/value pairs that identify a metric uniquely. e.g., InstanceId=i-1234. Up to 30 dimensions per metric.
Period
The time interval over which data is aggregated. Ranges from 1 second (high-resolution) to 1 day.
Statistics
Average, Sum, Min, Max, SampleCount, pNN.NN percentiles (e.g., p99)

Monitoring Tiers

TierFrequencyCostNotes
Basic5 minutesFreeDefault for EC2; auto-enabled for most AWS services
Detailed1 minutePaidEnable per resource; required for faster alarm response
High Resolution1–10 secondsPaidCustom metrics only; use PutMetricData with StorageResolution=1

Custom Metrics

You can publish your own metrics using the PutMetricData API call. Use cases: application memory usage, business KPIs (orders/min), custom error rates, queue depths from proprietary systems.

PutMetricData API AWS SDK CLI: aws cloudwatch put-metric-data CloudWatch Agent (for host metrics)

Metric Math

Perform arithmetic across multiple metrics. You can create expressions like m1/m2*100 to calculate error rate percentages, combine metrics from different services, and use math functions like SUM(), AVG(), FILL(), IF(). Results can be used in alarms and dashboards.

Data Retention

GranularityRetention Period
<60 seconds (High-Res)3 hours
1 minute15 days
5 minutes63 days
1 hour15 months
Tip: Metrics older than 2 weeks cannot be viewed in the CloudWatch console. Use the API (GetMetricStatistics) to access older data up to 15 months.

CloudWatch Alarms

Automated watchdogs — detect threshold breaches and trigger actions

Alarms watch a single metric (or metric math expression) and perform one or more actions when the metric crosses a defined threshold. They are stateful — they transition between OK, ALARM, and INSUFFICIENT_DATA states.

Alarm States

OK — metric within threshold ALARM — metric breached threshold INSUFFICIENT_DATA — not enough data yet

Alarm Types

TypeDescriptionUse Case
Standard AlarmWatches one metric against a static thresholdCPU > 80% for 5 minutes
High-Resolution AlarmEvaluates every 10 or 30 secondsCatching brief latency spikes
Composite AlarmCombines multiple alarms using AND/OR logicAlert only when BOTH CPU and memory are high
Anomaly Detection AlarmUses ML band instead of static thresholdAlert when metric deviates from normal pattern

Actions on Alarm

SNS Notification (email, SMS, HTTP) EC2 Action (stop, terminate, reboot, recover) Auto Scaling (scale in / scale out) Systems Manager OpsItem Lambda (via SNS trigger)

Configuration Options

Evaluation Periods
Number of periods to evaluate before transitioning state. Avoids false positives from transient spikes.
Datapoints to Alarm
M out of N datapoints must breach threshold. e.g., 3 out of 5 periods — allows short dips/spikes.
Missing Data Treatment
Options: breaching, not-breaching, ignore, or missing. Controls alarm behavior when data gaps exist.
Alarm Suppression
Composite alarms can suppress child alarms during maintenance windows using "alarm suppression" rules.
Note: Alarm actions fire when state changes (OK→ALARM), not on every evaluation. To re-trigger, the alarm must return to OK and then breach again.

CloudWatch Dashboards

Customizable visualization panels for metrics, logs, and alarms

Dashboards provide a unified visual view of your AWS environment. They are fully customizable and can display metrics graphs, alarm status, log query results, and text widgets. Dashboards persist indefinitely and can be shared across accounts and regions.

Widget Types

Line / Stacked Area charts Bar charts Number (single value) Alarm status Logs query results (tables) Text (Markdown) Custom widgets (Lambda-powered HTML) Explorer widgets (multi-resource)

Key Features

Automatic Dashboards
AWS pre-builds dashboards for EC2, Lambda, RDS, etc. — available with zero setup. Shows service-level health at a glance.
Cross-Account / Cross-Region
Aggregate metrics and alarms from multiple AWS accounts and regions into a single dashboard view.
Live Data
Toggle "Live Data" to include the most recent partial period's data. Useful for real-time monitoring during deployments.
Dashboard Sharing
Share publicly (no login required), with specific users, or embedded in internal tools via snapshot links.
Pricing: First 3 dashboards (up to 50 metrics each) are free. Beyond that, $3/dashboard/month.

CloudWatch Logs

Centralized log management — collect, store, analyze logs from any source

CloudWatch Logs is the central logging service for AWS. It stores logs in log groups and log streams, and provides powerful features for querying, filtering, routing, and alerting on log data. Sources include Lambda, EC2, ECS, API Gateway, VPC Flow Logs, CloudTrail, and custom applications.

Structure

Log Group
Top-level container. Defines retention period and access control. Usually one per application or service (e.g., /aws/lambda/my-function).
Log Stream
Sequence of log events from a single source (e.g., one EC2 instance). Multiple streams exist within one log group.
Log Event
Individual timestamped message. Each event has a timestamp and a raw message string.
Retention
Configurable: 1 day to 10 years, or Never Expire. Default is Never Expire (costs money — always set retention!).

Log Tiers

TierBest ForFeatures
StandardReal-time monitoring, alertingLive Tail, metric filters, alarms, data protection, Log Insights
Infrequent Access (IA)Archival, ad-hoc forensicsLower ingestion cost; cross-account analytics; no Live Tail or metric filters

Key Features

Metric Filters — extract numeric values from log text to create CW metrics Subscription Filters — stream logs in real-time to Lambda, Firehose, OpenSearch Live Tail — stream log events in real-time in the console (like tail -f) Data Protection — mask/redact PII (SSNs, credit cards, emails) automatically Log Anomaly Detection — ML detects unusual log patterns automatically Export to S3 — batch export for long-term archival or Athena analysis
Metric Filters: Extract data like HTTP 5xx count from log lines using filter patterns. Then create alarms on those extracted metrics — bridging unstructured logs into actionable signals.

CloudWatch Agent

Host-level collector — runs on EC2, on-premises, containers

The CloudWatch Agent is a lightweight daemon that runs on your EC2 instances and on-premises servers. It collects system-level metrics and custom log files that CloudWatch cannot get through its default AWS integration — most importantly, memory and disk usage, which EC2 does NOT report by default.

What It Collects

System Metrics
CPU (per-core), memory (used/available), disk I/O, disk usage, network packets/bytes, swap usage, process-level stats
Log Files
Tails any local log file and ships it to CloudWatch Logs. Supports multi-line logs, custom timestamps, encoding.
StatsD Protocol
Receives metrics from application code via StatsD (port 8125). Push custom counters/gauges from your app without API calls.
collectd Protocol
Receives metrics from collectd daemon for advanced Linux system monitoring scenarios.

Supported Platforms

Amazon Linux 2 / 2023 Ubuntu 22.04 / 24.04 RHEL 8/9/10 Debian 12 Windows Server On-premises (x86-64 & ARM64) Kubernetes (via DaemonSet)
Critical: EC2 does NOT report memory or disk metrics by default. You MUST install the CloudWatch Agent to get memory utilization. This is a very common exam and interview question.

CloudWatch Synthetics (Canaries)

Proactive outside-in monitoring — simulate users 24/7 even without real traffic

CloudWatch Synthetics lets you create canaries — scripts that continuously run on a schedule to simulate user interactions with your endpoints, APIs, and web flows. They detect issues before real users are affected, capture screenshots, and store HAR files for debugging.

How Canaries Work

Each canary is implemented as an AWS Lambda function. Scripts are written in Node.js, Python, or Java. They run on a defined schedule (as frequently as every 1 minute) and report success/failure metrics to CloudWatch. They have access to headless Chrome via Puppeteer, Playwright, or Selenium.

Blueprint Types

BlueprintWhat It Does
Heartbeat MonitorSimple availability check — hits a URL, checks for HTTP 200
API CanaryTests GET/POST REST API endpoints with custom assertions on response body/headers
Broken Link CheckerCrawls a page and checks that all links resolve successfully
Visual MonitoringCompares screenshots to a baseline — detects UI regressions
Canary RecorderChrome extension that records your browser actions and auto-generates the canary script
GUI Workflow BuilderDefine multi-step flows: click, type, verify — tests checkout flows, login forms, wizards

Output & Integrations

CloudWatch Metrics (SuccessPercent, Duration) CloudWatch Alarms on canary failure Screenshots stored in S3 HAR files (full HTTP request/response logs) X-Ray Trace integration Application Signals service map
Use Case: Deploy a canary on your checkout flow. If it fails at 2 AM, you get paged — even though zero real users are hitting your site at that hour.

Real User Monitoring (RUM)

Client-side telemetry — see your app through real users' eyes

CloudWatch RUM collects performance and error data from real user browsers and mobile apps (iOS/Android). Unlike Synthetics which simulates users, RUM shows you actual user experience — page load times, JavaScript errors, and geographic distribution of performance issues.

What RUM Captures

Page load performance (FCP, LCP, TTFB, FID) JavaScript errors + stack traces HTTP errors from API calls User sessions and navigation flows Browser type and version breakdown Geolocation of users Mobile: iOS & Android (2025 expansion)

How It Works

You create an App Monitor and embed a small JavaScript snippet (or use ADOT SDK for mobile) in your application. RUM collects events client-side and sends them to CloudWatch. You get an automatically populated dashboard showing anomalies, error rates, and session flows.

RUM vs Synthetics

DimensionRUMSynthetics
Data SourceReal user browsers/mobileSimulated scripts (Lambda)
CoverageOnly when users visitRuns even with zero traffic
InsightGeographic, device, browser diversityFixed path, predictable coverage
Best ForPost-launch UX analysisPre-launch / continuous availability

Application Signals

Auto-instrumented APM — zero-code performance monitoring for microservices

Application Signals is CloudWatch's Application Performance Monitoring (APM) capability. It automatically instruments your applications to collect latency, fault rate, error rate, and call volume — without you writing custom instrumentation code. It provides a service topology map, SLO tracking, and deep trace correlation.

Supported Languages & Platforms

Java Python Node.js .NET Amazon EKS Amazon ECS Amazon EC2

Core Features

Services Page
Lists all discovered services sorted by highest fault rate or latency. Shows SLI compliance at a glance.
Application Map
Visual topology of services, dependencies, canaries, and clients. Shows health metrics on each edge/node.
Service Detail
Per-service deep dive: operations breakdown, p50/p90/p99 latencies, correlated X-Ray traces for any time window.
Automated Audit Findings
AI analyzes your services and surfaces significant observations, anomalies, and root cause candidates automatically.
SLOs
Define latency/availability goals per service. Track error budgets. Alert before you breach your SLA.
Transaction Search
Search individual traces/spans stored in X-Ray. Deep dive into specific user transactions for debugging.
Gen AI Observability: As of re:Invent 2025, Application Signals supports monitoring AI agents built with Bedrock AgentCore, LangChain, LangGraph, and CrewAI — tracking token usage, latency, and agent-workflow tracing end-to-end.

Container Insights

Deep container observability — ECS, EKS, Fargate, and self-managed K8s

Container Insights collects, aggregates, and summarizes metrics from containerized workloads. It gives you CPU, memory, disk, and network usage at the cluster, node, pod, task, and service level. It also captures container restart failures and crash diagnostics.

Supported Platforms

Amazon ECS (EC2 & Fargate) Amazon EKS Self-managed Kubernetes on EC2 AWS Fargate (serverless containers)

Metrics Collected

Pod / TaskCPU/memory usage, network I/O, restart count

Enhanced Container Insights

Enhanced mode (EKS only) uses the CloudWatch Observability EKS Add-on and provides additional granularity: GPU metrics, accelerated workloads, and more detailed pod-level telemetry. It also enables cross-account container observability.

Contributor Insights

Identify top-N contributors driving performance problems

Contributor Insights analyzes structured log data to identify the top contributors creating the highest load, errors, or traffic. Instead of asking "what is the error rate?", it answers "which customer, IP address, or API endpoint is causing the most errors?"

How It Works

You define a rule specifying which log group to analyze, which fields to extract as contributors (e.g., clientIp, userId), and what to count or aggregate. CloudWatch continuously processes new log events and maintains a real-time ranking of your top contributors.

Common Use Cases

ScenarioContributor FieldValue

Noisy neighborTenantId / customerIdFind customer generating most load

Heavy DB queriesQuery patternIdentify slowest queries
VPC flow logsSource IP, destination portMap unexpected traffic patterns
Integration: Works with CloudWatch Logs (any log group), AWS-vended logs (VPC Flow, Route 53, ALB access logs), and supports cross-account log analysis.

Anomaly Detection

ML-powered baseline modeling — alarms that adapt to normal patterns

CloudWatch Anomaly Detection uses machine learning to establish a baseline "normal band" for any metric. It accounts for time-of-day patterns, day-of-week cycles, and seasonal trends automatically. You can set alarms that trigger when the metric deviates from this band rather than a static threshold.

How the Model Works

Training
Uses the past 2 weeks of metric history to train. The model is continuously retrained as new data arrives.
Exclusion Windows
You can exclude known anomalous time windows (e.g., a past outage) from training data so they don't skew the model.
Band Width
Control sensitivity with a "standard deviations" slider. Narrower band = more sensitive = more alarms.
Visualization
The predicted band is visible on metric graphs as a shaded area. Deviations outside are highlighted.

Anomaly Detection for Logs

Separate from metric anomaly detection, CloudWatch Logs also has its own anomaly detection. It automatically groups log lines into patterns and flags patterns that are statistically unusual — e.g., a new error message type appearing suddenly.

Limitation: The model needs at least 2 weeks of clean data to be reliable. Newly created resources or rapidly changing systems may produce noisy results initially.

ServiceLens

Correlated observability — metrics, logs, and traces in one view

ServiceLens ties together CloudWatch metrics, CloudWatch logs, and AWS X-Ray traces to give you a unified view of application health. It provides a service map showing dependencies, and lets you drill down from a metric anomaly into correlated traces to find root cause.

Three Observability Pillars in ServiceLens

Infrastructure Monitoring
Metrics and logs for the underlying resources (EC2, RDS, Lambda) supporting your services.
Transaction Monitoring
X-Ray traces showing the path of individual requests through your distributed system. Identifies bottlenecks.
End-User Monitoring
Synthetics canaries monitoring your endpoints from outside. Detect user-facing degradation immediately.
Service Map: The visual map auto-discovers services that emit X-Ray traces. Nodes are colored by health status. Click any node to see metrics, logs, and traces for that service in context.
Relationship to Application Signals: ServiceLens is the older service map feature. Application Signals (2024+) is the newer, more powerful APM capability that extends ServiceLens with auto-instrumentation, SLOs, and audit findings. They complement each other.

CloudWatch Events → Amazon EventBridge

Event-driven automation — react to changes in AWS resources in near-real-time

CloudWatch Events was the original event routing service. It has been superseded by Amazon EventBridge, which is the current evolution and is fully backward-compatible. It delivers a near-real-time stream of events about state changes in AWS services and routes them to targets like Lambda, SQS, Step Functions, etc.

Event Types

AWS Service Events (EC2 state change, RDS failover) Scheduled Events (cron / rate expressions) Custom Events (from your application via PutEvents API) CloudTrail API call events SaaS Partner Events (via EventBridge)

AIOps & CloudWatch Investigations

Gen AI-powered root cause analysis — resolve incidents faster with automated intelligence

CloudWatch Investigations (GA June 2025) is the flagship AIOps capability. When an alarm triggers, it automatically correlates metrics, logs, traces, deployment events, and config changes to identify probable root causes — presenting findings in plain English.

AIOps Features

Automated Root Cause Analysis
AI correlates telemetry across services and time windows. Surfaces the most likely cause ranked by confidence.
5 Whys Analysis
Implements AWS's internal COE methodology. Drills recursively through causation layers to find systemic root cause.
Incident Report Generation
One click generates a full incident report from operational data — metrics, deployments, config changes, investigation timeline.
Natural Language Queries
Ask "why is latency high?" in plain English. The system navigates your operational data and responds conversationally.
MCP Server Integration
CloudWatch exposes MCP (Model Context Protocol) servers so external AI agents can query your operational data.
Strands Agent SDK
Build custom AIOps workflows as AI agents using AWS's Strands SDK — tailored incident response automation.
2025 Launch Context: AIOps in CloudWatch is AWS's response to the growing complexity of distributed systems. The goal is to reduce MTTD (mean time to detect) and MTTR (mean time to resolve) using generative AI that has been trained on AWS's own operational expertise.

CloudWatch Network Monitor

Network path health — proactive connectivity monitoring for hybrid environments

Network Monitor (also called Network Synthetic Monitor) lets you proactively test network connectivity between your VPCs and on-premises endpoints using synthetic probes. It continuously measures round-trip time (RTT) and packet loss to detect network issues before applications are impacted.

Key Concepts

Probes
Define a VPC subnet (source) and an on-premises IP (destination). AWS creates managed infrastructure to run synthetic tests between them.
Metrics
Round-trip time (RTT), packet loss percentage. Published to CloudWatch every minute.
Requirement
Requires AWS Direct Connect or VPN connectivity. Designed for hybrid network health monitoring.
Detects
DNS issues, latency spikes, connectivity failures, packet loss — all without waiting for application-level failures to surface.

Cross-Account Observability

Central monitoring hub — aggregate data from all accounts in your organization

Cross-account observability (via CloudWatch Observability Access Manager — OAM) lets you designate one AWS account as a monitoring account and link source accounts to it. You can then query logs, view metrics, and analyze traces across all accounts from a single pane of glass.

Architecture

Monitoring Account
Central account with full read access to telemetry from all linked source accounts. Runs cross-account queries.
Source Accounts
Linked accounts that share their telemetry (metrics, logs, traces) via a sink. No data is copied — accessed in-place.
Sink
A CloudWatch resource in the monitoring account that acts as the endpoint receiving links from source accounts.

What Can Be Shared Cross-Account

CloudWatch Metrics CloudWatch Logs (Insights queries) X-Ray Traces Application Signals (services & SLOs) Container Insights data Contributor Insights rules

Service Level Objectives (SLOs)

Reliability engineering built-in — define, track, and alert on service commitments

CloudWatch SLOs allow you to formally define reliability targets for your services. You specify a Service Level Indicator (SLI), set a target (e.g., 99.9% of requests under 500ms), define a time window, and CloudWatch automatically tracks your error budget and alerts when you're at risk of breaching your SLO.

Key Concepts

SLI (Indicator)
The metric being measured. e.g., percentage of requests completing under 200ms, or percentage of successful responses.
SLO (Objective)
The target. e.g., "99.5% of requests must be successful over a rolling 30-day window."
Error Budget
The allowed failure margin. If SLO is 99.9%, you have 0.1% budget. CloudWatch tracks how much is remaining and alerts when low.
Burn Rate Alarms
Alert when the error budget is being consumed too fast — before you actually breach the SLO. Enables proactive response.

Integration Points

Application Signals — auto-detected SLIs from instrumented services Synthetics canaries — canary success rate as SLI Custom CloudWatch metrics — any metric as an SLI Cross-account SLO tracking
SRE Practice: SLOs bridge the gap between business SLAs (agreements with customers) and engineering metrics. CloudWatch makes implementing Google/SRE-style reliability engineering on AWS accessible without custom tooling.
