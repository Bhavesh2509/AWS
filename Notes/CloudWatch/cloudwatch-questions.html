AWS CloudWatch — Interview Questions

Basic / Conceptual Questions
Core concepts, definitions, and fundamental knowledge about CloudWatch. These are commonly tested in AWS certifications and initial screening rounds.
B01
What is Amazon CloudWatch and what problem does it solve?
▾

Amazon CloudWatch is AWS's unified monitoring and observability service. It collects metrics, logs, events, and traces from AWS resources, on-premises servers, and hybrid cloud environments — providing a single platform for operational visibility.

Problems it solves:

  • Eliminates siloed monitoring (separate tools for server, network, database)
  • Provides actionable insights to reduce Mean Time to Resolution (MTTR)
  • Enables automated responses to operational events without manual intervention
  • Centralizes logs from all AWS services in one queryable system
Key phrase: "Single pane of glass for metrics, logs, and events across your entire AWS stack."
B02
What is a CloudWatch metric? What are namespaces and dimensions?
▾

A metric is a time-ordered set of data points representing a measurable value (e.g., CPU utilization at 5-minute intervals).

A namespace is a logical container that groups related metrics. AWS services use namespaces like AWS/EC2, AWS/Lambda, AWS/RDS. You use custom namespaces for your own metrics.

Dimensions are name/value pairs that uniquely identify a metric. For example, InstanceId=i-1234abcd distinguishes the CPU metric of one EC2 instance from another. You can filter and aggregate metrics by dimensions.

Example: Namespace: AWS/EC2, Metric: CPUUtilization, Dimension: InstanceId=i-abc123
B03
What are the three states of a CloudWatch Alarm?
▾
  • OK — the metric is within the defined threshold
  • ALARM — the metric has breached the threshold for the required evaluation periods
  • INSUFFICIENT_DATA — not enough data has been collected yet to evaluate the alarm (e.g., newly created resource, or data gap)
Gotcha: Alarm actions fire on STATE CHANGES (OK → ALARM), not on every evaluation period. If the metric stays in ALARM, the action doesn't repeat unless you configure it explicitly via SNS re-delivery or similar.
B04
Why doesn't CloudWatch show memory utilization for EC2 by default? How do you fix this?
▾

CloudWatch can only collect metrics that the underlying AWS hypervisor can observe from outside the instance. Memory usage exists inside the OS — the hypervisor cannot see it. Similarly, disk space usage at the OS level is invisible to the hypervisor.

Solution: Install the CloudWatch Agent on the EC2 instance. The agent runs inside the OS, reads system-level data (memory, disk, process stats), and publishes them as custom metrics to CloudWatch in the CWAgent namespace.

Metrics NOT available by default: Memory utilization, disk space used/free, swap usage, per-process CPU — all require the CloudWatch Agent.

B06
How long does CloudWatch retain metric data?
▾

CloudWatch automatically aggregates and rolls up metric data as it ages:

Console limitation: The CloudWatch console only shows metrics from the last 2 weeks. To query older data (up to 15 months), use the GetMetricStatistics API or CLI.
B07
What is a Composite Alarm and when would you use it?
▾

A Composite Alarm combines the state of multiple child alarms using AND/OR Boolean logic. It only enters ALARM state when the combined rule is true.

Use cases:

  • Reduce alarm noise: only alert when BOTH CPU AND memory are high (AND logic) — avoiding false positives when only one is elevated
  • Group related alarms: one composite alarm for "service unhealthy" that fires if ANY of 5 sub-alarms are in ALARM (OR logic)
  • Alarm suppression: suppress child alarm notifications during maintenance windows
Cost benefit: Only the composite alarm needs an SNS action — you can suppress SNS actions on child alarms, reducing duplicate notifications and SNS costs.
B08
What are CloudWatch Log Groups, Log Streams, and Log Events?
▾

Log Group: Top-level container for logs, usually representing one application or service (e.g., /aws/lambda/my-function). Defines retention period and IAM access control.

Log Stream: Sequence of log events from a single source within a log group. For Lambda, each function execution context gets its own stream. For EC2 with the agent, each instance gets its own stream.

Log Event: An individual timestamped message. The smallest unit in CloudWatch Logs.

Default retention: Log groups default to Never Expire, which incurs storage costs indefinitely. Always set a retention policy when creating log groups!
B09
What is a CloudWatch Metric Filter? How is it different from Logs Insights?
▾

A Metric Filter watches incoming log events in real time and extracts numeric values from matching patterns, publishing them as CloudWatch metrics. For example, count all log lines containing "ERROR" per minute and create a metric for it.

Key difference from Logs Insights:

Metric FilterLogs Insights

Can alarm on?✅ Yes (it's a metric)❌ No (not a metric)

B10
What actions can a CloudWatch Alarm trigger?
▾
  • SNS Notifications — send email, SMS, or trigger HTTP endpoint; can fan out to many subscribers
  • EC2 Actions — Stop, Terminate, Reboot, or Recover an EC2 instance
  • Auto Scaling Actions — scale in or scale out an Auto Scaling group
  • Systems Manager OpsItem — create an incident ticket in AWS Systems Manager OpsCenter
  • Lambda (indirectly) — via SNS subscription or EventBridge rule on alarm state change
Exam tip: EC2 recovery action is special — it migrates the instance to a new host if underlying hardware fails, preserving the same instance ID, private IP, and Elastic IP.
B11
What is CloudWatch Anomaly Detection and how does it work?
▾

Anomaly Detection uses machine learning to analyze historical metric data and creates a "normal band" showing expected values over time. The model accounts for time-of-day, day-of-week, and seasonal patterns automatically.

How to use:

  • Enable it on any metric via the console or API
  • The model trains on the past 2 weeks of data
  • Set a CloudWatch Alarm using the anomaly detection band instead of a static threshold
  • Adjust sensitivity with a standard deviations multiplier
  • Exclude known bad time windows from training
Advantage over static thresholds: A website getting 10,000 req/min at noon vs 500 req/min at 3 AM is normal. Anomaly detection adapts; a static threshold would either miss noon spikes or false-alarm at 3 AM.
B12
What is CloudWatch Synthetics? What is a "canary"?
▾

CloudWatch Synthetics lets you create canaries — scheduled scripts that simulate user interactions with your endpoints and APIs from outside your application, 24/7, even when there's no real user traffic.

The name "canary" refers to canaries used in coal mines to detect dangerous gases — an early warning system. Similarly, a CloudWatch canary detects issues before real users are impacted.

What canaries can test:

  • API endpoint availability and latency
  • Multi-step web workflows (login → add to cart → checkout)
  • Broken link detection
  • Visual UI regression (screenshot comparison)
  • Page load performance

B14
What is Contributor Insights and what problem does it solve?
▾

Contributor Insights continuously analyzes log data to identify the top contributors driving high load, errors, or traffic. It shifts the question from "what is the error rate?" to "which customer/IP/endpoint is causing the errors?"

Classic example: Your API is returning 5xx errors. A metric alarm tells you the rate is high. Contributor Insights tells you it's one specific tenant ID making 95% of those failing requests — dramatically accelerating root cause analysis.

Works with: Any CloudWatch log group, VPC Flow Logs, Route 53 query logs, ALB access logs, and more.

B15
What is CloudWatch Logs Insights and what query languages does it support?
▾

CloudWatch Logs Insights is an interactive log query engine. It allows you to run fast queries across multiple log groups simultaneously and return results in seconds, even for large datasets.

Query languages supported:

  • CloudWatch Logs Insights Query Language — native language with commands like filter, stats, parse, sort
  • SQL — familiar SQL syntax for those coming from database backgrounds
  • PPL (Piped Processing Language) — OpenSearch compatible, great for pipeline-style transformations
  • Natural Language (Preview) — Gen AI powered, query in plain English
Cost: You're charged per GB of log data scanned. Use time range filters and specific log groups to reduce costs.
Scenario-Based Questions
Real-world problem-solving questions. These test your ability to apply CloudWatch knowledge to architectural decisions, troubleshooting, and design patterns.
S01
Your EC2-based application is running out of memory, but you see no CloudWatch alarm. Why, and how do you fix it?
▾

Root cause: Memory utilization is not reported by CloudWatch by default for EC2 instances. The CloudWatch hypervisor can only see metrics from outside the VM (CPU, network I/O, disk I/O from EBS) — not OS-level metrics like RAM usage.

Fix:

  1. Install the CloudWatch Agent on the EC2 instance (or use AWS Systems Manager to deploy it fleet-wide)
  2. Configure the agent to collect mem_used_percent metric
  3. Agent publishes the metric to CloudWatch under the CWAgent namespace
  4. Create a CloudWatch Alarm on CWAgent/mem_used_percent with threshold (e.g., > 85%)
  5. Wire the alarm to SNS → email/PagerDuty notification
Also missing by default: Disk space (%), swap usage, per-process CPU. All require the CloudWatch Agent.
S02
You have 50 microservices each with their own alarms. Your on-call team gets flooded with notifications during an incident. How do you reduce alert noise?
▾

Solution: Composite Alarms

  1. Keep individual service alarms for granularity, but suppress their SNS actions
  2. Create composite alarms that group related child alarms with AND/OR logic
  3. Example: A "Payment Service Critical" composite alarm fires only when latency alarm AND error rate alarm are both in ALARM
  4. The composite alarm sends the single notification — not all 10 sub-alarms individually

Additional strategies:

  • Use "M out of N datapoints" alarm configuration — require 3 of 5 periods to breach before alerting (reduces transient spike alarms)
  • Use anomaly detection instead of static thresholds to reduce false positives
  • Configure alarm suppression on composite alarms during maintenance windows
Result: On-call team gets one alert per incident, not 50 — with root cause already narrowed down by which child alarms are active.
S03
Your application has 3x more users on weekday mornings vs nights/weekends. Standard threshold alarms keep firing false positives. What's your solution?
▾

Solution: CloudWatch Anomaly Detection Alarms

Static thresholds fail for cyclical traffic patterns. If you set the threshold for peak load, night-time spikes get missed. If you set it for average load, you get false positives during every morning ramp-up.

Steps:

  1. Enable Anomaly Detection on the relevant metric (request count, latency, error rate)
  2. Allow the model to train for 2 weeks on your historical data (it learns weekday/weekend patterns automatically)
  3. Create an "Anomaly Detection Alarm" instead of a static threshold alarm
  4. Set the deviation multiplier (e.g., 2 standard deviations from the expected band)
Result: Alarm fires when Wednesday 2 PM traffic is 5x above the expected-for-Wednesday-2PM baseline — not when it's simply "high" by an absolute number.
S04
A user reports your checkout page is broken, but your backend metrics all look healthy. How do you investigate using CloudWatch?
▾

Backend metrics being healthy suggests the problem is client-side. This is a classic RUM + Synthetics investigation scenario.

Investigation steps:

  1. Check CloudWatch RUM — look at JavaScript error rates, page load performance, and user session recordings. Filter by browser/device/geography to see if the issue is isolated.
  2. Check Synthetics canary — if you have a checkout flow canary, check if it's failing or reporting elevated latency. Look at screenshots to see where the flow broke visually.
  3. Check Application Signals — look at the service map for the checkout service. Check fault rate and latency at the service boundary level, and drill into X-Ray traces for the failing requests.
  4. Check Logs Insights — query for error patterns in the front-end service logs correlated with the time of the report.
Root cause might be: A third-party payment script timing out (visible in RUM's network calls), a specific browser version bug, or a CDN issue in one region.
S05
Your Lambda function costs spiked last month. How would you use CloudWatch to diagnose and reduce costs?
▾

Lambda costs = invocation count × billed duration × memory allocated. You need to identify which of these drivers increased.

Investigation using CloudWatch:

  1. Lambda Insights — enable it to get memory usage per invocation. If functions are using 200MB but you allocated 1GB, you're wasting 800MB of billed memory.
  2. CloudWatch Metrics — check Duration (average, p99), Invocations, ConcurrentExecutions. Identify if invocation count or duration spiked.
  3. Logs Insights query: filter @type = "REPORT" | stats avg(@billedDuration), max(@billedDuration), avg(@memorySize), avg(@maxMemoryUsed) by bin(1h) — shows billed vs actual memory over time.
  4. Check for cold starts — Lambda Insights shows init duration. Excessive provisioned concurrency or recent deployments can cause cost spikes.
Common fix found this way: Function memory was set to 3GB but only using 400MB. Right-sizing to 512MB cuts Lambda costs by ~80% for that function.
S06
You run a multi-tenant SaaS. One customer's excessive API calls are degrading performance for all tenants. How do you identify and act on this with CloudWatch?
▾

Solution: Contributor Insights

  1. Ensure your API logs are being sent to CloudWatch Logs with tenantId or customerId as a structured field
  2. Create a Contributor Insights rule on the API log group, with tenantId as the contributor field and requestCount as the aggregation
  3. CloudWatch builds a real-time leaderboard — "Top N tenants by request count"
  4. Set an alarm: if a single tenant exceeds X% of total traffic, trigger an alert

Automated response:

  • Alarm → SNS → Lambda → call API Gateway usage plan API to throttle that tenant's key
  • Or notify the account team to reach out to the customer
Also useful for: Identifying which IP is causing DDoS traffic (use VPC Flow Logs + Contributor Insights on srcAddr), then auto-updating WAF IP blocklist via Lambda.
S07
Your team manages 15 AWS accounts. How do you create a unified monitoring experience without logging into each account?
▾

Solution: CloudWatch Cross-Account Observability (Observability Access Manager — OAM)

  1. Designate one account as the monitoring account (typically a dedicated tooling or operations account)
  2. In each source account, create a link to the monitoring account's sink via OAM
  3. Share the telemetry types you need: Metrics, Logs, Traces, Application Signals
  4. From the monitoring account, you can now:
    • Run Logs Insights queries across all 15 accounts simultaneously
    • View metrics from all accounts on a single dashboard
    • See Application Signals service maps spanning accounts
    • Create cross-account alarms
No data movement: Data is not copied. The monitoring account has read-access to telemetry in-place in each source account. No ETL pipelines needed.
S08
How would you set up automatic EC2 instance recovery when the underlying hardware fails?
▾

Solution: CloudWatch Alarm + EC2 Recover Action

  1. Create a CloudWatch Alarm on the StatusCheckFailed_System metric for the EC2 instance
  2. Set the threshold: if this metric is ≥ 1 for 2 consecutive minutes
  3. Set the alarm action to EC2 Action: Recover

What happens on recovery:

  • AWS migrates the instance to a new healthy host
  • The instance retains the same: Instance ID, Private IP, Elastic IP, EBS volumes
  • Instance memory (RAM) contents are lost — it's a restart on new hardware
  • An SNS notification can be configured to alert the team
Note: Recovery only works for instances backed by EBS (not instance store). The StatusCheckFailed_Instance metric (OS-level failure) does NOT trigger auto-recovery — only System status checks can.
S09
You're building a new microservice and your SRE team wants to track an SLO of 99.9% availability with a 30-day rolling window. How do you implement this in CloudWatch?
▾

Solution: CloudWatch Service Level Objectives (SLOs)

  1. Enable Application Signals on your service (or define a custom metric for availability)
  2. In CloudWatch, create a new SLO:
    • SLI: "Percentage of HTTP requests returning 2xx" from Application Signals auto-instrumentation
    • Target: 99.9%
    • Window: Rolling 30 days
  3. CloudWatch automatically calculates the error budget — how much downtime is allowed (0.1% of 30 days = ~43 minutes/month)
  4. Set burn rate alarms: alert when error budget is consuming at 2x or 5x normal rate — warning before actual breach
  5. Dashboard shows SLO compliance history, burn rate trends, and remaining budget
SLO vs SLA: An SLO is your internal target. An SLA is a commitment to customers with penalties. CloudWatch SLOs help you stay ahead of SLA breaches.
S10
Your CloudWatch Logs costs are extremely high. How do you investigate and reduce them?
▾

CloudWatch Logs costs come from three areas: ingestion, storage, and queries. Here's how to attack each:

1. Find the biggest log groups (ingestion/storage cost):

  • Use Logs Insights: fields @logStream | stats sum(@ingestionSize) as bytes by @logGroup | sort bytes desc
  • Check AWS Cost Explorer filtered by CloudWatch Logs usage type

2. Reduce ingestion:

  • Reduce Lambda log verbosity (change log level to WARN in production)
  • Sample verbose DEBUG logs — only log 1 in 100 requests at debug level
  • Use EMF (Embedded Metrics Format) instead of logging entire JSON objects for metrics

3. Reduce storage costs:

  • Set retention policies on all log groups (e.g., 7 days for debug, 30 days for application logs, 90 days for audit)
  • Move old logs to S3 via export and delete from CloudWatch
  • Convert log groups to Logs-IA tier for archival/forensic logs

4. Reduce query costs:

  • Always specify time ranges in Logs Insights queries
  • Query specific log groups rather than account-wide
Common offender: Lambda functions logging full request/response bodies at DEBUG level in production. A high-traffic function can generate GBs of logs per day.
S11
How would you implement a "deploy and watch" strategy that automatically rolls back a bad deployment using CloudWatch?
▾

Using CloudWatch Alarms + CodeDeploy / Auto Scaling rollback:

  1. Define baseline metric alarms (error rate, latency, 5xx count) in CloudWatch
  2. In CodeDeploy: configure deployment group to watch specific CloudWatch alarms
  3. Set: "Roll back deployment if alarm enters ALARM state during deployment or bake time"
  4. Deploy with "linear" or "canary" strategy — send 10% traffic to new version first
  5. If error rate alarm fires within 10 minutes, CodeDeploy automatically rolls back

For ECS/Lambda (via CodeDeploy):

  • Blue/green deployments with CodeDeploy alarms integration
  • Lambda weighted aliases: send 5% traffic to new version, alarm monitors it for 10 min before shifting 100%
Key design principle: Alarms should be pre-created and stable (in OK state) before deployment begins. CodeDeploy watches for transitions to ALARM during the deployment window.
S12
You need to monitor an on-premises application alongside AWS services in the same CloudWatch dashboard. Is this possible? How?
▾

Yes, CloudWatch supports hybrid monitoring for on-premises infrastructure.

For metrics:

  • Install the CloudWatch Agent on on-premises servers
  • Authenticate the agent using IAM credentials (IAM user with CloudWatch write permissions, or Instance Profile via AWS Systems Manager Hybrid Activations)
  • Agent publishes metrics to CloudWatch with custom namespace and dimension (e.g., OnPremises/Server, HostName=db-server-01)
  • These metrics appear alongside AWS metrics in dashboards and can trigger alarms

For logs:

  • Same CloudWatch Agent handles log file tailing from on-premises servers and ships to CloudWatch Logs
Security best practice: Use Systems Manager Hybrid Activations to register on-premises servers. This uses temporary credentials via SSM rather than long-lived IAM access keys on the server.
