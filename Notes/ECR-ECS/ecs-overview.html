What ECS is
Overview
▶

Amazon ECS is a fully managed container orchestration service. It runs Docker containers on AWS without you needing to provision, patch, or manage servers or a control plane. AWS handles the underlying infrastructure — you focus on your application.

The service is built around three layers: capacity (where containers run), the scheduler (what manages them), and provisioning (how you interact with it). Every other feature plugs into one of these three layers.

2
Provisioning layer
How you talk to ECS
▶
AWS Management Console

A web-based point-and-click interface. Good for exploring ECS, creating clusters, viewing running tasks, inspecting logs, and debugging — no code required.

AWS CLI

Command-line tool for scripting and automation. Supports the full ECS API: register task definitions, create services, run tasks, scale, deploy. Works on Windows, macOS, and Linux.

AWS SDKs

Language-specific libraries (Python/boto3, JavaScript, Go, Java, .NET, Ruby, etc.) that wrap the ECS API and handle connection management, retries, and request signing automatically.

AWS CDK

An open-source framework that lets you define ECS infrastructure using familiar programming languages (TypeScript, Python, Java, C#). CDK compiles your code into CloudFormation templates and deploys them.

AWS CloudFormation

Infrastructure-as-code using YAML or JSON. Declare clusters, services, task definitions, and all supporting resources. CloudFormation handles creation order, rollbacks, and drift detection.

Amazon Copilot CLI

A developer-first CLI that abstracts ECS complexity. One command scaffolds a production-ready containerized service on ECS and Fargate — including CI/CD pipelines, environments, load balancers, monitoring, and auto scaling — all with sensible defaults baked in.

ECS Clusters
Logical grouping
▶

A cluster is the top-level namespace that contains all your ECS resources — services, tasks, and capacity providers all live inside one.

What a cluster contains

A cluster includes the VPC and subnets your tasks run in, a Service Connect namespace for internal service discovery, the capacity providers supplying compute, and all running services and tasks. A single cluster can mix Fargate, Managed Instances, EC2, and ECS Anywhere capacity simultaneously.

Capacity types
TypeWho manages infraBest for
ECS Managed InstancesAWSMost workloads — GPU, ARM, specialized instance types

Task definitions
The blueprint
▶

A task definition is an immutable JSON document describing how your containers run. Every update creates a new numbered revision — my-app:1, my-app:2, etc.

Key fields
FieldWhat it controls
containerDefinitionsImage URI, port mappings, env vars, secrets, health checks, log driver per container
cpu / memoryHard limits for the whole task (required on Fargate)
networkModeawsvpc (Fargate/recommended), bridge, host, none
taskRoleArnIAM role granting AWS API permissions to your running app
executionRoleArnIAM role ECS uses to pull the image and push logs
volumesMount EFS or EBS volumes into containers
requiresCompatibilitiesFARGATE, EC2, or EXTERNAL
Sidecar containers

A task definition can hold multiple containers. Typically one is the main app and others are sidecars — a log forwarder, Service Connect proxy, metrics agent, or Datadog sidecar. All containers in a task share the same host and lifecycle.

5
Services & tasks
The workhorse
▶
Tasks (one-shot)

A task is a single running instance of a task definition. Run tasks directly for batch jobs, migrations, one-off scripts, and scheduled work. When the process exits, the task stops. You pay only for the runtime.

Services (always-on)

A service keeps a specific number of tasks running at all times. If a task crashes or its host fails, ECS automatically starts a replacement. Right choice for web APIs, background workers, and any long-lived process.

Scheduling strategies
StrategyBehaviourUse case
ReplicaKeeps N tasks running, spreads across AZs by defaultWeb servers, APIs, workers
DaemonRuns exactly one task per container instance automaticallyLog agents, monitoring, security scanners
Task placement (EC2)

Control where tasks land using strategies (binpack, spread, random) and constraints (memberOf expressions — e.g., "GPU instances only" or "us-east-1a only").

6
AWS Fargate
Serverless compute
▶

With Fargate there are no EC2 instances to provision, patch, or scale. You define the task, specify CPU and memory, and Fargate runs it. AWS manages the VM, OS, and container runtime entirely.

How Fargate works

Each Fargate task runs in its own microVM with its own kernel — fully isolated from other customers. The task gets a dedicated ENI in your VPC. Billing is per-vCPU-second and per-GB-second of memory used.

Fargate Spot

Runs tasks on spare AWS capacity at up to 70% discount. Tasks receive a 2-minute interruption warning. Ideal for fault-tolerant batch jobs, CI runners, and stateless workloads.

Requirements

Fargate tasks must use awsvpc network mode. CPU and memory must be specified at the task level. Supported combos range from 0.25 vCPU / 0.5 GB up to 16 vCPU / 120 GB.

7
ECS Managed Instances
AWS-managed EC2
▶

ECS Managed Instances is the middle ground between fully serverless Fargate and fully self-managed EC2. AWS provisions, patches, and right-sizes the instances; you choose the instance families you need.

Automatic right-sizing

ECS continuously monitors instance utilization. It drains underutilized instances, moves tasks to more efficiently packed hosts, and terminates idle instances automatically. You only pay for active resources.

Specialized workloads

Unlike Fargate, Managed Instances gives access to GPU instances (ML inference), ARM/Graviton instances (cost efficiency), high-memory instances, and high-networking instances. Declare the capabilities you need; AWS picks the instance type.

8
EC2 self-managed
Full control
▶

With the EC2 launch type you manage your own container instances — the AMI, instance type, Auto Scaling Group, and patching schedule are all yours.

ECS container agent

Each EC2 instance in an ECS cluster runs the ECS container agent — an open-source Go binary pre-installed on ECS-optimized AMIs. It polls ECS for task placements, manages container lifecycle, monitors health, and reports metrics back to the control plane.

When to choose self-managed EC2

Use this when you need specific kernel parameters, custom AMIs with proprietary software, Windows containers, or compliance requirements demanding full OS-level control.

9
ECS Anywhere & AWS Outposts
Hybrid & on-premises
▶

ECS Anywhere extends ECS to run tasks on your own hardware — on-premises VMs, bare-metal servers, or instances in another cloud — using the same task definitions, CLI, and APIs as in AWS.

How ECS Anywhere works

Install the ECS agent and AWS SSM agent on your external server. ECS registers it as an EXTERNAL container instance. Use the EXTERNAL launch type to place work on those servers — all managed from the ECS console or API in your AWS region.

AWS Outposts

AWS-managed racks installed in your data center. ECS runs natively on Outposts, giving locally placed tasks with single-digit millisecond latency to on-premises resources, while using the same ECS control plane.

Use cases

Data residency requirements, factory floor / edge processing, compliance mandates requiring on-premises compute, and gradual cloud migration — some workloads local, some in AWS, all managed via ECS.

10
Capacity providers
The scaling brain
▶

Capacity providers connect ECS services to specific compute pools and define rules for distributing and scaling tasks across them.

Capacity provider strategy

Each service or task run specifies a strategy — a list of providers with a weight (relative distribution) and a base (minimum tasks always on this provider). Example: base of 1 on Fargate + weight of 4 on Fargate Spot means 4 spot tasks for every 1 on-demand task, with at least 1 on-demand always running.

Managed scaling (EC2)

For EC2 capacity providers, ECS can manage the Auto Scaling Group directly. It calculates the optimal instance count to meet task demand and adjusts accordingly — no manual CloudWatch alarms needed.

Managed termination protection

Prevents ECS from terminating EC2 instances still running tasks during scale-in. ECS drains the instance first, then removes protection so the ASG can safely terminate it.

11
Networking
VPC · Load balancing · Service Connect
▶
Task networking — awsvpc mode

In awsvpc mode, each task gets its own elastic network interface (ENI) with a private VPC IP. Tasks behave like EC2 instances — security groups apply directly to tasks, with full VPC routing, VPC endpoints, and flow log support.

Load balancing

ECS integrates natively with Elastic Load Balancing. As tasks start and stop, ECS automatically registers and deregisters them from target groups.

Load balancerBest for
Application LB (ALB)HTTP/HTTPS, path-based routing, WebSockets, gRPC, host-based routing
Network LB (NLB)TCP/UDP, ultra-low latency, static IPs, PrivateLink
Gateway LB (GLB)Third-party network appliances — firewalls, IDS/IPS
Service Connect

ECS's built-in service mesh. Injects a managed Envoy proxy sidecar automatically. Services register under short DNS names (e.g., orders.prod). The proxy handles retries, circuit breaking, and timeouts with no code changes. Per-request metrics (count, latency, errors) appear in the ECS console and CloudWatch.

Service Discovery (Cloud Map)

An older but supported approach using AWS Cloud Map to register task IPs/ports as DNS records. Service Connect is generally preferred for new workloads due to better observability and resilience.

Auto scaling
Scale tasks dynamically
▶

ECS Service Auto Scaling uses Application Auto Scaling to adjust the desired task count based on demand.

Scaling policy types
PolicyHow it worksBest for
Target trackingKeeps a metric (e.g., CPU at 60%) at a target — adds/removes tasks automaticallyMost web workloads
Step scalingScales by a configured amount when a CloudWatch alarm breaches a thresholdPrecise incremental control
Scheduled scalingIncreases/decreases task count on a cron schedulePredictable traffic patterns
Supported scale metrics

CPU utilization, memory utilization, ALB request count per target, SQS queue depth (for worker scaling), and any custom CloudWatch metric your application publishes.

Deployment strategies
Rolling · Blue/green · Canary
▶
Rolling update (default)

Gradually replaces old task versions with new ones in-place. Two parameters control the pace: minimumHealthyPercent (never drop below this % of desired count) and maximumPercent (max capacity allowed during deploy, e.g. 200 = double tasks briefly). Zero external dependencies.

Blue/green deployment (native)

ECS spins up a full separate set of tasks (green) alongside the existing set (blue). Traffic shifts from blue to green; blue stays available for instant rollback. Previously required CodeDeploy — ECS now has this built in natively with six Lambda lifecycle hooks.

Lifecycle hookWhen it fires
pre-scale-upBefore new tasks are created
post-scale-upAfter new tasks are healthy, before any traffic shifts
test traffic shiftWhen test traffic is sent to green
post-test traffic shiftAfter test traffic validation completes
production traffic shiftDuring production traffic shifting
post-production traffic shiftAfter all production traffic is on green
Traffic shifting modes
ModeHow it works
All-at-once100% of traffic shifts instantly to green
CanarySmall % shifts first, then the rest after a wait period
LinearTraffic shifts in equal increments at regular time intervals
14
Storage
EFS · EBS · ECR
▶
Amazon EFS — shared file system

A fully managed file system that multiple ECS tasks can mount simultaneously. Ideal for shared config, content management, ML model weights accessed across multiple tasks, and stateful apps needing a shared filesystem. Data persists beyond any individual task's lifetime.

Amazon EBS — block storage

High-performance block storage attached to one task at a time. Up to 16,000 IOPS and 1,000 MB/s throughput. Best for databases, high-performance data processing, and workloads requiring consistent low-latency I/O. EBS volumes persist independently and can be detached and reattached between tasks.

Ephemeral storage

Every Fargate task gets local ephemeral storage (20 GB default, expandable to 200 GB). Destroyed when the task stops. Use for temporary files, build artifacts, and scratch space.

Amazon ECR — container registry

AWS's managed container image registry. Stores Docker and OCI images privately in your account — no credentials needed from ECS (IAM handles auth). Features: image scanning for CVEs, lifecycle policies to delete old images, cross-region replication, and immutable image tags to prevent overwriting production images.

15
Monitoring & observability
CloudWatch · Insights · X-Ray · FireLens
▶
CloudWatch metrics

ECS automatically publishes CPU and memory utilization and reservation metrics to CloudWatch — aggregated at cluster, service, and task levels. Set alarms and use them to drive auto scaling.

CloudWatch Container Insights

A managed observability layer that automatically collects and presents granular performance data from your ECS clusters, services, and tasks — CPU, memory, network, disk I/O, and task counts. Pre-built dashboards in the CloudWatch console. Also captures diagnostic information like container crash failures.

AWS X-Ray distributed tracing

Traces requests as they travel through your containerized microservices. Add the X-Ray SDK or AWS Distro for OpenTelemetry to your containers. X-Ray generates a service map showing latency, error rates, and dependencies across all services.

Service Connect telemetry

When Service Connect is enabled, ECS automatically captures per-connection metrics (request count, latency percentiles, error count) for every service-to-service call — no code changes needed. Results appear in the ECS console and as CloudWatch metrics.

FireLens — log routing

FireLens runs a managed Fluent Bit or Fluentd sidecar in your task. Configure routing rules to send container logs to CloudWatch Logs, S3, OpenSearch, Splunk, Datadog, or any HTTP endpoint — without modifying application code. ECS manages the sidecar lifecycle automatically.

16
Security
IAM · GuardDuty · network isolation
▶
IAM task roles

Each task carries its own IAM role with exactly the permissions it needs. Credentials are provisioned via the ECS task metadata endpoint and rotate automatically — no static keys in environment variables. Zero-trust at the task level.

Task execution role

A separate IAM role used by the ECS agent (not your app) to pull the container image from ECR and push logs to CloudWatch. Separating execution permissions from task permissions follows the principle of least privilege.

Secrets management

Reference secrets from AWS Secrets Manager or SSM Parameter Store directly in your task definition. ECS retrieves and injects them at task launch — as environment variables or files. Your app never calls the AWS secrets API, and values never appear in the ECS console.

Amazon GuardDuty for ECS

GuardDuty Runtime Monitoring uses eBPF sensors to analyze container runtime events. Detects threats like cryptomining, privilege escalation, unusual outbound calls, and command injection — at the individual container level — and reports findings to Security Hub.

Network isolation

Security groups apply directly to tasks in awsvpc mode — fine-grained inbound/outbound rules per task. Combine with private subnets and VPC endpoints to ensure container traffic never traverses the public internet.

Encrypted storage

EFS and EBS volumes support at-rest encryption via AWS KMS. Fargate ephemeral storage is encrypted at rest by default (AES-256). EFS data in transit is encrypted via TLS.

17
ECS Express Mode
Newest — one-command deploys
▶

Express Mode is ECS's newest high-level abstraction for teams who want to deploy containers to production without deep AWS knowledge.

What you provide

Three things: a container image, CPU/memory, and a port. Everything else is handled automatically.

What it creates automatically

A Fargate-based ECS service, an Application Load Balancer with SSL/TLS termination, a unique publicly accessible URL, auto scaling policies, VPC networking, CloudWatch monitoring, and all required IAM roles.

Cost

No charge for Express Mode itself. You pay only for the underlying resources it creates (Fargate tasks, ALB, etc.) — same pricing as manual configuration.

Best for

Web apps, REST/GraphQL APIs, internal tools, rapid prototyping, and enabling product teams to deploy independently without a dedicated platform engineering team.
