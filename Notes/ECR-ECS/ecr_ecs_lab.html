Auth & Push / Pull
1
You run docker push and get "no basic auth credentials". What's wrong?
▶
FixAuth
Docker is not authenticated to ECR. Run:
aws ecr get-login-password --region <region> | docker login --username AWS --password-stdin <account>.dkr.ecr.<region>.amazonaws.com

The token is valid for 12 hours. Username is always AWS. Also confirm your AWS CLI credentials/profile are correct.
2
ECS tasks fail to pull with "CannotPullContainerError: authorization failure". CI/CD push works fine. Why?
▶
IAMFix
The Task Execution Role is missing ECR permissions. Attach AmazonECSTaskExecutionRolePolicy which includes ecr:GetAuthorizationToken, ecr:BatchCheckLayerAvailability, and ecr:GetDownloadUrlForLayer. Also verify the image URI in the task definition is exact.
3
Push fails with "repository does not exist" even though it's visible in the console. Causes?
▶
Config
1. Wrong region — repo is in us-east-1 but you're pushing to ap-south-1.
2. Incomplete URI — the full format is <account>.dkr.ecr.<region>.amazonaws.com/<repo>:<tag>.
3. You forgot to docker tag your local image with the full ECR URI before pushing.
4
A third-party service needs to pull from your private ECR repo but can't use IAM roles. How do you grant access?
▶
IAMRepo Policy
Use an ECR repository resource policy (like an S3 bucket policy). Go to repo → Permissions → Edit JSON. Grant the external principal ecr:BatchGetImage and ecr:GetDownloadUrlForLayer.

Important: ecr:GetAuthorizationToken is an account-level action — it cannot go in the repo policy. It must be in an IAM identity policy.
Lifecycle & Image Management
5
Storage costs are climbing. You want to automatically keep only the last 5 prod-* tagged images. How?
▶
Lifecycle Policy
Create a Lifecycle Policy on the repo. Rule priority 1: tagPrefixList: ["prod-"], countType: imageCountMoreThan, countNumber: 5, action: expire. Add a second lower-priority rule for untagged images older than N days.

Always use dry run first — it shows which images would be deleted without actually deleting them.
6
You deleted an image tag but the image still shows in the repo. Why?
▶
Concept
An image can have multiple tags. Deleting one tag removes only that pointer — the image layers remain until all tags referencing that digest are deleted. Use the image digest directly to delete the manifest and all its tags at once.
Networking & VPC
7
ECS tasks in a private subnet with no NAT Gateway fail to pull ECR images. Fix without NAT.
▶
NetworkingFix
Create VPC Interface Endpoints (PrivateLink) for:
· com.amazonaws.<region>.ecr.api
· com.amazonaws.<region>.ecr.dkr
· com.amazonaws.<region>.s3 — commonly missed! ECR stores image layers in S3.
· com.amazonaws.<region>.logs — if using CloudWatch.

Endpoint security groups must allow TCP 443 inbound from the task's security group.
8
VPC endpoints are set up but image pulls still timeout. What did you likely miss?
▶
Networking
The S3 Gateway endpoint is missing or not attached to the subnet's route table. ECR layers are downloaded from S3 — without this endpoint, layer downloads fail even if the ECR endpoints are present. Also verify enableDnsSupport = true on the VPC.
Scanning & Security
9
You enabled image scanning but no results appear 10 minutes after a push. Why?
▶
Scanning
Two scan types exist:
· Basic (Clair) — triggered on push or manually. If set to Manual, you must trigger it explicitly.
· Enhanced (Inspector) — continuous scanning, near-real-time.

Check the repo-level scan setting — it can differ from the registry default. Basic scanning results appear under the image's "Vulnerabilities" tab.
10
How do you automatically block images with CRITICAL vulnerabilities from being deployed?
▶
Security
Use EventBridge to capture ECR image scan finding events. Trigger a Lambda that checks if severity = CRITICAL. If yes: send an SNS alert and block the pipeline (e.g., reject a CodePipeline approval or delete the image with ecr:BatchDeleteImage). For Enhanced scanning, use Amazon Inspector finding events with the same pattern.
11
You want ECS tasks to only use images from your own ECR — not Docker Hub. How do you enforce this?
▶
IAMSCP
Use an SCP or IAM permission boundary to deny ecs:RegisterTaskDefinition when the image field doesn't match your account's ECR pattern: <account-id>.dkr.ecr.*.amazonaws.com/*. Use string condition keys on the request to enforce this at the org/account level.
Task Failures
1
Your ECS task stops immediately with exit code 1. How do you debug it?
▶
Debug
1. ECS Console → Cluster → Task → Stopped reason — read the error.
2. Check CloudWatch Logs for container stdout/stderr (requires awslogs log driver and execution role with logs permission).
3. Common causes: app crash at startup, missing env vars, bad entrypoint, wrong CMD.
2
An ECS service keeps launching and killing tasks in a loop. Task definition looks fine. What do you check?
▶
Health CheckFix
Check the ALB health check. ECS replaces tasks that fail it. Verify:
· Health check path, port, and expected status code.
· Container HTTP server is binding on the right port.
· Set a health check grace period on the service to give slow-starting apps time to boot before ECS starts checking.
3
Task exits with code 137. What happened and how do you fix it?
▶
Fix
Exit 137 = OOM Kill (128 + SIGKILL). The container exceeded its memory limit. Fix: increase memory (hard limit) or memoryReservation (soft limit) in the task definition. For Fargate, increase the task-level CPU/memory pair — they're linked and must be a valid Fargate combination.
Networking & Load Balancing
4
ALB returns 502 Bad Gateway. ECS tasks are running. What are the likely causes?
▶
NetworkingFix
1. Target group health — check EC2 → Target Groups → health status reason.
2. Security group — ALB SG → inbound 80/443; task SG → inbound from ALB SG on the container port.
3. Port mismatch — containerPort in task definition ≠ target group port.
4. App is listening on 127.0.0.1 instead of 0.0.0.0.
5
Two ECS services need to communicate privately. How do you set this up?
▶
NetworkingConfig
Use AWS Cloud Map (Service Discovery). Enable it on the ECS service — it registers tasks in a Route 53 private hosted zone as service.namespace.local.

Gotcha: When tasks replace, the IP changes. Keep DNS TTL low (≤ 10s) so clients pick up the new address quickly. Alternatively, put an internal NLB in front of the backend service.
6
EC2 launch type — tasks are PENDING but instances are available. Why won't they schedule?
▶
FixPlacement
1. Insufficient reserved resources — instance shows as available but has no free CPU/memory for the task size.
2. Port conflict — static host port mapping already in use. Switch to dynamic ports.
3. Placement constraint not met — task requires an attribute no instance has.
4. ECS agent not connected — instance is up but agent isn't registered. Check agent status in the ECS console.
IAM & Roles
7
Container app gets AccessDenied on S3. The EC2 instance has an IAM role. What's different with Fargate?
▶
IAMFix
With awsvpc / Fargate, the EC2 instance metadata endpoint is not available. The app cannot inherit the instance profile. You must set a Task Role (taskRoleArn) in the task definition. The SDK automatically fetches credentials from the ECS task metadata endpoint instead.
8
What's the difference between the Task Execution Role and the Task Role?
▶
IAMConcept
Task Execution Role — used by the ECS agent / control plane to pull images from ECR, write logs to CloudWatch, and fetch secrets from Secrets Manager before the container starts.

Task Role — used by the application code inside the container at runtime to call AWS APIs (S3, DynamoDB, SQS, etc.).

Both are set in the task definition. Easy to mix up — remember: execution = infrastructure setup, task = your app.
Scaling & Deployments
9
Auto Scaling is configured but tasks don't scale out during high CPU. What do you check?
▶
ScalingFix
1. Container Insights not enabled — without it, ECSServiceAverageCPUUtilization may not be reporting.
2. Max task count equals current task count — bump max tasks.
3. Cooldown period still active from a previous scale event.
4. EC2 launch type: even if service scales, there may be no EC2 capacity. Use Capacity Providers to link ASG scaling to ECS.
10
A deployment is stuck — service shows degraded and new tasks keep failing. How do you unblock it?
▶
FixDeployment
ECS waits for new tasks to pass health checks before draining old ones. If new tasks keep failing, the deployment stalls at minimumHealthyPercent.

Options:
· Fix the bug in the new task definition and redeploy.
· aws ecs update-service --force-new-deployment to restart.
· Enable Deployment Circuit Breaker — it automatically rolls back to the last stable version on repeated failures.
Secrets
11
Injecting a Secrets Manager secret into a container fails at task start with a permissions error. Fix it.
▶
IAMFix
Secret injection via the secrets block in task definitions is done by the Task Execution Role (not the task role). Add secretsmanager:GetSecretValue to the execution role policy.

If the secret uses a custom KMS key, also add kms:Decrypt for that key ARN. The secret ARN in the task definition must match exactly.
Core Concepts
1
What are the three levels of ECR? Registry, Repository, Image.
▶
Concept
Registry — one per AWS account per region. The namespace: <account>.dkr.ecr.<region>.amazonaws.com.
Repository — like a folder; holds multiple versions of one image. Lifecycle policies and permissions are set here.
Image — identified by an immutable digest (SHA256) and zero or more tags (mutable). Tags like latest can be overwritten; the digest never changes.
2
ECR Public vs ECR Private — what's the difference?
▶
Concept
Private — requires IAM auth for all access. Supports VPC endpoints, lifecycle policies, replication, scanning. Use for your own application images.
Public (gallery.ecr.aws) — anonymous pull allowed. Push still requires auth. No lifecycle policies. Use for open-source or shared base images.
3
What is image tag immutability and why enable it?
▶
Concept
With tag immutability on, once a tag is pushed it cannot be overwritten — any push with the same tag fails with an error. This prevents accidental overwrite of production tags like v1.2.0 and ensures deployments always pull exactly what was tested. Set per-repository. Digests are always immutable regardless.
4
How does ECR cross-region and cross-account replication work?
▶
Config
Configure replication rules at the registry level — specify destination regions (cross-region) and/or destination account IDs (cross-account). For cross-account, the destination registry must have a permissions policy allowing the source account to replicate. Replication is asynchronous. Only images pushed after setup are replicated — existing images are not back-filled.
5
What happens when you delete an ECR repository?
▶
All images inside are permanently deleted. Requires --force in CLI or explicit confirmation in console. There is no soft-delete or recovery. Plan lifecycle policies to clean up first.
6
Can you use ECR with EKS? What's needed?
▶
Yes. For EKS on EC2, attach AmazonEC2ContainerRegistryReadOnly to the node group IAM role. For EKS Fargate, use IRSA (IAM Roles for Service Accounts) with the same ECR permissions on the pod's service account. The ECR credential helper handles token refresh automatically.
Must-Remember Facts

Auth token

  • Valid for 12 hours
  • Username is always AWS
  • Use get-login-password
  • One token per registry

Image URI format

  • <account>.dkr.ecr.<region>.amazonaws.com
  • /<repo>:<tag> or @sha256:<digest>
  • Must docker tag before push

Lifecycle rules

  • Lower priority = evaluated first
  • countType: imageCountMoreThan
  • countType: sinceImagePushed
  • Always dry run first

Scanning types

  • Basic = Clair (OS packages)
  • Enhanced = Inspector (+ app libs)
  • Scan on push or manual
  • Results via EventBridge

VPC endpoints needed

  • ecr.api
  • ecr.dkr
  • s3 (layers — often missed!)
  • logs (CloudWatch)

Permissions model

  • IAM identity policies
  • Repo resource policies
  • GetAuthorizationToken = account-level only
  • Supports cross-account
Core Architecture
1
Explain the ECS hierarchy: Cluster → Service → Task → Container.
▶
Concept
Cluster — logical grouping of compute (EC2 instances or Fargate). One region.
Service — maintains desired task count, handles rolling deployments, integrates with load balancers.
Task Definition — the blueprint (JSON): images, CPU/memory, ports, env vars, roles, volumes.
Task — a running instance of a task definition. One or more containers together (like a pod).
Container — individual Docker container inside a task.
2
What network modes are available in ECS task definitions?
▶
NetworkingConcept
awsvpc — each task gets its own ENI and private IP. Required for Fargate. Supports per-task security groups. Best default choice.
bridge — Docker bridge on EC2. Supports dynamic port mapping. Multiple tasks per instance.
host — task shares the host network namespace. Best performance, no isolation, port conflicts possible.
none — no external networking. For offline batch workloads.
3
What is a Capacity Provider? How is it different from specifying a launch type?
▶
Concept
Capacity Providers link ECS services to compute resources. For EC2, a provider ties to an Auto Scaling Group and enables Cluster Auto Scaling — ECS triggers ASG scale-out when tasks can't be placed. Built-in providers for Fargate: FARGATE and FARGATE_SPOT (Spot saves cost).

Launch type = simple static choice. Capacity Providers = dynamic, managed scaling with weights across providers (e.g., 70% on-demand + 30% Spot).
4
ECS Service vs standalone Task (RunTask) — when do you use each?
▶
Concept
Service — maintains desired task count, auto-replaces failed tasks, rolling deployments, ALB integration. Use for long-running workloads: APIs, web apps, daemons.
RunTask — runs once and exits. No auto-restart. Use for batch jobs, migrations, one-off scripts. Trigger from EventBridge Scheduler, Lambda, or CLI.
5
What are ECS task placement strategies? (EC2 only)
▶
Config
spread — distributes tasks evenly across AZs or instances. Best for high availability.
binpack — packs tasks onto fewest instances to save cost.
random — random placement.

Combined with placement constraints: distinctInstance (one task per instance) or memberOf (custom attribute filter). Not applicable to Fargate.
6
How do you securely pass secrets (DB passwords, API keys) into ECS containers?
▶
IAMConfig
Use the secrets field in the container definition. Reference a Secrets Manager ARN or SSM Parameter Store ARN. ECS fetches and injects as environment variables at container startup — never in plaintext in the task definition.

The Task Execution Role needs secretsmanager:GetSecretValue or ssm:GetParameters (+ kms:Decrypt for custom KMS keys).
7
What is Container Insights and why does it matter for auto scaling?
▶
Observability
Container Insights is a CloudWatch feature that collects CPU, memory, network, and disk metrics at the task, service, and cluster level. Without it, ECS auto scaling based on CPU/memory metrics doesn't have the data it needs.

Enable at cluster creation or via aws ecs update-cluster-settings --name containerInsights --value enabled. Note: incurs additional CloudWatch costs.
Must-Remember Facts

Fargate limits

  • CPU: 0.25 to 16 vCPU
  • Memory tied to CPU choice
  • Network mode: awsvpc only
  • No privileged containers

Deployment config

  • minimumHealthyPercent (default 100)
  • maximumPercent (default 200)
  • Circuit breaker = auto rollback
  • Default = rolling update

Two IAM roles

  • Execution role = control plane
  • Task role = app at runtime
  • Both in task definition
  • Easy to confuse — remember both

Logging

  • awslogs driver → CloudWatch
  • Execution role needs logs perms
  • awslogs-group must exist
  • FireLens for advanced routing

Health checks — two kinds

  • Container health check (Docker)
  • ALB target group check
  • Both can cause task replacement
  • ALB check has grace period

Service discovery

  • Cloud Map → Route 53 private DNS
  • Keep TTL low (≤ 10s)
  • Or use internal ALB/NLB
  • awsvpc required for per-task DNS
