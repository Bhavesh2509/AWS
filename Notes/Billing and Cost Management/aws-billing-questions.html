Q2 What is the AWS Free Tier? What are its three types? Basic▼
Answer
  • Always Free — Never expire, available to all AWS customers. Examples: Lambda 1M requests/month, DynamoDB 25 GB storage, CloudWatch 10 custom metrics, SNS 1M publishes/month.
  • 12 Months Free — Available for 12 months from account creation date. Examples: EC2 t2.micro 750 hours/month, S3 5 GB Standard storage, RDS db.t2.micro 750 hours/month, CloudFront 1 TB data transfer/month.
  • Trials — Short-term free access to specific services upon first use. Examples: Inspector 90-day free trial, Macie 30-day trial, Detective 30-day trial.
Each member account in an AWS Organization gets its own free tier — free tier limits are NOT shared across the organization.
Q3 What is the difference between unblended, blended, and amortized costs in AWS Cost Explorer? Basic▼
Answer
  • Unblended cost — The actual charge each account pays, as it appears on its invoice. On-Demand rates for regular usage, discounted rate when RI/SP covers usage. This is what you see by default.
  • Blended cost — An average rate calculated across all accounts in the organization for a given usage type. Used for internal chargeback where you want to charge all teams at the same rate regardless of which account's RI covers them.
  • Amortized cost — Spreads RI and Savings Plan upfront fees evenly across the commitment period. E.g., a $1,200/year All Upfront RI shows as $100/month instead of $1,200 in January. Best for true cost-per-period financial analysis.
  • Net amortized cost — Amortized cost after applying any credits, discounts, and EDP discounts.
Use amortized costs for budgeting and financial reporting. Use unblended for reconciling invoices. Use blended for cross-team chargeback at a flat rate.

Q6 What is the AWS Cost and Usage Report (CUR) and how is it typically used? Basic▼
Answer

The CUR is the most granular billing dataset AWS produces. It contains every line-item charge per resource per hour, including usage type, pricing, tags, RI/SP discount data, and account attribution.

Typical analytics stack: CUR delivered to S3 (in Parquet format) → AWS Glue Crawler (schema discovery) → Amazon Athena (SQL queries) → Amazon QuickSight or Grafana for dashboards.

Why CUR over Cost Explorer alone:

  • Cost Explorer has pre-built dimensions; CUR lets you query any combination including custom tag values.
  • CUR is the data source for third-party tools (Cloudability, Apptio, Datadog Cost).
  • CUR supports showback/chargeback at line-item level, including amortized RI/SP data.
Use Parquet format for CUR — it reduces file size by 80–90% and significantly speeds up Athena queries. Use hourly granularity only if you need sub-day analysis (daily is cheaper).
Q7 What is AWS Cost Anomaly Detection and how does it differ from a Budget alert? Basic▼
Answer

Cost Anomaly Detection uses machine learning to detect unexpected spending deviations from your historical baseline. It adapts to seasonal patterns and organic growth — reducing false positives. It identifies anomalies you didn't know to set a threshold for.

Budget alert fires when you cross a pre-defined threshold you set manually (e.g., 80% of a $5,000 monthly budget).

Run both. Anomaly Detection catches surprises; Budgets enforce planned limits. They complement each other perfectly.
Q8 What is the AWS Pricing Calculator and when should you use it vs Cost Explorer? Basic▼
Answer

Pricing Calculator — Pre-deployment estimation tool. Uses AWS list prices from the Pricing API. No actual usage data. No AWS account required. Use it to: estimate costs before building, compare architecture options, create budget proposals, present cost estimates to stakeholders.

Cost Explorer — Post-deployment analysis tool. Uses your actual billing data from CUR. Requires an AWS account with Cost Explorer enabled. Use it to: understand current/past spending, forecast future costs, identify optimization opportunities, investigate cost spikes.

Simple rule: planning a new workload → Pricing Calculator. Understanding existing spending → Cost Explorer.

Q10 What is the RI Utilization report vs the RI Coverage report? Basic▼
Answer

RI Utilization report — Shows what percentage of your purchased RIs are actually being used. If you bought 10 m5.large RIs but only 6 are running instances, utilization is 60%. Low utilization = you're wasting committed money on unused RIs.

RI Coverage report — Shows what percentage of your total eligible usage is covered by RIs (vs On-Demand). If you're running 20 m5.large instances but only 6 are covered by RIs, coverage is 30%. Low coverage = you could save more by buying more RIs.

Both reports together guide RI purchasing strategy: high utilization + low coverage = buy more RIs. Low utilization = you overbought and should consider selling on the RI Marketplace or modifying to better-utilized instance types.
Q11 When should you use Spot Instances vs On-Demand vs Reserved Instances? Basic▼
Answer
  • Spot Instances: When the workload is fault-tolerant and can handle a 2-minute interruption. Batch jobs, big data processing, ML training (with checkpointing), CI/CD, stateless web layers in ASGs. Up to 90% savings.
  • On-Demand: When workloads are unpredictable, short-lived, or cannot be interrupted. Spiky traffic, demos, development, any workload that can't use Spot or commitments yet.
  • Reserved Instances / Savings Plans: When workloads are predictable and will run continuously for 1+ years. Production databases, application servers, steady-state infrastructure. Up to 72% savings.
Optimal pattern: baseline load covered by Savings Plans/RIs → variable peaks handled by On-Demand → batch/flexible work runs on Spot. Use Spot with Auto Scaling Groups for automatic fallback to On-Demand if Spot unavailable.
④ Governance & Cost Allocation Basics
Q12 What are cost allocation tags and why must they be activated? Basic▼
Answer

Cost allocation tags are key-value metadata labels attached to AWS resources (e.g., Team=Payments, Environment=Prod). They allow you to filter and group costs in Cost Explorer, Budgets, and CUR by these dimensions.

Why activation is required: Tags on resources are NOT automatically included in billing reports. You must explicitly activate each tag key in the Billing Console under "Cost Allocation Tags". Activation takes up to 24 hours and does NOT apply retroactively to historical billing data — only from the date of activation onward.

Two types:

  • User-defined tags — Tags you create and apply to resources. Must be activated individually.
  • AWS-generated tags — Created by AWS (e.g., aws:createdBy). Activated separately.
Activate tags as early as possible — you lose historical data attribution if you wait. Use Tag Policies in AWS Organizations to enforce consistent tag standards across all accounts.
Q13 What are AWS Cost Categories and how do they differ from tags? Basic▼
Answer

Tags are metadata applied directly to individual AWS resources. They are the raw building blocks. They only work if resources are tagged — untagged resources don't appear in tag-based filters.

Cost Categories are rule-based groupings that map costs to categories using multiple criteria — not just tags, but also account IDs, service names, regions, and other cost category values. They work even for costs that can't be tagged (e.g., Support charges, data transfer fees).

Example: A "Payments Team" cost category might match: Tag:Team = payments OR account = 123456789012 OR (service = EC2 AND Tag:Project = checkout). This provides much more comprehensive coverage than tags alone.

Use tags as the foundation. Use Cost Categories to normalize and complete coverage — especially for shared costs, untaggable services, and multi-account attribution.
Q14 How does Consolidated Billing in AWS Organizations help with cost savings? Basic▼
Answer

Volume pricing aggregation: AWS applies tiered pricing based on usage volume. With consolidated billing, usage from all member accounts is combined to reach higher tiers faster, giving everyone the lower per-unit rate. For example, S3 pricing is cheaper above 50 TB — if Account A uses 40 TB and Account B uses 20 TB, combined they reach the cheaper tier.

Shared RI/Savings Plan benefits: With RI sharing enabled in Billing Preferences, unused Reserved Instance hours from one account automatically reduce On-Demand charges in another account in the same organization. This maximizes RI utilization across the org.

Single invoice: Administrative simplification — one bill, one payment method, one financial process for the entire organization.

Each member account still gets its own Free Tier limits — free tier is NOT aggregated or shared across the organization.
⑤ Scenarios — Cost Visibility & Analysis
SV1 A manager asks you "how much did the Payments team spend on AWS last month?" You have no tagging in place. What can you do and what is the long-term fix? Scenario▼
Answer

Immediate workaround (no tags):

  • If the Payments team runs in its own AWS account, use Cost Explorer filtered by that linked account — you can see costs per account without tags.
  • If they share an account, attempt to identify cost using service + region combinations you know the team uses. This is imprecise.
  • Check if there are any natural identifiers (prefix in resource names, specific S3 bucket names) that can be used to approximate costs in CUR queries.

Long-term fix (implement immediately):

  • Define a tagging strategy: minimum tags = Team, Environment, Project, CostCenter.
  • Activate the Team tag in Billing Console under Cost Allocation Tags. Do this immediately — no retroactive data.
  • Tag all existing resources with the correct team tag (use Resource Groups Tag Editor for bulk tagging).
  • Enforce tagging for new resources via AWS Config required-tags rule or SCP that requires tags on resource creation.
  • Create a Cost Category for "Payments Team" that covers: tagged resources + account ID + any other identifiers.
Without tags, meaningful cost attribution is very difficult. This scenario is the #1 reason organizations implement FinOps programs.
SV2 Your team wants to build a custom cost dashboard that shows costs by team, environment, and service over the past 6 months with daily granularity. Cost Explorer doesn't have the exact view you need. What do you build? Scenario▼
Answer

Solution: CUR + Athena + QuickSight

  • Configure a Cost and Usage Report (CUR 2.0 via Data Exports) with daily granularity, Parquet format, and resource IDs. Deliver to S3.
  • Set up AWS Glue Crawler to discover the CUR schema automatically and create a table in the Glue Data Catalog. AWS provides a CloudFormation template to automate this.
  • Query with Amazon Athena: write SQL to GROUP BY line_item_product_code (service), resource_tags_user_team, resource_tags_user_environment, line_item_usage_start_date.
  • Connect Amazon QuickSight to Athena as a data source. Build visualizations for the required dimensions. Schedule daily dataset refreshes.
  • Add filters for time range, team, environment to make the dashboard self-service for stakeholders.
This is the standard AWS cost analytics architecture. For maximum query speed, partition CUR data by month in Athena and use Parquet format. Cost Explorer API could also be used but has per-request charges and limited dimensions.
SV3 A developer asks "why does Cost Explorer show different numbers than the Bills page for the same month?" What explains this? Scenario▼
Answer

Cost Explorer data and Bills page data can legitimately differ due to several factors:

  • Cost grouping differences — Cost Explorer groups charges differently than the invoice (some charges appear under different service names).
  • Discounts, credits, refunds display — Bills page shows final invoice amounts including all adjustments. Cost Explorer may display costs before or after credits depending on your settings.
  • Timing for current month — Bills page shows estimated charges in real time. Cost Explorer updates at least once every 24 hours — it may lag slightly behind.
  • Rounding — Cost Explorer rounds values for display; invoice shows precise amounts.
  • Amortized vs unblended view — If Cost Explorer is set to "Amortized" view, RI upfront fees are spread across months and won't match the invoice (which shows full RI fee in the purchase month).
For invoice reconciliation, always use the Bills page and CUR unblended costs as the source of truth. Cost Explorer is for analysis and trend identification — not precise invoice matching.
⑥ Scenarios — Cost Spikes & Surprises
SC1 Your AWS bill this month is 3x higher than usual. Walk through how you investigate and identify the cause. Scenario▼
Answer

Step-by-step investigation:

  • 1. Cost Explorer — top-level: Go to Cost Explorer. Group by Service. Identify which service(s) show the increase. Sort by highest cost change month-over-month.
  • 2. Cost Explorer — drill down: Filter by the offending service. Group by Usage Type, Region, and Linked Account to narrow the source. Is it a specific account? A specific region? A new usage type?
  • 3. Check Cost Anomaly Detection: Did it fire an alert? It provides root cause analysis and identifies the specific service/account/tag driving the anomaly.
  • 4. Tag dimension: If tags are in use, group by Tag to see which team/project/environment is responsible.
  • 5. CloudWatch metrics: For the suspected service, check CloudWatch metrics for that time period — EC2 instance count, Lambda invocations, S3 request counts, NAT Gateway bytes processed.
  • 6. CUR query: If more granularity needed, query CUR in Athena: filter to the time period, group by resource ID to identify the specific resource generating costs.
  • 7. Common culprits: Accidentally left test resources running, data transfer spike (check NAT Gateway and cross-region transfer), runaway Lambda (infinite retry loop), S3 request costs (misconfigured lifecycle policies triggering excessive API calls), EC2 right-size missed.
Set up Cost Anomaly Detection BEFORE a spike happens — it would have notified you within hours instead of letting costs run for a month.
SC2 Your S3 data transfer costs suddenly doubled. You haven't changed the bucket or application. What do you investigate? Scenario▼
Answer
  • 1. Identify the traffic type: In Cost Explorer, filter by S3 and break down by Usage Type. Check if it's data transfer out (internet), cross-region transfer, or GET/PUT requests.
  • 2. S3 Server Access Logs or CloudTrail: Enable S3 server access logging (or check CloudTrail data events). Look for unexpected GetObject calls from unfamiliar IPs — could be a public bucket being scraped/hotlinked.
  • 3. Bucket ACLs/policies: Check if a bucket became public accidentally (S3 Block Public Access might have been disabled).
  • 4. CloudFront cache hit rate: If S3 is behind CloudFront and cache hit rate dropped (e.g., from 90% to 20%), many more requests are going to S3 origin. Check CloudFront cache policy changes.
  • 5. Application change: Even without intentional code changes, check if upstream traffic increased (marketing campaign, DDoS, bot scraping).
  • 6. S3 Intelligent-Tiering monitoring fees: If using S3 Intelligent-Tiering, there's a per-object monitoring fee for objects over 128 KB. A large number of small objects moved to Intelligent-Tiering could spike costs.
Prevent future surprises: restrict bucket access with bucket policies, use S3 Block Public Access, set up Cost Anomaly Detection for S3, and use CloudFront to absorb traffic before it hits S3.
SC3 A developer forgot to delete an EC2 instance they used for testing. It ran for 30 days. How could this have been prevented? Scenario▼
Answer

Prevention strategies, layered:

  • AWS Budgets with actions: Set a budget for the dev/test account. When it exceeds the threshold, automatically apply an IAM deny policy or SCP to prevent launching new instances. Stops the bleeding quickly.
  • Cost Anomaly Detection: Would have flagged the unexpected EC2 charges within hours of the instance running for longer than baseline expects.
  • Tag enforcement + TTL tagging: Require a DeleteAfter tag on all test resources. Use a Lambda function (scheduled EventBridge rule) to find and terminate resources where DeleteAfter date has passed.
  • AWS Instance Scheduler: Automatically stop EC2/RDS instances outside business hours (e.g., stop at 8 PM, start at 8 AM). Reduces dev costs by ~70%.
  • AWS Config rule: Flag EC2 instances without required tags as non-compliant — creates visibility pressure without hard enforcement.
  • Compute Optimizer + Cost Optimization Hub: Would flag idle EC2 instances with near-zero CPU utilization as "over-provisioned" or "idle".
  • SCP to restrict large instance types: In dev accounts, apply an SCP that blocks launching instances larger than m5.large — prevents accidentally launching prod-sized test instances.
The most impactful combo: Cost Anomaly Detection (detect it fast) + Budget with automated IAM action (stop it automatically) + TTL tagging (prevent it from happening again).
SC4 Your NAT Gateway data processing charges are unexpectedly high. How do you reduce them? Scenario▼
Answer

NAT Gateway charges $0.045/GB processed plus hourly costs. Common optimizations:

  • S3 Gateway Endpoint (free): If your traffic goes to S3, add a Gateway VPC Endpoint for S3 to the private subnet's route table. Traffic routes to S3 directly — never touches NAT Gateway. This is often the single biggest cost reduction.
  • Interface VPC Endpoints for AWS services: Traffic to services like SSM, Secrets Manager, CloudWatch, EC2 API, STS — add Interface Endpoints so it bypasses NAT Gateway. Endpoints cost ~$7/month each but save NAT charges for high-volume API traffic.
  • Check for cross-AZ traffic: If NAT Gateway is in AZ-a but private instances are in AZ-b, traffic crosses AZs ($0.01/GB each way) before hitting NAT. Deploy one NAT Gateway per AZ.
  • Identify top consumers: Query CUR or CloudWatch metrics on the NAT Gateway to see which instances generate the most traffic. May reveal a misconfigured application making excessive external calls.
  • Direct internet route for specific traffic: If instances need to access specific public endpoints (e.g., a public GitHub for package downloads), consider using an interface endpoint or HTTP proxy on EC2 instead of all traffic going through NAT.
The S3 Gateway Endpoint is free and often reduces NAT costs by 50–80% in data-heavy workloads. Add it to every private subnet route table as a default best practice.
⑦ Scenarios — Savings Plans & Cost Optimization
SS1 You're spending $50,000/month on EC2 On-Demand. How do you approach buying Savings Plans without over-committing? Scenario▼
Answer

Step 1 — Analyze usage patterns: In Cost Explorer, look at the last 3 months of EC2 usage. Identify the baseline (minimum consistent hourly spend) vs. the variable peaks. Your commitment should cover the baseline, not the peaks.

Step 2 — Use Cost Explorer recommendations: Go to Cost Explorer → Savings Plans → Recommendations. Set lookback period to 30 days. AWS recommends the optimal $/hour commitment with projected savings and payback period.

Step 3 — Conservative approach:

  • Start with covering 60–70% of your baseline with a 1-year Compute Savings Plan (most flexible). Don't commit to 3 years yet.
  • Review utilization quarterly. If consistently at 100%, add more coverage.
  • Reserve the remaining 30–40% as On-Demand for flexibility during growth, architecture changes, or unexpected traffic.

Step 4 — Monitor: Track Savings Plans utilization (aim for >95%) and coverage reports in Cost Explorer. Set a Budgets alert if utilization drops below 90% (wasted commitments).

Never commit more than your minimum baseline usage. Use a 1-year term for first purchase; move to 3-year once usage patterns are stable and well understood.
SS2 Your team bought a 3-year Reserved Instance for an m5.2xlarge. Six months later, you need to migrate to m6i instances. What are your options? Scenario▼
Answer

Depends on the RI type purchased:

If Standard RI:

  • You CANNOT change the instance family (m5 → m6i). Standard RIs are locked to the instance family.
  • Options: (1) Sell it on the RI Marketplace — list it for sale, buyers pay the remaining term. You get a partial refund. (2) Keep running m5 instances alongside your new m6i instances to use up the RI before switching fully.
  • You CAN modify scope (regional ↔ zonal), AZ (within region), and instance size within the same family (m5.large ↔ m5.2xlarge).

If Convertible RI:

  • You CAN convert to m6i instances. Submit a conversion request in the console. You can change instance family, size, OS, and tenancy.
  • New RI has the same expiration date. You may need to "top up" if converting to a more expensive instance type.
  • Cannot sell Convertible RIs on the Marketplace.

Lesson: If your architecture evolves frequently, buy Compute Savings Plans instead of RIs — Savings Plans automatically apply to any instance type/family without any conversion needed.

The newer recommendation is to prefer Compute Savings Plans over RIs for most EC2 workloads due to this exact flexibility issue.
SS3 Cost Optimizer shows your RDS db.r5.8xlarge instance has 5% average CPU utilization. What do you do? Scenario▼
Answer

1. Validate the data before acting:

  • Check if 5% average is accurate — are there peak periods with much higher CPU? Look at max CPU, not just average. Check memory, IOPS, and connections too. RDS can be memory or IOPS bound rather than CPU bound.
  • Check CloudWatch metrics for the last 2–4 weeks covering various usage patterns (month-end close, weekly batch jobs, etc.).

2. Right-size in stages:

  • Don't jump from r5.8xlarge to r5.large in one step. Move down one size (r5.4xlarge), test performance, then move further if metrics confirm.
  • For Aurora: can use Aurora Serverless v2 for variable workloads — it scales from 0.5 ACUs and only charges for actual usage.
  • Test in staging first with production-like load before applying to production.

3. If RDS RI was purchased: You can modify RDS RIs to a smaller instance within the same family (e.g., r5.8xlarge → r5.2xlarge + 2x r5.large to use the same normalized units).

RDS rightsizing has the highest potential savings per action. A db.r5.8xlarge costs ~$3,000/month; a db.r5.2xlarge is ~$750/month — 75% savings if the workload genuinely supports it.
⑧ Scenarios — Multi-Account & Organizations
SO1 Your organization has 20 AWS accounts. The CFO wants a single monthly cost report showing spend by team and environment across all accounts. How do you set this up? Scenario▼
Answer

Architecture:

  • Use AWS Organizations — all 20 accounts consolidated under a management (payer) account. The management account sees all costs.
  • Enable Cost Allocation Tags (Team, Environment) in the management account's Billing Console. Tags from all member accounts are included in the consolidated CUR.
  • Configure a CUR 2.0 (Data Export) in the management account — it includes all linked account costs with tag columns. Deliver to S3 in Parquet format.
  • Set up Athena + QuickSight: Glue Crawler on CUR S3 bucket → Athena queries GROUP BY resource_tags_user_team, resource_tags_user_environment, linked_account → QuickSight dashboard with monthly refresh.
  • Create Cost Categories in Cost Explorer for Team and Environment dimensions — use for interactive analysis in the console when CUR query is overkill.
  • Schedule Budget Reports for the CFO — weekly email with budget vs. actual per team. No console access needed.
The management account is the only place where consolidated, org-wide CUR can be configured. Ensure the management account's Billing preferences have "Linked Account Cost and Usage Data" visibility enabled for member accounts.
SO2 Your dev team's AWS account spent 10x its normal budget because someone launched 50 large GPU instances for ML training and forgot about them. How do you prevent this from happening again? Scenario▼
Answer

Layered prevention strategy:

  • SCP (Service Control Policy): Apply an SCP to the dev OU that denies launching instances larger than a defined size (e.g., deny ec2:RunInstances where ec2:InstanceType matches GPU families p3.*, p4.*, g4dn.* etc.). This is a hard guardrail — cannot be overridden without management account approval.
  • Budget with automated action: Set a monthly budget for the dev account (e.g., $5,000). Configure a Budget Action: when 80% is reached, apply a deny IAM policy that blocks new EC2 launches. When 100% is reached, stop all EC2 instances automatically.
  • Cost Anomaly Detection: Attach a monitor to the dev account. Configure an alert when any service exceeds $500 above expected in a single day. Would have caught this within hours.
  • Require approval tags: Use a Lambda + EventBridge rule that triggers on EC2 instance launch events. If the instance type is in a GPU family and no ApprovedBy tag is present, automatically stop the instance and notify the team.
  • AWS Instance Scheduler: Schedule all dev EC2 instances to stop at 8 PM and weekends. GPU instances left running overnight are automatically stopped.
SCPs are the hardest guardrail — they prevent the action entirely. Budgets with actions are the automatic responder. Anomaly Detection is the early warning. All three layers together create robust cost governance.
SO3 Your company is an AWS MSP. You resell AWS services to 30 customers at custom rates. How do you manage billing for each customer without exposing AWS list prices or your margins? Scenario▼
Answer

Solution: AWS Billing Conductor (ABC)

  • Create a billing group for each customer in AWS Billing Conductor. Add the customer's member accounts to the billing group.
  • Define custom pricing rules: apply percentage markup on each AWS service (e.g., EC2 +15%, S3 +10%). Rules apply to all accounts in the billing group.
  • Define shared discount rules: distribute RI/SP discounts within the billing group as agreed with the customer.
  • Member accounts see pro-forma Bills and pro-forma CUR in their own console that reflects the custom pricing — not AWS list prices and not your margin.
  • You (management account) see both the actual AWS charges and the pro-forma data for all billing groups.
  • Export pro-forma CUR per billing group to generate customer-specific invoices.
Billing Conductor is the correct native AWS solution for this. Without it, MSPs historically had to manually calculate and send separate invoices — a time-consuming monthly process prone to errors.
⑨ Scenarios — Tagging & Cost Allocation
ST1 You activated the "Team" cost allocation tag last month and can see team costs in Cost Explorer. But 30% of costs still show as "No tag." How do you address this? Scenario▼
Answer

The 30% untagged costs come from two sources: (1) resources that don't have the Team tag applied, and (2) AWS charges that cannot be tagged (data transfer, support, etc.).

Finding and fixing untagged resources:

  • Use Resource Groups Tag Editor to search for resources missing the Team tag across all regions and services. Bulk tag from the results.
  • Use AWS Config rule required-tags to continuously flag non-compliant resources. Create an SNS notification when new untagged resources are detected.
  • Query CUR in Athena: WHERE resource_tags_user_team IS NULL GROUP BY line_item_product_code — shows which services have the most untagged costs.

Handling inherently untaggable costs:

  • Data transfer, AWS Support, Marketplace charges, and some managed services cannot be tagged at the resource level.
  • Use Cost Categories with split charge rules to allocate these shared costs proportionally based on each team's tagged usage.

Prevent future untagged resources:

  • Enforce Tag Policies in AWS Organizations — define allowed tag keys and values.
  • Use an SCP that requires the Team tag to be present when creating certain resources.
Getting below 5% unallocated costs is the industry FinOps benchmark. 100% allocation is practically impossible due to service-level charges that cannot be resource-tagged.
ST2 Two teams share a Transit Gateway and a NAT Gateway. Both teams want to see their "fair share" of these shared infrastructure costs in their cost reports. How do you implement this? Scenario▼
Answer

Shared infrastructure costs like TGW and NAT Gateway cannot be tagged per-team (they serve both teams). Use AWS Cost Categories with split charge rules:

  • Create a Cost Category called "Team".
  • Add a split charge rule for the "Shared Infrastructure" category (which matches TGW and NAT Gateway costs).
  • Configure the split: proportional allocation based on each team's usage of the shared resource. Cost Categories supports three split methods: proportional (based on each team's costs), fixed percentage (you define the split), or equal split.
  • Each team's Cost Category value in Cost Explorer now includes their attributed share of the shared infrastructure costs.

For more granular NAT Gateway per-team attribution, check if teams use separate subnets — each subnet's traffic through NAT can be analyzed from VPC Flow Logs + CUR to estimate per-team bytes.

Split charge rules are the recommended native AWS solution for shared cost allocation. They appear in Cost Explorer and CUR — enabling both self-service analysis and automated reporting.
⑩ Scenarios — FinOps Architecture & Design
SD1 You are starting a FinOps practice from scratch at a company with 10 AWS accounts, no tagging, and no cost visibility. What is your 90-day plan? Hard▼
Answer

Days 1–30: Visibility

  • Enable Cost Explorer in the management account. Review current spending by service, account, and region.
  • Enable Cost Anomaly Detection (free) with SNS alerts to engineering and finance.
  • Set up CUR 2.0 data export (Parquet) to S3 + Athena for raw data access.
  • Define tagging strategy: minimum tags — Team, Environment, Project, CostCenter.
  • Activate cost allocation tags in Billing Console immediately (no retroactive data).
  • Use Resource Groups Tag Editor to begin tagging existing resources.
  • Set simple monthly Budget alerts per account (email only — no actions yet).

Days 31–60: Control

  • Deploy AWS Config required-tags rule to flag untagged resources.
  • Create Cost Categories in Cost Explorer for Team and Environment.
  • Set Budget Actions on dev/test accounts: deny new EC2 launches at 100% budget.
  • Run Cost Optimization Hub — generate rightsizing recommendations. Prioritize top 5 by savings potential.
  • Identify and delete or stop idle resources (unattached EBS volumes, unused Elastic IPs, underutilized EC2 instances).
  • Add S3 Gateway Endpoints to all private subnet route tables (free, immediate NAT savings).

Days 61–90: Optimize

  • Purchase initial Compute Savings Plans (1-year, conservative — cover 60% of baseline).
  • Build a QuickSight cost dashboard from CUR data for leadership reporting.
  • Implement Instance Scheduler for dev/test environments (stop outside business hours).
  • Establish monthly FinOps review cadence: compare actuals vs. forecast, review new recommendations, track optimization progress.
  • Set Tag Policies in Organizations to enforce tagging for new resources.
Most organizations find 20–40% savings in the first 90 days through idle resource cleanup, rightsizing, and initial Savings Plans purchases. The biggest long-term value comes from governance and culture change.
SD2 A product team argues that their AWS costs are high because of shared infrastructure (networking, monitoring, security). They want a "true" cost of running their product. How do you solve this? Hard▼
Answer

This is a classic showback/chargeback problem. The "true cost" of a product includes both direct costs (their EC2, RDS, S3) and an allocated share of shared infrastructure.

Approach:

  • Direct costs: Tag all product-specific resources with the product name. Activate the tag as a cost allocation tag. These costs are directly attributed in Cost Explorer and CUR.
  • Shared infrastructure allocation: Create a Cost Category called "Product". Add a split charge rule for the "Shared Infrastructure" category (TGW, Network Firewall, logging, monitoring, security services). Choose proportional split based on each product's direct costs — products that consume more AWS resources get a proportionally larger share of shared infrastructure.
  • Fixed overhead allocation: AWS Support plan costs, management account charges — split equally across products, or by headcount (fixed percentage).
  • Unit cost calculation: Once total attributed cost per product is known, divide by the product's metric (e.g., number of active users, number of transactions) to calculate cost per unit. This is the "true" cost the team cares about.
This is the core of FinOps showback. The goal is to make product teams cost-aware and accountable — not necessarily to charge them internally, but to drive cost-conscious architecture decisions.
SD3 Your company spends $2M/month on AWS. Leadership wants to cut costs by 25% without impacting production reliability. What is your optimization roadmap? Hard▼
Answer

$500K/month savings target — typical optimization levers:

1. Commitment-based discounts (highest ROI, lowest risk)

  • Analyze SP/RI coverage. If only 40% covered, purchasing Compute Savings Plans to reach 70–80% coverage can save $100K–$200K/month immediately on EC2/Fargate/Lambda. Zero reliability risk — same instances, just cheaper.

2. Rightsizing (medium ROI, low risk with testing)

  • Run Cost Optimization Hub + Compute Optimizer recommendations. Prioritize by highest savings. Stage rightsizing: test in dev → staging → production. EC2 and RDS rightsizing often yields 15–30% compute cost reduction.

3. Idle resource cleanup (quick wins)

  • Delete unattached EBS volumes, unused Elastic IPs, idle load balancers, stopped instances with attached volumes. Usually $5K–$20K/month in wasted spend.

4. Architecture optimizations (medium term, high ROI)

  • S3 Intelligent-Tiering or lifecycle policies for infrequently accessed data. EBS gp2 → gp3 migration (20% cheaper, same or better performance). Spot Instances for batch/dev workloads. Instance Scheduler for non-production environments.

5. Data transfer optimization

  • Audit cross-AZ traffic. Add S3 Gateway Endpoints. Review CloudFront cache hit rates. Consolidate cross-region data movement.

Tracking: Set a baseline spend metric per month. Report savings monthly against baseline. Use Cost Explorer Cost Comparison report to validate actual savings.

Commitment purchases (Savings Plans) are usually the fastest path to large savings with zero reliability risk. Rightsizing carries some risk but is validated in staging first. Architecture changes require the most effort but deliver sustainable long-term savings.
SD4 What is the difference between a Showback and a Chargeback model, and when would you use each? Hard▼
Answer

Showback — Allocate and display costs to teams/products for visibility and accountability, but don't actually transfer money between departments. The central IT/cloud team absorbs the cost. Teams see what they "would have paid."

Chargeback — Allocate costs and actually bill each team/department/business unit. Funds are transferred via internal financial mechanisms (GL journal entries, P&L adjustments). Teams are financially accountable for their consumption.

ShowbackChargeback
Financial impactInformational onlyReal budget/P&L impact
ComplexityLower — just reportingHigher — requires finance integration
Behavioral changeModerate — teams aware but not accountableStrong — teams feel the cost directly
Common inEarly FinOps maturity, cost-aware culture buildingMature FinOps, product P&Ls, MSP billing

When to use each:

  • Start with showback to build cost awareness culture without organizational resistance.
  • Move to chargeback when: product teams have their own P&Ls, leadership requires financial accountability, or you need to accurately calculate product COGS (cost of goods sold).
Most organizations start with showback and evolve toward chargeback as FinOps maturity increases. AWS tools (Cost Categories, CUR, Billing Conductor) support both models.
