Core primitives

Identities

The four principal types in IAM. Each authenticates differently and fits a different use case. Roles with temporary credentials are almost always the right choice over IAM users with long-term keys.

Principal types

🔴 Root User Avoid daily use
▶
What it is
The original account identity created with email + password when the AWS account is opened. Cannot be deleted or restricted by any IAM policy.
Permissions
Complete, unrestricted access to all AWS services and billing. No policy can limit it.
MFA
Enable hardware MFA immediately. This is the most important security action in any AWS account.
When to use
Only for tasks that explicitly require root: closing the account, changing the support plan, restoring a locked-out IAM admin, first-time MFA setup.
2025–26 update
New managed policies IAMAuditRootUserCredentials and IAMDeleteRootUserCredentials enable centralized root credential auditing via AWS Organizations.
👤 IAM Users Long-term credentials
▶
What it is
A persistent identity with a unique ARN: arn:aws:iam::ACCOUNT:user/NAME. Tied to a specific person or application.
Credential types
  • Console password (username + password)
  • Access keys (Access Key ID + Secret Access Key) for CLI/API
  • SSH keys for CodeCommit
  • HTTPS credentials for CodeCommit
  • X.509 certificates for legacy SOAP APIs
Limits
5,000 IAM users per account. 2 access key pairs per user maximum.
MFA enforcement
Add aws:MultiFactorAuthPresent condition to identity policies to require MFA for sensitive operations.
When to use
Only when federation or Identity Center is not possible: legacy third-party tools, break-glass emergency accounts, CodeCommit.
Best practice
Rotate access keys regularly. Use AWS Config rule access-keys-rotated. Delete unused keys immediately.
👥 IAM Groups Bulk permission management
▶
What it is
A named collection of IAM users. Not a true principal — cannot be specified as a Principal in resource-based policies.
How it works
Attach policies to the group. All members immediately inherit those permissions. Remove a user and they lose access instantly.
Limits
300 groups per account. A user can belong to up to 10 groups. Groups cannot be nested (no group-in-group).
AWS job functions
Pre-built managed policies: AdministratorAccess, PowerUserAccess, DatabaseAdministrator, NetworkAdministrator, SecurityAudit — good defaults for group policies.
🎭 IAM Roles Preferred approach
▶
What it is
An assumable identity. Not tied to a specific person — any trusted entity can assume it. Issues temporary credentials automatically.
Two required policies
  • Trust policy — defines who/what can assume this role (the Principal)
  • Permissions policy — defines what the role can do once assumed
Credentials issued
Temporary: Access Key ID + Secret Access Key + Session Token via STS. Auto-expire (default 1hr, configurable up to 12hr).
Role types
  • Service roles — assumed by EC2, Lambda, ECS, etc.
  • Cross-account roles — Account A accesses Account B
  • Federated roles — assumed by SAML/OIDC users
  • Service-linked roles — AWS-managed, predefined per service
  • Instance profiles — wrapper that holds a role for EC2
Role chaining
Using one role to assume another. Hard 1-hour cap on chained sessions regardless of individual role duration settings.
Session tags
Pass key-value tags at assume-time (e.g. team=payments). Mark tags transitive to carry through role chains. Use in ABAC policies.
Authorization engine

Policies & Permissions

JSON documents that define what actions are allowed or denied. AWS evaluates all applicable policies together. Explicit Deny always wins. Requests are denied by default unless explicitly allowed.

Policy evaluation order

ℹ
Effective permissions = intersection of all allows minus any explicit deny. Each layer can only restrict — never expand what the layer above permits.
1
Explicit Deny
Any policy with Effect:Deny → immediate block, no appeal
2
SCPs / RCPs (Organizations)
Account-level ceiling — restricts everything below
3
Permissions Boundary
Per-entity maximum — intersection with identity policy
4
Identity-based Policies
Policies attached to user/group/role
5
Resource-based Policies
Policies on S3, KMS, SQS, Lambda, etc.
6
Session Policies
Inline policies passed at AssumeRole time — can only narrow

The 7 policy types

Type Attached to Key notes
Identity-based
AWS managed · Customer managed · Inline
 Users, Groups, Roles Defines what the identity can do. AWS managed = maintained by AWS. Customer managed = yours to version/share. Inline = embedded 1:1, not reusable.
Resource-based
Inline only
 S3 buckets, KMS keys, SQS, Lambda, SNS, Secrets Manager… Requires Principal element. Enables cross-account access without role assumption. Inline only — no standalone managed version.
Permissions Boundary
Advanced feature
 IAM Users or Roles Sets the maximum ceiling. Effective permissions = intersection(identity policy, boundary). Used to safely delegate IAM management — developers can't grant more than the boundary allows.
Service Control Policies (SCPs)
Organizations
 AWS accounts / OUs in an Organization Does NOT grant permissions — only restricts. A service allowed by an SCP still needs an identity policy allow. Use to enforce guardrails: restrict regions, prevent CloudTrail deletion.
Resource Control Policies (RCPs)
New · 2024
 Resources across an Organization Complementary to SCPs. Restrict what resource-based policies can accept. Enforce data perimeters — only trusted identities from within the org can access sensitive resources.
Access Control Lists (ACLs)
Legacy
 S3 objects/buckets, VPC Different syntax. Limited permission sets (READ, WRITE, FULL_CONTROL). AWS recommends migrating to resource-based policies.
Session Policies
Inline JSON at assume-time
 Role sessions (AssumeRole, GetFederationToken) Passed inline when assuming a role. Can only restrict the role's permissions, never expand them. Perfect for scoped-down temporary credentials for a single task.

Policy JSON structure

policy.json JSON 
{
  "Version": "2012-10-17",         ← always this version string
  "Statement": [
    {
      "Sid": "AllowS3Read",          ← optional, human-readable ID
      "Effect": "Allow",            ← Allow | Deny
      "Principal": "*",           ← who (required in resource-based only)
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ],
      "Condition": {             ← optional, all keys must match
        "StringEquals": {
          "aws:RequestedRegion": "us-east-1"
        },
        "Bool": {
          "aws:MultiFactorAuthPresent": "true"
        }
      }
    }
  ]
}

ABAC — Attribute-Based Access Control

✦
Scale trick: Instead of one role per resource, tag both resources and principals with the same key/value pair. One policy rule dynamically matches them — scales to thousands of resources without touching the policy.
ABAC condition example JSON 
// Allow access only when principal tag "team" matches resource tag "team"
"Condition": {
  "StringEquals": {
    "aws:ResourceTag/team": "${aws:PrincipalTag/team}"
  }
}
AWS Security Token Service

STS — Temporary Credentials

The global service that issues short-lived temporary credentials. Every role assumption, every federated login, every cross-account call goes through STS. Global endpoint: sts.amazonaws.com. Free of charge.

API operations

AssumeRole Most common
▶
Returns
Access Key ID, Secret Access Key, Session Token, Expiration timestamp.
Duration
Min 900s (15 min) — Max 43,200s (12 hours) if the role is configured for it. Default: 3,600s (1 hour).
Role chaining cap
Hard 1-hour cap when assuming a role through another role session — overrides DurationSeconds.
Key parameters
  • RoleSessionName (required) — appears in CloudTrail, use meaningful values
  • ExternalId — prevents confused deputy attacks in cross-account scenarios
  • Policy — inline session policy to narrow permissions
  • Tags / TransitiveTagKeys — session tags for ABAC
ExternalId
A shared secret added to the trust policy of cross-account roles. A malicious third-party service cannot assume your role without knowing your ExternalId — prevents confused deputy.
AssumeRoleWithSAML Enterprise SSO
▶
Flow
User authenticates with IdP (Okta/ADFS) → IdP issues SAML assertion → browser POSTs to AWS sign-in endpoint → STS validates assertion → returns temporary credentials → console or CLI access.
Setup
Register IdP as a SAML provider in IAM. Create role with trust policy listing the SAML provider ARN as Principal.
Attribute mapping
IdP passes SAML attributes as session tags. Map groups, cost centers, department to AWS ABAC conditions without modifying IAM roles.
AssumeRoleWithWebIdentity GitHub Actions / OIDC
▶
Flow
App authenticates with OIDC provider (Google, GitHub, Auth0) → receives JWT → STS validates JWT against provider's public keys → issues temporary AWS credentials.
GitHub Actions
No stored secrets. GitHub issues a JWT per workflow run. IAM role trust policy conditions on repo + branch: token.actions.githubusercontent.com:sub. Per-run short-lived credentials.
Security tip
Always use StringEquals on both aud (audience) and sub (subject) claims. Never use StringLike with * on both simultaneously.
GetSessionToken MFA-protected API
▶
Purpose
Called by an IAM user with MFA code. Returns temporary credentials that satisfy aws:MultiFactorAuthPresent condition. Enables MFA-protected API/CLI access.
Duration
15 minutes to 36 hours for IAM users.
GetCallerIdentity Always allowed
▶
Returns
Account ID, User ID, ARN of the caller (user, role session, federated user, or root).
Special property
Cannot be denied by any policy, SCP, or boundary. Always allowed. Safe to call in any context for identity verification and debugging.
Use cases
CI/CD pipelines verifying which role is active, debugging unexpected "Access Denied" errors, confirming account context before deploying.

Formerly AWS SSO

IAM Identity Center

The recommended way to manage human workforce access across multiple AWS accounts and applications. Provides centralized SSO, permission management, and auditing. Completely free of charge.

Identity sources

⚠
One source per organization, ever. You can have exactly one identity source per AWS Organization. Changing it is destructive — all assignments are deleted. Plan before switching.
SourceBest forNotes
Built-in directory Small teams, startups Create users/groups directly in the console or via SCIM. No external IdP needed.
External SAML IdP Enterprises with Okta, Azure AD, Google Workspace, Ping Use SCIM to auto-sync users/groups. Users authenticate with their existing corporate credentials.
AWS Managed Microsoft AD Windows-heavy organizations Uses Directory Service (AD Connector or AWS Managed AD). Heavy Windows/AD environments.

Permission Sets

How permission sets work
▶
What it is
A named collection of IAM policies (AWS managed, customer managed, inline). Identity Center creates the actual IAM role in each target account automatically.
Assignment model
User/group + permission set → target account. Identity Center creates and maintains the IAM role. Update the permission set once — all 50 accounts update automatically.
Permissions boundary
Optionally attach an AWS managed or customer managed policy as a boundary to limit the maximum permissions of every role created from this set.
ABAC support
Pass user attributes (department, costCenter) as session tags when users assume the role via Identity Center. Use in ABAC conditions across all assigned accounts.
Trusted Identity Propagation New 2025–26
▶
Problem solved
When a user calls QuickSight which calls S3, the downstream call normally appears as the service role — losing the original user in audit logs. Trusted Identity Propagation carries the human identity through the entire call chain.
How it works
STS attaches an Identity Center identity context to the role session. Downstream services (S3, Redshift, QuickSight) read this context and apply user-specific access controls. All logs show the original human user.
Guard policy
AWSIAMIdentityCenterAllowListForIdentityContext is automatically attached. Only a curated set of actions is allowed with this context — all other actions are blocked for security.
External identity providers

Identity Federation

Allow external identities — from corporate directories, social providers, and CI/CD systems — to access AWS without creating IAM users. Federated users get temporary credentials via STS.

🏢
Enterprise
SAML 2.0 Federation
For enterprise IdPs: Okta, Azure AD, ADFS, Shibboleth. SAML assertions exchange for temporary AWS credentials via AssumeRoleWithSAML. Setup: register IdP → create SAML provider in IAM → create role with SAML provider as Principal.
⚡
Modern apps / CI/CD
OIDC Federation
OpenID Connect providers (Google, GitHub, Auth0) issue JWTs exchanged for AWS credentials via AssumeRoleWithWebIdentity. Standard for GitHub Actions — no stored secrets, per-run ephemeral credentials.
📱
Mobile / Web
Amazon Cognito
AWS-managed identity broker. User Pools handle auth (sign-up, sign-in, MFA, TOTP). Identity Pools exchange tokens from any supported provider for AWS credentials. Supports authenticated + unauthenticated (guest) users.
🔧
Custom systems
Custom Identity Broker
Build your own broker that authenticates against internal LDAP or Active Directory, then calls STS AssumeRole or GetFederationToken to get AWS credentials. Used when a third-party IdP is not available.
🌐
New Nov 2025
Outbound Identity Federation
AWS workloads authenticate with external third-party services using short-lived JWTs issued by IAM — eliminating the need to store third-party API keys inside AWS. Reverse of inbound federation.

ComponentRoleOutputs
User Pools Managed user directory. Handles sign-up, sign-in, MFA, password policy, email/phone verification. Can federate to external IdPs itself. OIDC-compatible JWTs: ID token, access token, refresh token.

Hybrid workloads

IAM Roles Anywhere

Extends temporary AWS credentials to workloads running entirely outside of AWS — on-premises servers, data centers, VMware. Uses X.509 certificates from a trusted PKI instead of long-term access keys. No secrets stored on disk.

Three components

🏛
Component 1
Trust Anchor
A registered CA (certificate authority) that Roles Anywhere trusts. Any certificate signed by this CA can be used to obtain credentials. Can be ACM Private CA or an external/self-managed CA.
🗂
Component 2
Profile
Links trust anchors to IAM roles. Defines which roles can be assumed, maximum session duration (1–12 hours), and optional session policies to scope down permissions.
💻
Component 3
aws_signing_helper
AWS-provided credential helper binary installed on the on-premises workload. Signs requests with the X.509 certificate and retrieves temporary credentials. Integrates with SDK credential provider chain.

Setup flow

Complete setup walkthrough
▶
Step 1
Create or use an existing PKI CA (ACM Private CA or your own).
Step 2
Register the CA as a Trust Anchor in IAM Roles Anywhere console or via API.
Step 3
Create an IAM role with trust policy allowing rolesanywhere.amazonaws.com as the principal service.
Step 4
Create a Profile linking the trust anchor to the role with desired session duration.
Step 5
Install aws_signing_helper on the on-premises server. Issue an X.509 certificate signed by your CA to the server.
Step 6
Configure ~/.aws/credentials or SDK to use the signing helper with the certificate + private key path as a credential process.
Security advantage
Certificate private keys never leave the host. Certificates can be revoked via CRL/OCSP instantly. No static secrets stored in config files.
Visibility & debugging

Tools & Auditing

IAM ships with a full suite of management, auditing, and troubleshooting tools. Most are free and available in every account by default.

IAM Policy Simulator Free
▶
What it does
Tests what actions a user, group, or role is allowed to perform without making real API calls. Evaluates all policy types together including boundaries and SCPs.
What you can test
Any user/group/role in your account, policies you're about to create (before attaching), specific resource ARNs, condition context (MFA presence, IP address, time of day, tags).
API
SimulatePrincipalPolicy and SimulateCustomPolicy — integrable into automated testing pipelines and CI/CD.
Limitation
IAM logic only — does not make real service calls. Some complex service-side access (VPC endpoint policies, KMS interactions) may not be fully captured.
Credential Report Compliance
▶
What it is
Downloadable CSV covering all IAM users in the account. Generated on demand, available within 4 hours of the previous generation.
Columns included
Username, ARN, creation date, password enabled/last changed/last used, MFA active, access key 1+2 (active, last rotated, last used), X.509 certificate status.
Use case
Security audits, SOC 2 / PCI-DSS / ISO 27001 evidence, identifying dormant accounts, enforcing key rotation SLAs.
Access Advisor Least privilege
▶
What it shows
Per-service last-accessed timestamps for any IAM entity (user, group, role). Shows which AWS services a policy allows and when they were last called by that entity.
Retention
400 days of history.
Use case
Identify services in policies that are never used and safely remove them. Works best when combined with Access Analyzer's Policy Generation for a full least-privilege workflow.
CloudTrail Integration Audit trail
▶
What's logged
Every IAM API call (CreateUser, AttachRolePolicy, PutGroupPolicy…) and every STS call (AssumeRole, GetCallerIdentity…). Includes source IP, user agent, request params, response.
Forensics fields
userIdentity.arn, eventName, requestParameters, errorCode, errorMessage, sourceIPAddress, userAgent, eventTime.
RoleSessionName tip
The RoleSessionName you set in AssumeRole appears in ALL subsequent CloudTrail events for that session. Use meaningful names: "deploy-pipeline", "user@company.com".
Recommended alerts
  • ConsoleLogin without MFA
  • Any root account usage
  • CreateUser or CreateAccessKey
  • AttachUserPolicy / PutRolePolicy with wildcard "*"
  • DeleteTrail or StopLogging
IAM Policy Autopilot New Nov 2025
▶
What it is
Open-source MCP (Model Context Protocol) server that analyzes application code and generates valid, minimal IAM policies using AI coding assistants. Reduces manual policy authoring for developers.
How to use
Integrate with compatible AI coding tools. Point it at your application code. It identifies all AWS SDK calls and generates the minimum IAM policy required.
Policy Variables Dynamic scoping
▶
What they are
Runtime context embedded in policies using ${} syntax. AWS substitutes the actual value at evaluation time.
Common variables
  • ${aws:username} — IAM username of the caller
  • ${aws:userid} — unique user ID
  • ${aws:PrincipalTag/key} — session tag value
Example use case
Grant every user access to their own S3 prefix with one policy: Resource: "arn:aws:s3:::user-data/${aws:username}/*" — no per-user policies needed.
2026 hardening guide

Best Practices

Current AWS and industry recommendations for a hardened, scalable IAM posture. Following these eliminates the most common IAM-related security incidents.

🔥
The single most common IAM mistake: storing long-term access keys in code, config files, or CI/CD secrets. Eliminate every static key — use roles everywhere.
1
Federation-first for all human access
Use IAM Identity Center with your corporate IdP for all developer, ops, and admin access. When an employee leaves, disabling their IdP account revokes all AWS access instantly — no IAM cleanup required across every account.
2
Zero long-term keys for workloads
EC2/Lambda → IAM roles with instance profiles. On-premises → Roles Anywhere with X.509 certificates. CI/CD pipelines → OIDC federation (GitHub Actions, GitLab). Delete every static access key from every config file and secrets manager that isn't absolutely necessary.
3
Secure root immediately and permanently
Enable hardware MFA on the root account. Delete root access keys (they should not exist). Set a CloudTrail + EventBridge alert for any root console login. Restrict root usage via Organizations. Consider IAMAuditRootUserCredentials policy.
4
Use SCPs as organizational guardrails
SCPs define the maximum permissions for entire AWS accounts. Use them to: restrict to specific regions, prevent disabling CloudTrail, block root access key creation, enforce tagging, prevent leaving the Organization. Remember: SCPs do not grant — they only restrict.
5
Build a data perimeter with SCPs + RCPs
SCPs restrict what identities can do. RCPs (Resource Control Policies) restrict what resources accept. Together: only trusted identities, from trusted accounts, accessing trusted resources. Add VPC endpoint policies for the third dimension: only from trusted networks.
6
Enforce least privilege continuously
Start with AWS managed policies. Use Access Analyzer Policy Generation + Access Advisor to right-size actual usage. Enable Unused Access Findings to catch permission drift. Review and prune quarterly. Least privilege is a process, not a one-time event.
7
Enable org-level Access Analyzer
Organization-level analyzer catches cross-account access across your entire estate in one view. Enable in the management or delegated admin account. Review all active (non-archived) findings monthly. Archive expected cross-account access with explanatory notes.
8
Use permissions boundaries for safe delegation
When team leads need to manage IAM for their own team, attach a permissions boundary to all roles and users they create. They cannot accidentally grant more access than the boundary allows — safe self-service IAM management at scale.
9
Use ABAC for large environments
Attribute-Based Access Control with resource and principal tags scales to thousands of resources without policy sprawl. Tag your resources (team, environment, project) and your principals. One ABAC policy replaces hundreds of resource-specific policies.
10
Audit everything with CloudTrail alerts
Every IAM change must be logged and alerted on. At minimum: EventBridge rules for ConsoleLogin without MFA, root account usage, CreateUser, policy attachment with wildcard "*", DeleteTrail. Ship all CloudTrail logs to an immutable S3 bucket in a separate security account.
