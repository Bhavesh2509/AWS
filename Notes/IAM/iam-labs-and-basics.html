Lab Module 01

Access Denied Errors

The most common IAM errors — each scenario gives you a broken setup and asks you to find the root cause and fix it.

01
IAM user with AdministratorAccess still gets AccessDenied on a specific S3 bucket
A user has the AdministratorAccess managed policy attached. They can access all other services but keep getting Access Denied when trying to delete objects in one particular S3 bucket.
AccessDenied Bucket Policy Explicit Deny Hard
▶
Error Seen
An error occurred (AccessDenied) when calling the DeleteObject operation: User: arn:aws:iam::123456789012:user/alice is not authorized to perform: s3:DeleteObject on resource: arn:aws:s3:::prod-archive/logs/2024.log with an explicit deny in a resource-based policy
Clue
Notice: The error says "with an explicit deny in a resource-based policy." This means the S3 bucket policy has a Deny statement. An explicit Deny in any policy — including resource-based policies — overrides every Allow, including AdministratorAccess. No amount of IAM Allow can override an explicit Deny.
Root Cause & Fix
  • 1
    Go to the S3 bucket → Permissions → Bucket Policy. Look for a statement with "Effect": "Deny" and s3:DeleteObject in the Action.
  • 2
    The bucket likely has a retention policy or compliance Deny that blocks deletions. Example:
    "Condition": {"StringNotEquals": {"aws:PrincipalArn": "arn:aws:iam::123456789012:role/archive-cleanup"}}
  • 3
    Fix Option A: Remove or modify the Deny statement in the bucket policy. Fix Option B: Add the user's ARN to the exception condition in the Deny.
  • 4
    Fix Option C: Assume the role that is permitted (archive-cleanup role) to perform the deletion instead.
Key rule: Explicit Deny always wins. AdministratorAccess is just an Allow — it cannot beat a Deny. This is the most important IAM evaluation rule to memorize.
02
Lambda function gets AccessDenied calling DynamoDB — it has an execution role
A Lambda function has an execution role attached with a policy that includes dynamodb:PutItem. When the function runs it gets AccessDenied on DynamoDB.
AccessDenied Lambda Execution Role Resource ARN Mismatch Medium
▶
Error Seen
AccessDeniedException: User: arn:aws:sts::123456789012:assumed-role/lambda-execution-role/my-function is not authorized to perform: dynamodb:PutItem on resource: arn:aws:dynamodb:us-east-1:123456789012:table/orders
Clue
Check: Look at the Resource field in the Lambda execution role's policy. It may say arn:aws:dynamodb:us-east-1:123456789012:table/Orders (capital O) while the actual table is orders (lowercase). DynamoDB table names are case-sensitive in ARNs. Also check the region — many people copy a policy from us-west-2 but deploy in us-east-1.
Root Cause & Fix
  • 1
    Open IAM → find the Lambda execution role → view the attached policy.
  • 2
    Compare the Resource ARN in the policy exactly to the actual DynamoDB table ARN (check: account ID, region, table name case).
  • 3
    Fix the ARN typo or use a wildcard if multiple tables are needed: arn:aws:dynamodb:us-east-1:123456789012:table/*
  • 4
    Also verify the Lambda function is actually using the role — check the function's Configuration → Permissions to confirm the role is attached.
Key rule: IAM resource ARNs are case-sensitive and region-specific. A policy granting access to table/Orders does NOT grant access to table/orders.
03
New IAM user created but cannot log in to the console
An admin creates a new IAM user and shares the credentials. The user tries to log in at the AWS console but gets "Invalid username or password" or a redirect loop.
Login Failure Console Access Easy
▶
Clue
Check: Creating an IAM user does NOT automatically give them console access. Console access requires explicitly enabling a console password. Also — the user must log in at the correct IAM sign-in URL, not the root account URL.
Root Cause & Fix
  • 1
    Go to IAM → Users → select the user → Security credentials tab.
  • 2
    Check "Console sign-in" — if it says "Disabled," click Enable console access and set a password.
  • 3
    Share the correct sign-in URL: https://[account-id].signin.aws.amazon.com/console — users cannot log in at the root sign-in page.
  • 4
    If the admin checked "User must create new password at next sign-in," the user must change the password on first login — they need the temporary password you set.
Key rule: IAM user creation ≠ console access. Console access is explicitly opt-in. Access keys (for CLI) and console passwords are separate credentials that must each be individually enabled.
04
EC2 instance can't call AWS APIs even though it has an instance profile attached
An EC2 instance has an IAM instance profile with S3 read permissions. Code running on it gets AccessDenied when calling s3:GetObject. The AWS CLI on the instance returns "Unable to locate credentials."
No Credentials Instance Profile IMDS Medium
▶
Error Seen
Unable to locate credentials. You can configure credentials by running "aws configure". -- OR -- An error occurred (AccessDenied): Not authorized to perform sts:AssumeRole
Clue
Check multiple things: (1) Is the instance profile attached to the instance? (2) Is IMDSv2 required but the code only uses IMDSv1? (3) Is the instance profile an actual profile or just a role ARN? (4) Was the instance profile attached after launch — did the app restart?
Root Cause & Fix
  • 1
    Verify attachment: EC2 console → Instance → Actions → Security → Modify IAM Role. Confirm the instance profile appears there.
  • 2
    Test IMDS manually: SSH into instance and run:
    curl http://169.254.169.254/latest/meta-data/iam/security-credentials/
    If empty — profile isn't attached. If 401 — IMDSv2 is required.
  • 3
    IMDSv2 fix: If IMDSv2 is enforced, the SDK/CLI must use a token. Update SDK to latest version, or run:
    TOKEN=$(curl -X PUT "http://169.254.169.254/latest/api/token" -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")
  • 4
    Note: Instance profile ≠ IAM role. The instance profile is a container. If you create a role and attach it directly without creating an instance profile, it won't work. The console auto-creates the profile when using the wizard.
Key rule: EC2 credentials come from the Instance Metadata Service (IMDS) at 169.254.169.254. IMDSv2 requires a session token. Always test with curl before blaming IAM policies.
05
IAM user can list objects in S3 bucket but can't download any file
A user successfully runs aws s3 ls s3://my-bucket/ but aws s3 cp s3://my-bucket/file.txt . returns AccessDenied.
AccessDenied s3:GetObject Action Mismatch Easy
▶
Clue
Listing vs downloading are different IAM actions. s3:ListBucket applies to the bucket ARN (arn:aws:s3:::my-bucket). s3:GetObject applies to objects (arn:aws:s3:::my-bucket/*). Having one does not grant the other.
Root Cause & Fix
  • 1
    The IAM policy only grants s3:ListBucket on the bucket ARN. Missing: s3:GetObject on the objects ARN (arn:aws:s3:::my-bucket/*).
  • 2
    Add a second statement: {"Effect":"Allow","Action":"s3:GetObject","Resource":"arn:aws:s3:::my-bucket/*"}
  • 3
    Remember: the /* suffix is required for object-level actions. Using just arn:aws:s3:::my-bucket for GetObject will NOT work.
Key rule: S3 has a bucket-level ARN and an object-level ARN. List/create/delete bucket operations use the bucket ARN. Object operations (Get, Put, Delete object) use bucket-name/*.
Lab Module 02

Roles & Assume Role

AssumeRole failures, cross-account role issues, trust policy misconfigurations, and role chaining gotchas.

06
Cross-account role assumption fails with "not authorized to assume role"
Account A (111111111111) tries to assume a role in Account B (222222222222). The role in Account B has a trust policy. The user gets AccessDenied on sts:AssumeRole.
AssumeRole Denied Cross-Account Trust Policy Hard
▶
Error Seen
An error occurred (AccessDenied) when calling the AssumeRole operation: User: arn:aws:iam::111111111111:user/dev is not authorized to perform: sts:AssumeRole on resource: arn:aws:iam::222222222222:role/cross-account-role
Clue
Two-sided handshake: Cross-account AssumeRole requires BOTH sides to allow it. Account B's role trust policy must list Account A's principal. AND Account A's IAM user/role must have an identity policy allowing sts:AssumeRole on Account B's role ARN.
Root Cause & Fix
  • 1
    Check trust policy in Account B's role: must include "Principal": {"AWS": "arn:aws:iam::111111111111:user/dev"} or "arn:aws:iam::111111111111:root".
  • 2
    Check Account A's identity policy: must include {"Effect":"Allow","Action":"sts:AssumeRole","Resource":"arn:aws:iam::222222222222:role/cross-account-role"}.
  • 3
    If Account B's trust policy uses arn:aws:iam::111111111111:root (the account root), then any user in Account A WITH the sts:AssumeRole identity policy permission can assume it.
  • 4
    Also check for an ExternalId condition in the trust policy — if one is set, you must pass --external-id VALUE in the CLI command.
Key rule: Cross-account = both sides must allow. Same-account role assumption only needs the trust policy (identity policy on the user isn't strictly required for same-account assumes to roles that list the specific user/role ARN).
07
Role assumed successfully but session expires after 1 hour despite setting 12 hours
A developer assumes a role and passes --duration-seconds 43200 (12 hours). The session still expires after 1 hour.
Session Duration MaxSessionDuration Medium
▶
Clue
Check the role's MaxSessionDuration setting. The role itself has a maximum session duration (default: 1 hour, max possible: 12 hours). You cannot request a session longer than this value. Also: role chaining hard-caps sessions at 1 hour regardless.
Root Cause & Fix
  • 1
    IAM → Roles → find the role → Summary tab → look at "Maximum session duration." It likely says 1 hour.
  • 2
    Click Edit → increase it up to 12 hours (43200 seconds).
  • 3
    Role chaining exception: If this role was assumed from ANOTHER role (not from a user), the max is hard-capped at 1 hour and cannot be changed. This is a hard AWS limit.
  • 4
    Temporary credentials: passed --duration-seconds is only honored up to the role's MaxSessionDuration. Requesting more than that returns a validation error.
Key rule: Role MaxSessionDuration defaults to 3600s (1hr). Must be explicitly increased per-role (up to 43200s = 12hr). Role chaining is always capped at 3600s — no exceptions.
08
Service role for EC2 throws "Entity Already Exists" when creating
An automation script creates an IAM role for EC2. It fails with "EntityAlreadyExists." Running it again throws the same error. The role doesn't appear in the console.
EntityAlreadyExists Instance Profile Medium
▶
Error Seen
An error occurred (EntityAlreadyExists) when calling the CreateRole operation: Role with name my-ec2-role already exists. -- OR -- An error occurred (EntityAlreadyExists) when calling the CreateInstanceProfile operation: Instance Profile my-ec2-role already exists.
Clue
IAM roles and instance profiles are separate resources with the same name. When the console creates an EC2 role, it auto-creates both a role and an instance profile with the same name. Your script created the role but the instance profile already existed from a previous partial run — or vice versa.
Root Cause & Fix
  • 1
    Check if the role exists: aws iam get-role --role-name my-ec2-role
  • 2
    Check if an instance profile exists: aws iam get-instance-profile --instance-profile-name my-ec2-role
  • 3
    To delete and recreate: first remove the role from the instance profile (aws iam remove-role-from-instance-profile), then delete the instance profile, then delete the role.
  • 4
    Idempotent approach: wrap creation calls in try/except and check for EntityAlreadyExists — if it exists, use it; if not, create it.
Key rule: EC2 instance profiles are separate IAM resources. They are containers for roles, not roles themselves. The AWS Console hides this by auto-creating and naming them identically. IaC tools (Terraform, CloudFormation) handle them as separate resources.
09
GitHub Actions OIDC role assumption works locally but fails in CI pipeline
A developer configures OIDC federation for GitHub Actions. Testing locally with a manually obtained token works. The actual pipeline throws "Not authorized to perform sts:AssumeRoleWithWebIdentity."
OIDC Failure GitHub Actions Trust Policy Hard
▶
Error Seen
Error: Not authorized to perform sts:AssumeRoleWithWebIdentity Condition: StringEquals for key token.actions.githubusercontent.com:sub does not match the request
Clue
The sub claim format is environment-dependent. The trust policy condition checks the sub claim of the GitHub OIDC JWT. The format is repo:OWNER/REPO:ref:refs/heads/BRANCH or repo:OWNER/REPO:environment:ENVIRONMENT. The exact string must match.
Root Cause & Fix
  • 1
    Check the trust policy's condition: "token.actions.githubusercontent.com:sub": "repo:myorg/myrepo:ref:refs/heads/main"
  • 2
    The CI pipeline might be running on a PR branch (refs/heads/feature-x), a tag (refs/tags/v1.0), or an environment — none of which match refs/heads/main.
  • 3
    Fix: Use StringLike instead of StringEquals with a wildcard: "repo:myorg/myrepo:*" — but scope to only what you need for security.
  • 4
    Also verify the aud (audience) claim condition: "token.actions.githubusercontent.com:aud": "sts.amazonaws.com" — must match exactly.
  • 5
    Ensure the GitHub OIDC provider is registered in IAM with the correct thumbprint for token.actions.githubusercontent.com.
Key rule: OIDC trust conditions must match the exact JWT claims the CI platform issues. Always print the actual token claims in your pipeline first (jq -R 'split(".") | .[1] | @base64d | fromjson') to see the exact sub value.
Lab Module 03

Policy Conflicts

Scenarios where multiple policies interact — group policies, inline vs managed, wildcard confusion, and NotAction traps.

10
User in two groups gets LESS access than either group has individually
Group A has S3 full access. Group B has a Deny on s3:DeleteObject. A user in both groups can't delete S3 objects — but also unexpectedly can't read objects either.
Unexpected Deny Group Policy Medium
▶
Clue
Inspect Group B's Deny statement closely. It may say "Action": "s3:*" with a condition, or the Deny may have an overly broad resource scope. A single poorly written Deny can block all S3 access when combined with other group policies.
Root Cause & Fix
  • 1
    Use the IAM Policy Simulator: enter the user ARN, test s3:GetObject, and check which policy statement produces the Deny.
  • 2
    Group B's Deny is likely "Action": "s3:*" with a condition like StringNotEquals s3:prefix: allowed/ — this Deny applies to ALL s3 actions, not just delete.
  • 3
    Fix: Narrow the Deny action to only s3:DeleteObject: {"Effect":"Deny","Action":"s3:DeleteObject","Resource":"*"}
  • 4
    IAM Policy Simulator is the fastest debugging tool — always use it before blaming infrastructure.
Key rule: All policies for a user are merged. A Deny in ANY attached policy (user, group A, group B, inline) blocks the action. Groups do not isolate permissions from each other.
11
NotAction policy gives too much access unexpectedly
A security policy intends to block all access EXCEPT reading CloudWatch logs. It uses NotAction with logs:GetLogEvents. But the user can now create EC2 instances, manage IAM users, and more.
Overly Permissive NotAction Hard
▶
Clue
NotAction with Allow is NOT a Deny-all-else. "Effect":"Allow","NotAction":"logs:GetLogEvents" means "allow every action EXCEPT logs:GetLogEvents." This grants ALL other AWS actions — the opposite of what was intended. NotAction with Allow is almost never the right choice.
Root Cause & Fix
  • 1
    The policy author confused NotAction semantics. Allow + NotAction = grant everything except listed. Deny + NotAction = deny everything except listed (this is the safe version).
  • 2
    Correct approach for "only allow CloudWatch read":
    {"Effect":"Allow","Action":["logs:GetLogEvents","logs:DescribeLogGroups"],"Resource":"*"}
  • 3
    Valid use of Deny+NotAction: lock an account down to only IAM actions during an incident: {"Effect":"Deny","NotAction":["iam:*","sts:*"],"Resource":"*"}
Key rule: Allow + NotAction = "allow everything EXCEPT." Deny + NotAction = "deny everything EXCEPT." These are completely different behaviors. NotAction with Allow is almost always a security mistake.
12
Inline policy and managed policy conflict — user has both attached
A user has an inline policy allowing ec2:* and a managed policy denying ec2:TerminateInstances. The user still cannot terminate instances. They try deleting the managed policy but it's detached from the user but they still can't terminate.
Inline vs Managed Deny Persists Medium
▶
Clue
Check ALL sources of policies: Directly attached policies (managed), inline policies on the user, AND policies from every group the user belongs to. Detaching a managed policy from the user doesn't remove it from their groups.
Root Cause & Fix
  • 1
    Go to IAM → User → Permissions tab. Click the triangle beside each group. The Deny policy is likely attached to a GROUP the user belongs to, not the user directly.
  • 2
    IAM Policy Simulator shows "Deny" with the source policy — it will show the group name where the Deny originates.
  • 3
    Fix: Remove the Deny policy from the group, OR remove the user from that group, OR modify the Deny to add a condition that excludes this user.
Key rule: A user's effective permissions = ALL inline + ALL directly attached managed + ALL group policies combined. An explicit Deny in any one of these sources blocks the action everywhere.
Lab Module 04

MFA & Credentials

MFA enforcement failures, access key rotation issues, and credential precedence in the CLI.

13
MFA enforcement policy blocks EVERYTHING including changing user's own MFA device
An admin adds an MFA enforcement policy. A new user who hasn't set up MFA yet can't log in or do anything — including setting up MFA. They're completely locked out.
MFA Lockout Self-Service MFA Hard
▶
Clue
The MFA enforcement policy must explicitly ALLOW the actions needed to set up MFA, even without MFA present. Otherwise a user without MFA can't set up MFA — a catch-22 created by overly strict MFA policies.
Root Cause & Fix
  • 1
    The policy Denies all actions when aws:MultiFactorAuthPresent is false — but doesn't make exceptions for MFA setup actions.
  • 2
    Add explicit Allow (without MFA condition) for self-service MFA actions:
    iam:CreateVirtualMFADevice, iam:EnableMFADevice, iam:GetUser, iam:ListMFADevices, iam:ListVirtualMFADevices, iam:ResyncMFADevice, sts:GetSessionToken
  • 3
    Also allow: iam:ChangePassword (so they can change their initial temporary password without MFA) and iam:GetAccountPasswordPolicy.
  • 4
    Emergency recovery: admin temporarily detaches the MFA enforcement policy from the user, lets them set up MFA, then re-attaches it.
Key rule: MFA enforcement policies must include a carve-out for the IAM actions needed to enroll in MFA. AWS publishes a well-known MFA enforcement policy template — always use it as a starting point rather than writing your own from scratch.
14
AWS CLI uses wrong account credentials even after aws configure
A developer configures new credentials with aws configure. But CLI commands still use old credentials from a different account. aws sts get-caller-identity shows the wrong account ID.
Credential Precedence Environment Variables Easy
▶
Clue
AWS CLI has a credential precedence chain. Environment variables (AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY) take priority OVER ~/.aws/credentials. If env vars are set in the shell, aws configure changes the file but the env vars still win.
Root Cause & Fix
  • 1
    Check for env vars: env | grep AWS — if AWS_ACCESS_KEY_ID is set, that's what's being used.
  • 2
    Unset them: unset AWS_ACCESS_KEY_ID AWS_SECRET_ACCESS_KEY AWS_SESSION_TOKEN AWS_DEFAULT_REGION
  • 3
    Full CLI precedence order (highest to lowest):
    1. CLI options (--profile, --region)
    2. Env vars (AWS_ACCESS_KEY_ID etc.)
    3. ~/.aws/credentials file (default profile)
    4. ~/.aws/config file
    5. Container credential provider (ECS)
    6. Instance metadata (EC2 IMDS)
  • 4
    Use named profiles to avoid conflicts: aws configure --profile dev, then aws s3 ls --profile dev or export AWS_PROFILE=dev.
Key rule: Credential precedence is the most common CLI confusion. Always run aws sts get-caller-identity to confirm which identity the CLI is using before debugging anything else.
15
Access key works in CLI but the same key fails in application code
An access key pair works perfectly in the AWS CLI. The same keys copied into application environment variables throw "InvalidClientTokenId."
InvalidClientTokenId Access Keys Medium
▶
Error Seen
An error occurred (InvalidClientTokenId): The security token included in the request is invalid.
Clue
The error is "InvalidClientTokenId" not "AccessDenied" — this means the credentials themselves are malformed or wrong, not that they lack permissions. Check for copy-paste errors, trailing spaces, or line breaks.
Root Cause & Fix
  • 1
    Print the exact values: echo "$AWS_ACCESS_KEY_ID" | xxd | head — look for invisible characters, trailing newlines, or quote marks.
  • 2
    Access Key IDs always start with AKIA (long-term) or ASIA (temporary/STS). If yours doesn't, it's malformed.
  • 3
    STS keys need a third variable: AWS_SESSION_TOKEN. If you copy an access key + secret key from a role session but forget the session token, you'll get this error.
  • 4
    Also check if the key has been deactivated: IAM → Users → Security credentials → Access keys status must be "Active."
Key rule: InvalidClientTokenId = bad credentials format. AccessDenied = valid credentials but insufficient permissions. These require completely different debugging paths.
Lab Module 05

Federation & SSO

SAML federation failures, Identity Center misconfigurations, and web identity issues.

16
SAML federation succeeds but user lands on "You don't have permissions" page
SAML assertion from Okta is accepted by AWS. The user is redirected to the AWS console but immediately sees an authorization error and can't access any service.
Auth Error SAML 2.0 Role ARN Hard
▶
Clue
Authentication succeeded but authorization is empty. The role the SAML assertion maps to likely has no permissions policies attached — or the SAML attribute mapping is pointing to a role with only a trust policy and no permissions policy, or the role ARN in the Okta app config is wrong.
Root Cause & Fix
  • 1
    Decode the SAML assertion (base64 decode the SAMLResponse in browser dev tools). Find the https://aws.amazon.com/SAML/Attributes/Role attribute.
  • 2
    Confirm the role ARN in the SAML attribute exactly matches an existing IAM role ARN. Check for typos, wrong account ID, or stale role names.
  • 3
    In IAM, open the role and check the Permissions tab — if there are no policies attached, the user can authenticate but do nothing. Attach appropriate policies.
  • 4
    Also verify the Trust policy of the role contains: "Principal": {"Federated": "arn:aws:iam::ACCOUNT:saml-provider/PROVIDER_NAME"} and "Action": "sts:AssumeRoleWithSAML".
Key rule: SAML authentication and authorization are separate. AWS accepts the SAML assertion → issues credentials for the role → user can only do what the role's permission policies allow. A role with no policies = authenticated but unauthorized.
17
IAM Identity Center user can access console but CLI token expires immediately
A developer uses aws sso login and gets console access. But when they run CLI commands with the SSO profile, they get "Token has expired and refresh failed."
SSO Token Expiry Identity Center Medium
▶
Root Cause & Fix
  • 1
    AWS SSO/Identity Center CLI tokens are stored in ~/.aws/sso/cache/. They expire based on the session duration configured in the Permission Set.
  • 2
    Run aws sso login --profile PROFILE_NAME to get a fresh token. Ensure the ~/.aws/config profile has all four fields: sso_start_url, sso_account_id, sso_role_name, sso_region.
  • 3
    The AWS CLI v1 does NOT support SSO profiles natively — upgrade to CLI v2: aws --version must show 2.x.x.
  • 4
    To increase session duration: Identity Center → Permission Sets → edit session duration (max 12 hours). The default is often 1 hour.
Key rule: Identity Center SSO requires AWS CLI v2 and the profile must be configured with the SSO fields, not regular access key fields. CLI v1 will not work with SSO profiles.
Lab Module 06

Permissions Boundaries

The most misunderstood IAM feature — scenarios where boundaries silently restrict access or allow privilege escalation gaps.

18
User with AdministratorAccess can't create a role with more than S3 access
A developer has AdministratorAccess but has a permissions boundary attached. They can create IAM roles but the roles can only ever access S3 — attaching any other policy to roles they create fails.
Boundary Restriction Permissions Boundary Hard
▶
Clue
Permissions boundaries cap what a principal can grant to others. If a developer has a boundary that only allows S3 actions, they can only create roles/users with at most S3 permissions — they cannot grant broader permissions than their own boundary allows, preventing privilege escalation.
Root Cause & Fix
  • 1
    Check the user's boundary: IAM → User → Permissions tab → look for "Permissions boundary."
  • 2
    The boundary policy only allows s3:*. Effective permissions = intersection(AdministratorAccess, boundary) = only S3 actions.
  • 3
    The boundary also controls what IAM resources they can create — you can't grant a role more permissions than your own effective permissions.
  • 4
    Fix: Update the boundary policy to include the additional services needed, OR have an admin with no boundary (or a broader boundary) create the role.
  • 5
    This behavior is intentional for safe delegation — team leads can manage IAM within their own scope without risk of privilege escalation.
Key rule: Effective permissions = intersection(identity policy, boundary). AdministratorAccess with an S3-only boundary = S3-only. The boundary is a ceiling — it can never expand what the identity policy allows.
19
Boundary is set on a user but has no effect — user still has full access
A security team attaches a permissions boundary restricting access to only EC2. The user still has full admin access and can call IAM, S3, and everything else.
Boundary Bypassed Permissions Boundary Hard
▶
Clue
Boundaries apply to IAM users and roles, NOT to the root user, and NOT to roles that haven't had the boundary attached. If the user frequently assumes a role, the boundary on the user doesn't apply to the assumed role session — the role needs its own boundary.
Root Cause & Fix
  • 1
    Confirm the user is actually using the IAM user identity and not assuming a role. Run aws sts get-caller-identity — if the ARN shows assumed-role, the user is using a role, not their user identity.
  • 2
    Boundaries on a user do NOT apply to roles that user assumes. The role needs its own separate boundary attached.
  • 3
    Also verify the boundary is actually ATTACHED: IAM → User → Permissions tab → "Permissions boundary" section should show the boundary policy name, not "None."
  • 4
    Root account: boundaries never apply to root. This is intentional — root is unrestricted by design.
Key rule: Permissions boundaries apply to the specific entity they're attached to. Assuming a role creates a NEW session identity — that role's effective permissions are governed by the role's own policies and any boundary on the role itself.
Lab Module 07

SCPs & Organizations

Service Control Policy pitfalls — why allows don't work, how denies cascade, and management account exceptions.

20
SCP allows S3 access but users in the OU still get AccessDenied
A new OU has an SCP with "Effect":"Allow","Action":"s3:*","Resource":"*". Users in accounts under this OU get AccessDenied on all S3 operations.
SCP Misunderstanding Allow vs Deny Hard
▶
Clue
SCPs do NOT grant permissions — they only restrict them. An SCP Allow does nothing on its own. Users still need identity-based policies that allow the actions. The SCP Allow just means "this action is NOT blocked by the SCP" — users still need a real Allow in their IAM policy.
Root Cause & Fix
  • 1
    SCP semantics: an SCP Allow means "don't block this." It does NOT mean "grant this." Think of SCPs as a filter on top of IAM, not as a source of permissions.
  • 2
    The users in the OU accounts still need IAM identity policies (user policies, role policies, or group policies) that Allow s3 actions.
  • 3
    Full permission logic: SCP allows it AND IAM policy allows it → access granted. If either denies or is missing an allow → access denied.
  • 4
    Check: is the AWS account using the default FullAWSAccess SCP or has it been replaced? New OUs may have removed the default FullAWSAccess SCP, which blocks everything.
Key rule: SCPs are guardrails, not grants. "Allow" in an SCP means "not blocked." Users still need IAM identity policies. The FullAWSAccess SCP (which all accounts start with) is what makes normal IAM policies work — it "allows" everything to pass through.
21
Management account admin is blocked by an SCP that blocks all regions except us-east-1
A company enforces a region restriction SCP. A developer in the management account tries to launch EC2 in eu-west-1 and gets AccessDenied.
SCP Region Lock Management Account Hard
▶
Clue
SCPs do NOT apply to the management account (master account) of an AWS Organization. If the developer is in the management account, SCPs never affect them regardless of what the SCP says. If they're still blocked, the restriction comes from elsewhere.
Root Cause & Fix
  • 1
    Confirm whether the user is in the management account: AWS Organizations → check which account is the management account.
  • 2
    If user is in management account and still blocked → SCPs are NOT the cause. Look at IAM policies, resource policies, or IAM permissions boundaries on the user.
  • 3
    If user is in a member account → SCPs apply. Check the OU hierarchy — each OU in the chain (root → OU → account) must allow the action. All SCPs in the hierarchy are ANDed.
  • 4
    Fix for member account: modify the SCP region restriction to add eu-west-1, OR move the account to an OU without the restriction, OR use a Deny SCP with a condition exception for specific roles.
Key rule: SCPs never apply to the management account. They only apply to member accounts. This is a critical architecture consideration — put as little as possible in the management account and do actual work in member accounts.
22
Resource Control Policy (RCP) blocks access from an external vendor's account
A vendor in Account X needs to read S3 objects in your org's Account Y. Bucket policy allows it. SCP allows it. But the vendor still gets AccessDenied. An RCP was recently deployed.
RCP BlockData PerimeterHard
▶
Root Cause & Fix
  • 1
    Resource Control Policies (RCPs, introduced late 2024) restrict what resources in your org can accept — they're a "data perimeter" complement to SCPs.
  • 2
    An RCP like Deny s3:GetObject if aws:PrincipalOrgID != your-org-id blocks ALL external accounts including legitimate vendors.
  • 3
    Fix: add a condition exception to the RCP: StringNotEquals aws:PrincipalAccount: ["vendor-account-id"] to allow specific external accounts through.
Key rule: RCPs are applied to resources in your org and affect who can access them from outside. They layer on top of bucket policies — both must allow access for external principals to succeed.
23
After tagging a role, tag-based SCP suddenly starts blocking it
An SCP denies all actions if the principal doesn't have tag team=platform. A role that worked fine gets the tag added. Suddenly it's blocked from EC2.
ABAC + SCPPrincipal TagsHard
▶
Root Cause & Fix
  • 1
    The SCP checks aws:PrincipalTag/team. The role was tagged with team=platform. But the SCP condition uses StringNotEquals on a different value — the tag value may be misspelled or using wrong case.
  • 2
    Session tags (passed at AssumeRole time) override principal tags for condition evaluation. If the role's session tag has a different value, the condition evaluates the session tag value, not the role tag.
  • 3
    Fix: verify the exact tag key and value. IAM tags are case-sensitive. team=Platform ≠ team=platform.
24
IAM changes made by admin take 5+ minutes to take effect in an application
An admin adds a policy to a role. The application still gets AccessDenied for 5-10 minutes, then starts working. This happens every time a policy change is made.
Eventual ConsistencyIAM PropagationMedium
▶
Root Cause & Fix
  • 1
    IAM is eventually consistent — changes replicate across AWS global infrastructure and typically take a few seconds to minutes to propagate everywhere.
  • 2
    The application's existing role session credentials are cached. Even after the policy update, the old credentials in the app cache don't reflect the change until the session is refreshed.
  • 3
    Fix for the application: force a credential refresh (restart the app or invalidate the credential cache). For SDKs, credential refresh is automatic when the session expires.
  • 4
    Architectural fix: avoid making IAM changes in critical high-availability code paths. Pre-configure roles and policies before deploying applications.
Key rule: IAM is eventually consistent. Factor this into deployments — don't apply IAM changes and immediately deploy code that depends on them. Add a wait or retry mechanism.
25
Service-linked role can't be deleted even though no service is using it
An admin tries to delete a service-linked role for EKS. Gets "must delete the service or resources using the role first." No EKS clusters exist in the account.
Deletion BlockedService-Linked RoleMedium
▶
Root Cause & Fix
  • 1
    Service-linked roles can only be deleted after the AWS service confirms no resources are still using them. EKS may have residual resources (node groups, add-ons, vpc configs) even without visible clusters.
  • 2
    Check ALL regions: EKS clusters or node groups may exist in other regions. Service-linked roles are global but the resources using them may be in any region.
  • 3
    Use deletion validation: aws iam delete-service-linked-role --role-name AWSServiceRoleForAmazonEKS — if it returns a deletion task ID, poll aws iam get-service-linked-role-deletion-status to see what's blocking it.
  • 4
    The status response will list the exact resources in each region that must be deleted first.
Key rule: Service-linked roles are region-aware but globally scoped. You cannot delete them while any regional resource (in ANY region) depends on them. The deletion task API gives you the exact blocking resources.

Core IAM Concepts

The building blocks of IAM — everything you must understand cold before any other IAM concept makes sense.

What IAM is
ℹ
IAM is a global service — not region-specific. Users, roles, and policies created in IAM are available in all AWS regions. However, IAM is eventually consistent — changes may take seconds to minutes to propagate globally.
👤
IAM Users
A persistent identity for a person or app. Has long-term credentials (password or access keys). ARN: arn:aws:iam::ACCT:user/NAME. Limit: 5,000 per account.
👥
IAM Groups
Collection of users sharing policies. NOT a principal — cannot be used as a Principal in resource policies. Max 300 groups/account. A user can be in max 10 groups. No nested groups.
🎭
IAM Roles
Assumable identity. Requires a trust policy (who can assume) + permissions policy (what it can do). Issues temporary STS credentials. No long-term credentials.
📄
IAM Policies
JSON documents defining permissions. 7 types: identity-based, resource-based, permissions boundary, SCP, RCP, ACL, session policy.
🔑
Principal
Any entity that can make a request: IAM user, IAM role, AWS service, federated user, anonymous user. Root user is a special principal that cannot be restricted.
🛡
Default Deny
All requests are denied by default. An explicit Allow is needed. An explicit Deny always overrides any Allow. There is no "override the Deny" mechanism.
The 4 credential types for IAM users
CredentialUsed ForKey Facts
Console PasswordAWS Management Console loginMust be explicitly enabled. Password policy enforced at account level. Initial password can be set to require reset.
Access KeysCLI, SDK, API callsMax 2 per user. Start with AKIA (long-term) or ASIA (STS). Can be active or inactive. Should be rotated regularly.
SSH Public KeysAWS CodeCommitUp to 5 per user. Only for CodeCommit — not general AWS authentication.
X.509 CertificatesLegacy SOAP protocol APIsLargely obsolete. Only needed for old SOAP-based AWS services. Not used for modern AWS APIs.
Authentication vs Authorization
✓
Authentication = Who are you? (credentials verified)
Authorization = What are you allowed to do? (policies evaluated)
IAM handles both. You can be authenticated but have zero authorization (empty policy = deny all).
ARN Format — memorize this

│ │ │ │ └─ resource type/name
│ │ │ └─ account ID (empty for IAM = any account)
│ │ └─ service
│ └─ partition (aws, aws-cn, aws-us-gov)
└─ literal "arn"

Roles & AWS STS

How roles work, what STS does, and all the ways temporary credentials can be issued.

Role components
🤝
Trust Policy
Defines WHO can assume the role (the Principal). Required. Without it, no one can use the role. This is the "who am I willing to give these powers to?" document.
📋
Permissions Policy
Defines WHAT the role can do once assumed. Can be managed or inline. Without it, the role has no powers — identity authenticated but authorized for nothing.
⏱
Session Duration
Default 1hr. Configurable up to 12hr per role. Role chaining hard cap: 1hr regardless of role setting. Requested duration cannot exceed the role's MaxSessionDuration.
🔖
Session Tags
Key-value pairs passed at assume-time. Used in ABAC conditions. Marked transitive to carry through role chains. Cannot be broader than the role's own tags.

Quick-Fire Q&A

40 questions covering everything from the documentation. Click any question to reveal the answer.

Identities
Q1What is the maximum number of IAM users you can create in one AWS account?▶
Answer
5,000 IAM users per account. This is a soft limit that can be increased via Service Quotas. If you need more than 5,000 human users, use IAM Identity Center with an external identity provider instead.
Q2Can an IAM group be specified as a Principal in a resource-based policy (e.g., an S3 bucket policy)?▶
Answer
No. IAM groups are NOT principals. They cannot appear in the Principal element of any policy. Only users, roles, AWS services, federated identities, and AWS account roots can be principals.
Q3How many access key pairs can a single IAM user have at one time?▶
Answer
Maximum 2. This exists to allow key rotation: create new key → update applications → deactivate old key → delete old key. An active key starts with AKIA. An STS-issued temporary key starts with ASIA.
Q4A user is in Group A and Group B. Group A allows S3 full access. Group B denies S3 DeleteObject. Can the user delete S3 objects?▶
Answer
No. The Deny in Group B overrides the Allow in Group A. All policies are merged — any explicit Deny from any source wins over any Allow. The user can do all S3 actions EXCEPT delete objects.
Q5What happens when you create a new IAM user — can they immediately log in to the console?▶
Answer
No. Console access must be explicitly enabled by setting a console password. Creating a user only creates the identity — no credentials are automatically issued. You must explicitly enable console access and/or create access keys.
Policies
Q6Does an SCP with Effect:Allow grant permissions to IAM users in that account?▶
Answer
No. SCPs never grant permissions — they only restrict them. An SCP Allow means "this service/action is not blocked at the organizational level." Users still need identity-based or resource-based policies that actually grant the permissions.
Q7What is the difference between an AWS managed policy and a customer managed policy?▶
Answer
AWS managed: Created and maintained by AWS. Updated by AWS when services change. You cannot edit them. Prefixed with "arn:aws:iam::aws:policy/".
Customer managed: You create and maintain them. Up to 5 versions. Can be attached to multiple users/roles/groups. You control updates.
Q8What does "Effect":"Allow","NotAction":"iam:*" mean?▶
Answer
Allow every AWS action EXCEPT iam:* actions. NotAction with Allow is "allow everything not in this list." This grants extremely broad access. The intentional use case for NotAction is usually with Deny: Deny + NotAction + iam:* = "deny everything except IAM actions" — used to lock down an account while preserving IAM access for emergency recovery.
Q9A user has a permissions boundary that only allows EC2 actions. Their identity policy allows S3 full access. What can they actually do?▶
Answer
Nothing on S3. Only EC2 actions (if their identity policy also allows them). Effective permissions = intersection(identity policy, boundary). The boundary is a ceiling — S3 is blocked by the boundary. EC2 is allowed by the boundary, but the identity policy must also allow the specific EC2 actions for them to work.
Q10What is the Version element in an IAM policy and what should it always be set to?▶
Answer
Always set to "2012-10-17". This is the current policy language version. The older version "2008-10-17" doesn't support policy variables like ${aws:username}. Always use "2012-10-17" — without it, policy variables won't work and you'll get unexpected behavior.
Roles & STS
Q11What is the maximum duration for a role session when role chaining is used?▶
Answer
1 hour (3,600 seconds) — hard limit, cannot be changed. Role chaining = assuming a role from within another role session. Even if the target role's MaxSessionDuration is set to 12 hours, chained sessions are always capped at 1 hour. This is a security measure to limit the duration of privilege escalation chains.
Q12What are the three values returned by every STS AssumeRole call?▶
Answer
AccessKeyId, SecretAccessKey, and SessionToken (plus Expiration). All three are required for API calls. Forgetting the SessionToken causes InvalidClientTokenId. Temporary credentials (ASIA...) always require all three — unlike long-term access keys (AKIA...) which only need the key pair.
Q13What is the ExternalId used for in cross-account role assumptions?▶
Answer
Prevents the "confused deputy" attack. When a third-party service assumes your role on your behalf, a malicious third party could trick that service into assuming your role without your knowledge. The ExternalId is a shared secret you add to the role's trust policy condition — the service must provide it to assume the role. Without it, any entity with sts:AssumeRole permission could assume your role.
Q14What is an IAM instance profile and how does it differ from an IAM role?▶
Answer
An instance profile is a container that holds a single IAM role for attachment to EC2 instances. The role holds the policies. The instance profile is the mechanism through which EC2 gets the role. The AWS Console auto-creates a matching instance profile when you create a role via the EC2 use case wizard. In the CLI/IaC, you must create them separately: aws iam create-instance-profile then aws iam add-role-to-instance-profile.
Q15Which STS API call can never be denied by any IAM policy?▶
Answer
GetCallerIdentity. This call returns the account ID, user ID, and ARN of the current caller. It cannot be denied by any IAM policy, SCP, permissions boundary, or resource policy. Useful for debugging, confirming identity in CI/CD, and verifying which credentials are in use.
Security & Best Practices
Q16Do SCPs apply to the management account (master account) in AWS Organizations?▶
Answer
No. SCPs never apply to the management account. The management account is always exempt from SCPs, even if it's in an OU with an SCP. This is intentional — the management account needs unrestricted access to manage the organization. Best practice: don't run workloads in the management account for this reason.
Q17What is the AWS CLI credential precedence order?▶
Answer
Highest to lowest:
1. CLI options (--region, --profile flags)
2. Environment variables (AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_SESSION_TOKEN)
3. AWS SSO session credentials
4. ~/.aws/credentials file (default profile)
5. ~/.aws/config file
6. Container credentials (ECS task role)
7. Instance metadata (EC2/ECS IMDS role)

Env vars override ~/.aws/credentials — this is the most common source of "wrong account" issues.
Q18What is IAM Access Analyzer's "Policy Generation" feature and what does it require?▶
Answer
Policy Generation reads CloudTrail logs for an IAM entity and synthesizes a least-privilege policy containing only the actions actually called during the observed period. Requirements: CloudTrail must be enabled and capturing management events for the desired time window. Output is a starting point — add edge-case actions the workload might need but hasn't exercised yet.
Q19What is the difference between IAM Access Analyzer external access analysis and unused access analysis in terms of cost?▶
Answer
External access analysis is FREE. Unused access analysis is a paid feature charged per IAM role/user analyzed per day. Policy validation and policy generation are also free. Custom policy checks (CI/CD pipeline integration) are paid.
Q20What does "IAM is eventually consistent" mean in practice?▶
Answer
IAM changes (creating users, attaching policies, updating roles) are replicated across multiple AWS global data centers. This takes seconds to minutes. Consequences: (1) A just-created IAM role may not immediately be usable — add retry logic. (2) Don't put IAM changes in critical high-availability startup sequences. (3) Use IAM change management and pre-create all IAM resources before deploying applications that need them.
Federation & Identity Center
Q21How many identity sources can IAM Identity Center have per AWS Organization?▶
Answer
Exactly ONE per organization. Options: (1) Built-in Identity Center directory, (2) External SAML 2.0 IdP (Okta, Azure AD, Google Workspace), (3) AWS Managed Microsoft AD. Changing the source is destructive — all assignments are deleted. Plan carefully before switching.
Q22What is a Permission Set in IAM Identity Center?▶
Answer
A reusable template defining a collection of IAM policies. When assigned to a user/group + target account, Identity Center automatically creates an IAM role in that account. Update the permission set → all roles across all accounts update automatically. Supports AWS managed policies, customer managed policies, inline policies, and permissions boundaries.
Q23What version of the AWS CLI is required for IAM Identity Center SSO profiles?▶
Answer
AWS CLI version 2. CLI v1 does not support SSO/Identity Center profiles. If you're using CLI v1 with an SSO profile, you'll get token errors. Verify with aws --version — must show 2.x.x.
Q24In SAML federation, which SAML attribute maps to the IAM role the user will assume?▶
Answer
https://aws.amazon.com/SAML/Attributes/Role — value format: arn:aws:iam::ACCOUNT:role/ROLE_NAME,arn:aws:iam::ACCOUNT:saml-provider/PROVIDER_NAME (role ARN and provider ARN separated by comma). Other key attributes: .../RoleSessionName (sets the session name in CloudTrail) and .../SessionDuration (controls credential lifetime).
Q25What is the maximum pre-signed URL expiry when generated using IAM user long-term credentials vs STS temporary credentials?▶
Answer
IAM user (AKIA... keys): max 7 days.
STS temporary credentials (ASIA...): expires at the EARLIER of the URL expiry OR the STS credential expiry. If the STS session is set to 1 hour, the pre-signed URL expires in 1 hour even if you set it to 24 hours. This is the most common pre-signed URL gotcha.
Misc & Advanced
Q26What is ABAC in IAM and what condition key is used to match principal tags to resource tags?▶
Answer
ABAC = Attribute-Based Access Control. Uses tags on both principals and resources. The condition key is aws:ResourceTag/KEY (resource's tag) compared against aws:PrincipalTag/KEY (principal's tag). Example: allow access when aws:ResourceTag/team == ${aws:PrincipalTag/team}. One policy scales to thousands of resources — no need to update the policy when adding new resources.
Q27What is the IAM policy variable ${aws:username} and what policy version must be set for it to work?▶
Answer
${aws:username} is a policy variable that AWS replaces with the calling user's username at evaluation time. Policy version must be "2012-10-17" (not "2008-10-17"). Classic use: "Resource": "arn:aws:s3:::user-data/${aws:username}/*" — each user only accesses their own S3 prefix with one shared policy.
Q28A service-linked role exists for EKS. You try to delete it but get an error. What must you do first?▶
Answer
Delete all AWS resources that use the service-linked role in ALL regions. EKS clusters, node groups, or Fargate profiles in any region block deletion. Use aws iam delete-service-linked-role — it returns a deletion task ID. Poll aws iam get-service-linked-role-deletion-status to get the exact list of blocking resources in each region.
Q29What is "Trusted Identity Propagation" in IAM Identity Center?▶
Answer
A feature (2025) that allows AWS services like S3, Redshift, and QuickSight to receive and act on the original user's identity when making downstream service calls. Without it, downstream calls appear as the service role — losing the human user in audit trails. With it, CloudTrail shows the original human user through the entire call chain, enabling end-to-end auditability for data services.
Q30What does IAM Roles Anywhere allow and what are its three main components?▶
Answer
Allows on-premises servers and workloads to get temporary AWS credentials using X.509 certificates — no long-term access keys needed.

Three components:
1. Trust Anchor — registered CA certificate authority AWS trusts
2. Profile — links trust anchor to IAM role(s) + sets session duration
3. aws_signing_helper — binary on-prem that signs requests with X.509 cert → receives temporary STS credentials
Tricky / Exam-style
Q31Can root account access be restricted by IAM policies, SCPs, or permissions boundaries?▶
Answer
No. Root account access cannot be restricted by IAM policies or permissions boundaries. SCPs also do NOT apply to root users within member accounts — SCPs restrict the account's IAM principals, not the root user. The only ways to protect the root account are: enable MFA, delete root access keys, and restrict tasks requiring root to only those explicitly requiring it.
Q32A resource-based policy on an S3 bucket allows Account B full access. Account B has no IAM policies allowing S3. Can users in Account B access the bucket?▶
Answer
No (for IAM users/roles in Account B). Cross-account access requires both sides: the resource policy must allow Account B AND Account B's IAM users/roles must have identity policies allowing the actions. The bucket policy grants Account B the ability to be allowed, but individual users still need IAM policies.

Exception: if the bucket policy grants access to arn:aws:iam::ACCOUNT-B:root, Account B can then delegate via IAM to its users.
Q33What happens if you delete a customer managed policy that is still attached to users and roles?▶
Answer
You cannot. AWS prevents you from deleting a managed policy that is still attached to any entity. You must first detach it from all users, roles, and groups. Contrast with inline policies — when you delete a user or role, its inline policies are deleted with it automatically.
Q34What is the "confused deputy" problem and how does IAM address it?▶
Answer
When service A can assume role R in your account, a malicious actor using service A could trick it into assuming your role to access your resources — A acts as a "confused deputy." IAM addresses this with the ExternalId condition on cross-account role trust policies. The ExternalId is a secret only you and the legitimate third-party service know — any other caller cannot provide it and thus cannot assume the role.
Q35You have a role with MaxSessionDuration=12hr. A user assumes it requesting 8hr. Then from that session assumes another role with MaxSessionDuration=12hr requesting 8hr. How long is the second session?▶
Answer
1 hour maximum — the role chaining hard cap. When role B is assumed from within role A's session (chaining), the resulting session is ALWAYS capped at 1 hour regardless of what either role's MaxSessionDuration says, and regardless of what duration was requested. This cap exists to limit the temporal scope of privilege escalation chains.
Q36What is the minimum and maximum expiry for a pre-signed S3 URL?▶
Answer
Minimum: 1 second. Maximum: 604,800 seconds (7 days) when signed with long-term IAM user credentials using AWS Signature Version 4. With STS temporary credentials, max expiry is bounded by the credential lifetime (may be as short as 15 minutes). The URL is signed at generation time — AWS doesn't validate whether the object exists until the URL is accessed.
Q37Can you attach a permissions boundary to an IAM group?▶
Answer
No. Permissions boundaries can only be attached to IAM users and IAM roles — not groups, not root, not any other entity type. Groups are not principals and do not have boundaries. To limit what a group of users can do, attach boundaries to each individual user, or use SCPs at the account level.
Q38What is the last_used field in the IAM credential report, and how current is the data?▶
Answer
The access_key_N_last_used_date field shows the most recent date and time the access key was used to authenticate. Data is updated approximately once per day — it is NOT real-time. For more granular last-access data, use the IAM Access Advisor (shows per-service last accessed, tracked up to 400 days). The credential report itself can be regenerated at most once every 4 hours.
Q39What IAM condition key enforces that a request came from within a specific AWS Organization?▶
Answer
aws:PrincipalOrgID — checks the AWS Organization ID of the principal making the request. Use in resource policies to restrict access to only principals within your organization: {"StringEquals": {"aws:PrincipalOrgID": "o-xxxxxxxxxx"}}. More maintainable than listing all account ARNs — new accounts automatically inherit access when they join the org.
Q40What is Outbound Identity Federation in IAM (new Nov 2025) and what problem does it solve?▶
Answer
Outbound Identity Federation allows AWS workloads to authenticate with external third-party services using short-lived JWTs issued by IAM — without storing third-party API keys inside AWS. Previously, workloads needing to call external APIs stored those credentials in Secrets Manager or environment variables. Now IAM can issue a JWT that the external service validates, eliminating long-term secret storage. The reverse of traditional inbound federation (where external IdPs authenticate into AWS).
