Public Access Scenarios
L-01 You need to host a static website on S3. You set bucket public access to OFF (disabled Block Public Access), add a bucket policy allowing s3:GetObject for everyone — but users still get 403. Why? Medium
Setup: Block Public Access disabled at bucket level. Bucket policy with Principal: "*", Action: "s3:GetObject". Static website hosting enabled. Objects uploaded by an IAM user.
Root Cause
  • Object Ownership is set to "Bucket owner enforced" — this is the default for new buckets and it disables ACLs, which is fine — BUT the real issue is the bucket policy missing the Resource ARN for objects.
  • Common mistake: bucket policy Resource is set to arn:aws:s3:::my-bucket (the bucket) instead of arn:aws:s3:::my-bucket/* (objects inside).
  • s3:GetObject applies to objects, not the bucket itself. If Resource points to the bucket ARN, every GetObject request is denied.
// WRONG — Resource points to bucket, not objects { "Effect": "Allow", "Principal": "*", "Action": "s3:GetObject", "Resource": "arn:aws:s3:::my-bucket" } // CORRECT { "Effect": "Allow", "Principal": "*", "Action": "s3:GetObject", "Resource": "arn:aws:s3:::my-bucket/*" }
Fix
  • Update the bucket policy Resource to arn:aws:s3:::my-bucket/*
  • Verify Block Public Access is OFF at both account level and bucket level — account-level setting overrides bucket-level
L-02 Block Public Access is OFF. Bucket policy grants s3:GetObject to everyone. Resource is correct (/*). Still 403. You check and the account-level BPA is also OFF. What else could cause this? Hard
Setup: Everything looks right. Policy is syntactically valid. BPA off everywhere. Objects exist. Yet 403 persists.
5 Possible Root Causes
  • 1. SCP (Service Control Policy) at AWS Organization level — a Deny on s3:GetObject or s3:* in the Org SCP overrides everything. SCPs are invisible from the bucket console.
  • 2. VPC endpoint policy — if the request comes from a VPC with an S3 gateway endpoint, the endpoint policy might deny public bucket access.
  • 3. Object-level ACL set to private — if ACLs are still enabled (not using Object Ownership enforced), individual objects can have a private ACL overriding the bucket policy.
  • 4. Server-side encryption with SSE-KMS — the requester (anonymous) cannot use KMS keys. Public access only works with SSE-S3 encrypted objects.
  • 5. Bucket policy has a conflicting Deny statement — a Deny always wins over Allow. Check for any Deny with conditions like aws:SecureTransport: false that catches HTTP requests.
L-03 You want to make only one specific folder (prefix) in a bucket public, not the entire bucket. How do you do this correctly with a bucket policy? Medium
Goal: Bucket has folders: public/, private/, internal/. Only public/ should be accessible without auth.
{ "Version": "2012-10-17", "Statement": [ { "Sid": "PublicReadOnlyForPublicFolder", "Effect": "Allow", "Principal": "*", "Action": "s3:GetObject", "Resource": "arn:aws:s3:::my-bucket/public/*" } ] }
Key Points
  • Block Public Access must still be OFF — BPA is bucket-level, not prefix-level
  • No separate statement needed to deny private/ — absence of Allow is implicit deny
  • Gotcha: ListBucket is NOT granted here — users can GET objects if they know the key, but cannot list the bucket. Add s3:ListBucket with condition StringLike: s3:prefix: public/* if listing is needed
L-04 A developer accidentally made the entire bucket public. You re-enable Block Public Access but the bucket policy still says Principal: "*". Is the bucket still public? Tricky
Answer
  • No — BPA overrides the bucket policy. Even if the policy still has Principal: "*", enabling RestrictPublicBuckets (part of BPA) means unauthenticated requests are denied.
  • You do NOT need to delete or edit the bucket policy — BPA acts as an enforcement layer on top of it.
  • However: it's still best practice to also clean up the policy, because if someone later disables BPA the bucket becomes instantly public again.
  • BPA setting BlockPublicPolicy prevents any new policy from granting public access from being saved — this is the preventive control.
// BPA has 4 settings — know each one: BlockPublicAcls → Blocks new ACLs that grant public access IgnorePublicAcls → Ignores ALL existing public ACLs BlockPublicPolicy → Blocks saving any policy granting public access RestrictPublicBuckets → Restricts access for anonymous requests
IAM & Cross-Account Scenarios
L-05 An EC2 instance in Account A has an IAM role with s3:GetObject permission on a bucket in Account B. The EC2 is getting AccessDenied. The IAM policy looks correct. Why? Hard
Cross-account access requires BOTH sides to allow: the identity policy in Account A AND the resource policy (bucket policy) in Account B.
Root Cause
  • For cross-account access, an IAM policy alone is not enough. The bucket in Account B must have a bucket policy explicitly granting access to the IAM role/account from Account A.
  • This is different from same-account access, where an IAM policy alone is sufficient.
// Account B bucket policy must include: { "Effect": "Allow", "Principal": { "AWS": "arn:aws:iam::ACCOUNT-A-ID:role/my-ec2-role" }, "Action": ["s3:GetObject"], "Resource": "arn:aws:s3:::account-b-bucket/*" }
Remember
  • Same account: IAM policy alone works (bucket trusts same account)
  • Cross-account: both IAM policy (Account A) + bucket policy (Account B) required
  • If a bucket policy has an explicit Deny, it blocks even the bucket owner
L-06 You upload objects to a bucket from Account B (cross-account). The bucket owner (Account A) cannot access those objects even though they own the bucket. Why? Tricky
The Object Ownership Trap
  • In S3, objects are owned by the uploader, not the bucket owner. If Account B uploads an object to Account A's bucket, Account B owns that object.
  • Account A cannot access, delete, or read that object unless Account B grants them access via ACL or Account A uses a bucket policy with the bucket-owner-full-control condition.
Solutions
  • Option 1: Require uploaders to grant bucket owner full control via ACL: x-amz-acl: bucket-owner-full-control header on PUT
  • Option 2 (Recommended): Enable S3 Object Ownership → Bucket owner enforced. This forces all objects to be owned by the bucket owner regardless of who uploaded them. Disables ACLs.
  • Option 3: Bucket policy Deny on any PUT that doesn't include bucket-owner-full-control ACL
// Deny uploads without bucket-owner-full-control ACL { "Effect": "Deny", "Principal": "*", "Action": "s3:PutObject", "Resource": "arn:aws:s3:::my-bucket/*", "Condition": { "StringNotEquals": { "s3:x-amz-acl": "bucket-owner-full-control" } } }
L-07 An IAM user has s3:* permissions on all resources ("Resource": "*"). They're getting AccessDenied when trying to delete a bucket. What could cause this? Medium
Possible Causes (in order of likelihood)
  • 1. Bucket is not empty. S3 requires a bucket to be empty before deletion. DeleteBucket fails with BucketNotEmpty error, which surfaces as AccessDenied in some SDKs.
  • 2. MFA Delete is enabled. Requires MFA token with every delete request, including bucket deletion.
  • 3. Object Lock is active. Buckets with Object Lock enabled cannot be deleted until all lock retentions expire.
  • 4. SCP Deny at Org level. Even s3:* in IAM won't help if an Org SCP explicitly denies s3:DeleteBucket.
  • 5. Permission Boundary. The IAM user may have a permission boundary that doesn't include s3:DeleteBucket even if their policy does.
Replication & Versioning Scenarios
L-08 You set up Cross-Region Replication (CRR) from bucket-A (us-east-1) to bucket-B (ap-south-1). New objects replicate fine, but 10,000 existing objects are NOT in bucket-B. Why, and how do you fix it? Medium
Root Cause
  • CRR only replicates objects uploaded AFTER replication is configured. It does not backfill existing objects automatically.
Fix — S3 Batch Replication
  • Use S3 Batch Replication to replicate existing objects on-demand
  • Go to S3 Console → Bucket → Management → Replication rules → Replicate existing objects
  • Or use S3 Batch Operations with the Replicate action and an S3 Inventory manifest as input
Also remember
  • Versioning must be enabled on both source and destination buckets
  • Delete markers are NOT replicated by default — must explicitly enable "Delete marker replication"
  • SSE-KMS encrypted objects need additional KMS key permissions in the replication IAM role
L-09 Versioning is enabled. A developer ran aws s3 rm s3://my-bucket/important-file.txt. They want to recover it. What happened and how do you recover? Easy
What Happened
  • With versioning ON, s3 rm does NOT permanently delete. It inserts a delete marker as the latest version. The file appears gone to normal GET requests.
  • All previous versions still exist and are recoverable.
Recovery Steps
# Step 1: List all versions to find the delete marker aws s3api list-object-versions \ --bucket my-bucket \ --prefix important-file.txt # Step 2: Delete the delete marker (NOT the object) aws s3api delete-object \ --bucket my-bucket \ --key important-file.txt \ --version-id DELETE_MARKER_VERSION_ID # The previous version is now the current version again
Permanent Delete
  • To permanently delete with versioning on, you must specify the --version-id in the delete call
  • MFA Delete can protect against permanent deletion by requiring an MFA token
L-10 Your S3 costs doubled after enabling versioning. How do you investigate and reduce cost without disabling versioning? Medium
Investigation
  • Use S3 Storage Lens or S3 Inventory to see how many non-current versions exist and their total size
  • S3 Inventory report: enable "List all versions" to see version counts per object
Cost Reduction — Lifecycle Rules
// Lifecycle rule to expire non-current versions { "Rules": [{ "Status": "Enabled", "NoncurrentVersionExpiration": { "NoncurrentDays": 30, "NewerNoncurrentVersions": 3 }, "AbortIncompleteMultipartUpload": { "DaysAfterInitiation": 7 } }] }
  • NoncurrentDays: 30 — delete versions older than 30 days
  • NewerNoncurrentVersions: 3 — keep at most 3 non-current versions regardless of age
  • Also clean up incomplete multipart uploads — these are invisible in the console but cost money
Encryption & Compliance Scenarios
L-11 You want to enforce that ALL objects uploaded to a bucket are encrypted. Some developers are uploading without encryption headers. How do you enforce this at the bucket policy level? Medium
Note First
  • Since Jan 2023, S3 encrypts all new objects by default with SSE-S3 even without headers. So "unencrypted uploads" are actually SSE-S3 encrypted automatically. But if you want to enforce SSE-KMS specifically:
// Deny uploads without SSE-KMS encryption { "Effect": "Deny", "Principal": "*", "Action": "s3:PutObject", "Resource": "arn:aws:s3:::my-bucket/*", "Condition": { "StringNotEquals": { "s3:x-amz-server-side-encryption": "aws:kms" } } } // Also enforce HTTPS-only access (deny HTTP) { "Effect": "Deny", "Principal": "*", "Action": "s3:*", "Resource": [ "arn:aws:s3:::my-bucket", "arn:aws:s3:::my-bucket/*" ], "Condition": { "Bool": { "aws:SecureTransport": "false" } } }
L-12 You enabled Object Lock in Compliance mode with a 7-year retention. Auditors now say 3 years is sufficient. You try to shorten the retention period — it fails. Why? Is there any way out? Hard
Answer
  • Compliance mode is immutable — no one can shorten it. Not the root account, not AWS Support, not AWS itself.
  • You can extend the retention period in Compliance mode, but never shorten it.
  • This is by design to meet SEC 17a-4, FINRA requirements where auditors need guaranteed immutability.
What You Can Do
  • Wait it out — retention expires after 7 years as configured
  • For future objects: use Governance mode instead, which allows authorized users (with s3:BypassGovernanceRetention permission) to shorten or remove retention
  • Legal Hold is independent of retention — you can always remove a Legal Hold even in Compliance mode (just not the retention period itself)
Performance & Cost Scenarios
L-13 Your application reads 10,000 objects/second from S3 Standard. Latency is 100–200ms. You need sub-10ms latency. The workload is ML training on EC2. What do you change? Medium
Solution Path
  • Switch to S3 Express One Zone — single-digit ms latency, co-located with EC2 in the same AZ
  • Create a Directory Bucket in the same AZ as your EC2 fleet
  • Use Mountpoint for S3 to mount the bucket as a local filesystem — applications read files without S3 API changes
  • Ensure EC2 and the Express One Zone bucket are in the same AZ — cross-AZ defeats the purpose
Additional Optimizations
  • Use byte-range fetches to parallelize large object reads
  • Spread keys across multiple prefixes to avoid request-rate throttling (S3 supports 5,500 GET/s per prefix)
  • Use S3 Transfer Acceleration only for uploads from distant clients — not relevant here
L-14 A 50 GB file upload to S3 keeps failing halfway through. The AWS SDK retries but eventually gives up. How do you fix the upload process? Easy
Root Cause
  • Uploading a 50 GB file as a single PUT request is impossible — max single PUT is 5 GB. Must use multipart upload. Network failures mid-upload also fail the entire upload.
Solution — Multipart Upload
# Using AWS CLI — automatically uses multipart for large files aws s3 cp large-file.bin s3://my-bucket/ \ --part-size 100MB \ --expected-size 53687091200 # Or explicitly with low-level API: # 1. Initiate aws s3api create-multipart-upload --bucket my-bucket --key large-file.bin # 2. Upload parts (in parallel) aws s3api upload-part --bucket my-bucket --key large-file.bin \ --part-number 1 --upload-id <UploadId> --body part1.bin # 3. Complete aws s3api complete-multipart-upload --bucket my-bucket --key large-file.bin \ --upload-id <UploadId> --multipart-upload file://parts.json
  • Failed individual parts can be retried without restarting the upload
  • Add a lifecycle rule to abort incomplete multipart uploads after 7 days — they cost money even if never completed
  • Max object size: 50 TB (as of Dec 2025). Max part size: 5 GB. Min part size: 5 MB (except last part)
L-15 You have 100 TB in S3 Standard. Half was last accessed 8 months ago. S3 Intelligent-Tiering is not enabled. Your bill is high. What lifecycle strategy would you design? Medium
Recommended Strategy
Standard (days 0-30) → Standard-IA at day 30 (if access pattern unknown) → Glacier Instant Retrieval at day 90 (if accessed < quarterly) → Glacier Deep Archive at day 365 (if accessed < yearly) → Delete at day 2555 (7 years, if regulatory permits) # Cost comparison per GB/month (us-east-1 approx): Standard $0.023 Standard-IA $0.0125 (-46%) Glacier Instant $0.004 (-83%) Glacier Flex $0.0036 (-84%) Deep Archive $0.00099 (-96%)
  • For truly unpredictable access patterns, enable S3 Intelligent-Tiering instead — no retrieval fees, auto-tiering, $6B+ saved industry-wide
  • Watch out for minimum duration charges: Standard-IA (30 days), Glacier Instant/Flex (90 days), Deep Archive (180 days)
  • Small objects (<128 KB) should stay in Standard — the per-object monitoring overhead of IA tiers costs more than storage savings
Advanced & Tricky Scenarios
L-16 Your bucket policy enforces HTTPS only (aws:SecureTransport: false → Deny). A pre-signed URL generated by an IAM role is giving 403. The URL uses HTTPS. What's wrong? Hard
The Pre-signed URL Expiry Trap
  • The URL is HTTPS — so SecureTransport is not the issue.
  • Most likely: the pre-signed URL has expired. Default expiry is 1 hour (3600s). Max expiry with STS temporary credentials is 12 hours. Max with IAM user credentials is 7 days.
  • Second possibility: The IAM role/user that generated the URL no longer has the permissions (role was modified after URL was generated). Pre-signed URLs are evaluated at request time, not generation time — if the underlying IAM permission was revoked, the URL stops working immediately.
  • Third possibility: The role was a temporary role (STS) and the session expired.
L-17 You deleted a bucket and want to recreate it with the same name immediately. The creation fails. Why? Tricky
S3 Bucket Name Propagation Delay
  • S3 bucket names are globally unique. After deletion, the name is not immediately released back into the pool. There is a propagation delay before the name becomes available again.
  • In some cases, the name may also be reserved by AWS for a period to prevent DNS rebinding attacks (where an attacker claims a recently-deleted bucket name).
  • Fix: Wait a few minutes to hours, or use a slightly different bucket name.
  • Real-world implication: Never hardcode bucket names that might be deleted and recreated in automated pipelines — use a naming convention with a timestamp or UUID suffix.
L-18 You have a Lambda triggered on s3:ObjectCreated. The Lambda processes the file and writes a result back to the SAME bucket. Your Lambda is running infinitely in a loop. Why and how do you fix it? Hard
Infinite Trigger Loop
  • Lambda writes output back to the same bucket → triggers another ObjectCreated event → Lambda invoked again → writes again → infinite loop. Each invocation costs money and can exhaust Lambda concurrency limits.
Fix Options
  • Option 1 (Best): Write results to a different bucket — clean separation of input/output
  • Option 2: Use a different prefix for output and configure the event notification to only trigger on the input prefix: Filter: Prefix: input/
  • Option 3: Use a different suffix — trigger on .csv, write output as .json, filter events on .csv suffix only
  • Option 4: Add idempotency — check a DynamoDB table or object metadata tag before processing to skip already-processed files
// Event notification filter — only trigger on input/ prefix { "Filter": { "Key": { "FilterRules": [ { "Name": "prefix", "Value": "input/" }, { "Name": "suffix", "Value": ".csv" } ] } } }
L-19 A bucket in us-east-1 is configured with CRR to eu-west-1. An object is deleted in us-east-1. The object still exists in eu-west-1. Is this a bug? Tricky
Not a Bug — This is Default Behavior
  • Delete marker replication is OFF by default. CRR replicates new objects but does NOT replicate deletions unless explicitly configured.
  • This is intentional — it protects the destination bucket from accidental or malicious mass deletion in the source.
To enable delete marker replication:
// In the replication configuration { "DeleteMarkerReplication": { "Status": "Enabled" } }
  • Even with this enabled, permanent deletes (specifying a version ID) are never replicated — only delete markers
  • This asymmetry is useful: destination bucket can serve as a "soft delete" recovery point
L-20 You have a CloudFront distribution serving content from a private S3 bucket. Users are getting 403 Access Denied. The S3 bucket policy grants access to the CloudFront OAI. What changed recently that might cause this? Hard
OAI vs OAC Migration Issue
  • AWS deprecated Origin Access Identity (OAI) and recommends Origin Access Control (OAC). If the distribution was recently upgraded from OAI to OAC, the bucket policy must be updated.
  • OAI policy principal: arn:aws:iam::cloudfront:user/CloudFront Origin Access Identity <ID>
  • OAC policy principal: service: cloudfront.amazonaws.com with a condition on the distribution ARN
// OAC bucket policy (new way) { "Effect": "Allow", "Principal": { "Service": "cloudfront.amazonaws.com" }, "Action": "s3:GetObject", "Resource": "arn:aws:s3:::my-bucket/*", "Condition": { "StringEquals": { "AWS:SourceArn": "arn:aws:cloudfront::ACCOUNT:distribution/DISTRO-ID" } } }

Concepts — How S3 Actually Works
What is a bucket key in S3? Is it a path?
show ▾
S3 is a flat object store — there are no real folders or directories. A key is just a string like images/2024/photo.jpg. The slash "/" has no special meaning to S3 — it's just part of the key name. The console shows "folders" as a UI convenience by grouping keys with the same prefix. Deleting a "folder" in the console is just deleting all objects with that prefix — it is not a real directory delete operation.
What is the S3 consistency model? Has it always been this way?
show ▾
As of December 2020, S3 provides strong read-after-write consistency for all operations: PUTs, DELETEs, and LIST operations. Before this, S3 only had eventual consistency for overwrites and deletes, which meant reading a deleted object could still return data for a short period. Now: immediately after a successful PUT, any subsequent GET returns the new data. Immediately after a DELETE, GET returns 404. LIST always reflects the latest state.
How does S3 request rate scaling work? What is prefix-based scaling?
show ▾
S3 automatically scales to high request rates but the rate limits are per prefix: 5,500 GET/HEAD requests per second and 3,500 PUT/POST/DELETE per second per prefix. A prefix is anything before a common delimiter — e.g., 2024/01/ is one prefix, 2024/02/ is another. If you have 10 distinct prefixes, you effectively get 10× the throughput. Old advice was to randomize key prefixes (hashing) but since 2018 S3 auto-partitions without this — you only need prefix distribution for extreme throughput workloads.
What is the difference between a bucket policy and an IAM policy for S3?
show ▾
Bucket policyResource-based. Attached to the bucket. Defines who can access THIS bucket. Can grant cross-account access. Max size: 20 KB.
IAM policyIdentity-based. Attached to a user/role/group. Defines what S3 resources THAT IDENTITY can access.
EvaluationBoth must allow (union). Any explicit Deny wins. For cross-account: both sides must allow.
ACLLegacy. Per-object. AWS recommends disabling via Object Ownership → Bucket owner enforced.
What is S3 Versioning and what are its implications?
show ▾
Once enabled, versioning cannot be disabled — only suspended. When suspended, new objects get a null version ID but old versions are retained. Key behaviors:

• Every PUT creates a new version with a unique ID
• DELETE inserts a delete marker (object appears gone, all versions preserved)
• To permanently delete, you must DELETE with a specific version-id
• Storage costs accumulate for ALL versions — use lifecycle rules to expire old versions
• MFA Delete requires MFA token for permanent deletion (extra protection)
• Required for: Cross-Region Replication, S3 Object Lock
What are the 4 Block Public Access settings and what does each do?
show ▾

How does S3 Intelligent-Tiering actually work internally?
show ▾
S3 IT monitors access at the per-object level. It has 5 tiers:
• Frequent Access — default tier for all newly uploaded objects
• Infrequent Access — moved here after 30 consecutive days without access
• Archive Instant Access — opt-in, moved here after 90 days (millisecond retrieval)
• Archive Access — opt-in, 90–730 days without access (async retrieval)
• Deep Archive Access — opt-in, 180–730+ days (12-hour retrieval)

Any access resets the object back to Frequent Access. No retrieval fees for automatic tier transitions. Small monitoring charge per 1,000 objects. Not cost-effective for objects under 128 KB or data accessed more than monthly.
What is S3 Object Ownership and why does it matter?
show ▾
In S3, objects are owned by the AWS account that uploads them — NOT the bucket owner. This causes problems when Account B uploads to Account A's bucket — Account A can't access those objects by default.

Object Ownership has 3 settings:
• Bucket owner enforced (recommended): All objects owned by bucket owner. ACLs disabled. Simplest security model.
• Bucket owner preferred: Objects uploaded with bucket-owner-full-control ACL are owned by bucket owner. Others still owned by uploader.
• Object writer (legacy default): Uploader owns the object.
What events can S3 send notifications for? What are the destinations?
show ▾
Events: ObjectCreated (Put, Post, Copy, CompleteMultipartUpload), ObjectRemoved (Delete, DeleteMarkerCreated), ObjectRestore (Post, Completed), Replication (OperationFailedReplication, etc.), LifecycleTransition, IntelligentTiering, ObjectTagging, ObjectAcl

Destinations: Amazon SNS, Amazon SQS, AWS Lambda, Amazon EventBridge

EventBridge advantage: Filter by event type, prefix, suffix, object size, tags. Route to Step Functions, CodePipeline, API Gateway, and 18+ other targets. Single configuration for all event types.

Limitation of native S3 notifications (non-EventBridge): Can only have ONE destination per event type per bucket.

What is S3 Transfer Acceleration? When should you use it vs NOT use it?
show ▾
Routes upload traffic through the nearest CloudFront edge location, then over the AWS private backbone to S3. Uses a different endpoint: bucket.s3-accelerate.amazonaws.com.

Use when: Large file uploads (1 GB+) from geographically distant clients. Cross-continental transfers. Mobile uploads from high-latency networks.

Don't use when: Clients are in the same region as the S3 bucket — acceleration adds latency via the edge detour. Use the S3 Speed Comparison tool first.

Extra cost: $0.04/GB for transfers from internet to S3 via acceleration (on top of normal transfer costs).
What happens to S3 replication when the source object is deleted?
show ▾
Delete markers are NOT replicated by default. You must enable "Delete marker replication" explicitly in the replication config.

Permanent deletes (specifying version-id) are NEVER replicated — even with delete marker replication enabled. This protects the destination from data loss.

Replication direction: Bidirectional replication requires configuring rules in BOTH directions. CRR is one-way by default.

What IS replicated: Objects, metadata, tags, ACLs, object lock retention, SSE-S3 and SSE-KMS encryption (KMS key config needed).
How does S3 pricing work? What are all the cost dimensions?
show ▾
S3 costs come from 6 dimensions:

1. Storage — per GB/month by storage class. Standard ~$0.023, Deep Archive ~$0.001
2. Requests — per 1,000 API calls. PUT/POST/DELETE/LIST more expensive than GET/HEAD
3. Data transfer out — free to CloudFront, free within same region, charged to internet/other regions
4. Data retrieval — charged per GB for IA and Glacier classes
5. Minimum storage duration — charged for full minimum even if deleted early (30/90/180 days)
6. Feature charges — Replication, S3 Intelligent-Tiering monitoring, S3 Inventory, CloudWatch metrics, Batch Operations, Storage Lens advanced
What is S3 Access Points and when do you use them over bucket policies?
show ▾
Access Points are named endpoints with their own IAM policies, allowing you to create up to 10,000 different "views" into the same bucket.

Use access points when:
• Multiple teams/applications access the same bucket with different permission needs
• You want per-application VPC restriction (access point can be limited to a specific VPC)
• A single bucket policy would become too complex or hit the 20 KB limit

Key rule: The bucket policy must also delegate access to access points (or trust all access points): include a statement allowing access via the access point ARN. Both the access point policy AND bucket policy must allow the action.

Multi-Region Access Points are different — they're a global routing layer across multiple buckets in different regions.
