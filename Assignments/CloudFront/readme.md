# AWS S3 + CloudFront CDN Hosting (Private S3) — Complete Guide

## Overview

This guide explains how to host static content using **Amazon S3 as the origin** and **Amazon CloudFront as the CDN**, while keeping the S3 bucket **private**. It includes the setup steps, configuration details, and common troubleshooting scenarios encountered during implementation.

Architecture:

```
User
  │
  ▼
CloudFront (CDN)
  │
  ▼
Private S3 Bucket
```

Users access content through CloudFront, and CloudFront securely fetches files from S3.

---

# Services Used

* Amazon S3 – Object storage used to host static files
* Amazon CloudFront – Content Delivery Network (CDN)
* Origin Access Control (OAC) – Secure CloudFront access to private S3

---

# Step 1 — Create an S3 Bucket

1. Go to AWS Console → S3
2. Click **Create Bucket**

Example configuration:

```
Bucket Name: policybucket123456789098765
Region: any preferred region
```

Important settings:

* **Block Public Access:** Enabled
* **Object Ownership:** Bucket owner enforced (ACLs disabled)

This ensures the bucket is **private**.

---

# Step 2 — Upload Website Files

Upload your static files:

```
index.html
error.html
images/
css/
js/
```

Example `index.html`:

```html
<!DOCTYPE html>
<html>
<head>
<title>CloudFront CDN Demo</title>
</head>

<body>
<h1>Hello from CloudFront CDN</h1>
</body>

</html>
```

---

# Step 3 — Create CloudFront Distribution

Go to:

```
CloudFront → Create Distribution
```

Origin configuration:

```
Origin Domain:
policybucket123456789098765.s3.amazonaws.com
```

Important:

Do NOT use the S3 website endpoint.

Correct:

```
bucket-name.s3.amazonaws.com
```

Incorrect:

```
bucket-name.s3-website-region.amazonaws.com
```

The website endpoint requires a **public bucket**.

---

# Step 4 — Configure Origin Access Control (OAC)

In CloudFront Origin settings:

```
Origin Access → Origin Access Control
```

Click:

```
Create new OAC
```

Attach it to the origin.

This allows CloudFront to securely access the private S3 bucket.

---

# Step 5 — Configure Default Root Object

In CloudFront settings:

```
Default Root Object: index.html
```

This ensures:

```
https://distribution-id.cloudfront.net
```

loads `index.html`.

---

# Step 6 — Configure S3 Bucket Policy

Go to:

```
S3 → Bucket → Permissions → Bucket Policy
```

Add the policy allowing CloudFront access.

Example:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCloudFrontServicePrincipal",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudfront.amazonaws.com"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::policybucket123456789098765/*",
      "Condition": {
        "StringEquals": {
          "AWS:SourceArn": [
            "arn:aws:cloudfront::836763845852:distribution/E3NNMX19BADG74",
            "arn:aws:cloudfront::836763845852:distribution/E1RW2AMUBCT67T"
          ]
        }
      }
    }
  ]
}
```

Notes:

* `Version` must be **2012-10-17**
* Use **StringEquals**, not `ArnLike`

# Default CloudFront Errors

CloudFront has built-in error pages such as:

```
403 ERROR
The request could not be satisfied.
```

These appear when:

* Origin denies access
* File does not exist
* Permission issues occur

CloudFront error pages typically show:

```
Generated by cloudfront (CloudFront)
Request ID: XXXXX
```

---

# Custom Error Responses

CloudFront allows overriding errors.

Example configuration:

| Error Code | Response Page | Response Code |
| ---------- | ------------- | ------------- |
| 403        | /index.html   | 200           |
| 404        | /index.html   | 200           |

This is commonly used for **Single Page Applications (SPA)**.

---

# SPA Routing Fix (React / Angular / Vue)

Problem:

When user visits:

```
/dashboard
```

CloudFront asks S3:

```
GET /dashboard
```

S3 response:

```
403 AccessDenied
```

Solution:

Configure CloudFront:

```
403 → /index.html
404 → /index.html
```

CloudFront serves `index.html`, and the frontend router handles the route.

---

# Access Control (ACL vs Policy)

Recommended modern setup:

| Feature           | Setting               |
| ----------------- | --------------------- |
| ACLs              | Disabled              |
| Object Ownership  | Bucket owner enforced |
| Access Method     | Bucket Policy         |
| CloudFront Access | OAC                   |

ACLs are **not required**.

# Troubleshooting Guide

## Issue: AccessDenied XML Error

Example:

```
<Error>
<Code>AccessDenied</Code>
<Message>Access Denied</Message>
</Error>
```

Cause:

S3 permission issue.

Solutions:

* Verify bucket policy
* Ensure correct distribution ARN
* Confirm OAC is attached

---

## Issue: CloudFront Cannot Access S3

Possible causes:

* Missing bucket policy
* Wrong origin endpoint
* OAC not attached

Fix:

* Attach Origin Access Control
* Update bucket policy

---

## Issue: Using S3 Website Endpoint

Incorrect origin:

```
bucket-name.s3-website-region.amazonaws.com
```

Result:

CloudFront cannot access private bucket.

Fix:

Use:

```
bucket-name.s3.amazonaws.com
```

---

## Issue: Default Root Object Missing

Symptoms:

```
AccessDenied when opening /
```

Fix:

Set:

```
Default Root Object = index.html
```

---

## Issue: CloudFront Error Not Triggering

Cause:

Configured error code does not match origin response.

Example:

```
Configured error → 404
Actual origin response → 403
```

Fix:

Add rule for **403**.

# Final Best Practices

1. Keep S3 bucket **private**
2. Use **CloudFront OAC**
3. Disable **ACLs**
4. Use **bucket policy for access**
5. Configure **Default Root Object**
6. Add **error handling for SPA apps**
7. Use CloudFront for all public access
